# 待确认可能的伪自由编码

> 审计日期: 2026-05-16
> 审计范围: `packages/koishi-plugin-dongxuelian-ai/lib/` 全量代码
> 判定标准: 代码是否替模型做了本应由模型自主完成的决策

---

## 🔴 CRITICAL：模型根本没机会参与决策

### 1. `http-search.js:302` — 拒绝所有 baike.baidu.com 链接

```js
function isHomepageUrl(url = '') {
  if (/baike\.baidu\.com/i.test(parsed.hostname)) return true
}
```

**问题**: 百度百科是合法中文参考源，全面封锁导致搜索结果中百科页面被丢弃。
**建议**: 删除这条规则，让模型自己判断百科内容是否相关。

---

## 🟠 HIGH：代码替模型做了不可逆选择

### 2. `engine.js:23-25` — 硬上限约束

```js
const MAX_ROUNDS = 5
const MAX_TOOLS_PER_ROUND = 3
const MAX_WEB_SEARCH_CALLS = 3
```

**问题**: 
- 模型无法预判"我还有多少轮"，可能在第 4 轮就差一步时被中止
- 每轮多余的 tool_calls 被静默丢弃，模型看到不一致的状态
- 80000 token 硬停止 (`context.js:173-178`) — 应该告知模型而非直接杀死

**建议**: 
- 保留上限作为安全网，但让模型在接近上限时收到提示
- 静默丢弃改为告知模型"本轮只执行了前 N 个工具"

---

### 3. `chat-tools.js:12-14` — 工具轻/重分类模型无感知

```js
const LIGHTWEIGHT_TOOLS = new Set(['get_current_time', 'calculate', 'search_memory'])
const HEAVY_TOOLS = new Set(['web_search', 'browser_action', 'execute_shell', 'file_write'])
```

**问题**: 模型不知道这个分类存在。它调 `web_search` 以为在执行搜索，实际上整个交互被重路由到 agent 管道。

**建议**: 在 chat 工具定义的 description 中标注"此工具耗时较长，调用后会异步处理"，让模型有知情权。

---

### 4. `search-results.js:176-220` — 代码判断搜索失败原因并生成重试 query

```js
function detectFailurePattern(ranked, pages, allCandidates) { ... }
function buildStrategyQueries(failurePattern, originalQuery, usedQueries) { ... }
```

**问题**: 代码替模型做了"为什么搜索失败"的分析（字典歧义/首页充斥/正文浅薄/低相关/无结果），并生成重试关键词。这本该是模型的分析能力。

**建议**: 将失败模式分析作为结果的一部分告知模型（而非直接生成 retry query），让模型自己决定下一步。

---

## 🟡 MEDIUM：代码偏见影响了模型判断

### 5. `messages.js:13-17` — STATIC_SYSTEM 规定工具调用顺序

```js
'遇到文件、人格、lore、日志、服务器状态问题时，先用可用工具定位和读取，再总结',
'遇到需要特定工作流的任务时，先调用 read_agent_skill 读取对应 Skill 说明，再执行后续步骤',
'遇到最新角色、新闻、版本、天气等实时问题时，优先使用 web_search',
```

**问题**: "先用/先调用/优先使用" 是硬编码的工具选择顺序。模型可能有比这些模板更好的策略。

**建议**: 改为松耦合建议而非硬性顺序指令。例: "可用工具包括...根据任务选择合适的工具" 而非 "遇 X 先用 Y"。

---

### 6. `search-results.js:145-154` — 硬编码结果质量分类阈值

```js
function classifySearchResult(ranked, pages) {
  if (hasPageContent && hasHighScore(>=50)) return 'usable_hit'
  if (hasVeryHighScore(>=80) && text >= 40 chars) return 'usable_hit'
}
```

**问题**: 50 分是高可信，80 分是极高可信，40 字符算可用正文——这些数字是编的，且对所有查询一刀切。

**建议**: 放松阈值或将其降为提示信息而非直接影响重试策略。

---

### 7. `search-results.js:12-13` — 字典/广告结果过滤

```js
const DICTIONARY_RESULT_RE = /(?:字典|百科|汉典|汉语|词典|说文|...)/i
const SEARCH_NOISE_RE = /(?:广告|推广|免责声明)/
```

**问题**: 用户问"这个词什么意思"时字典结果是目标；"广告"可能是官方推广帖。正则不可逆地过滤了模型可能需要的内容。

**建议**: 保留过滤但放宽阈值，或把低分结果标记为 `(低可信)` 而非直接丢弃。
