# 莲莲Bot 借鉴 QwenPaw & Claude Code：Agent 能力渐进式开发计划

> 目标：在不破坏莲莲Bot现有 QQ 聊天人格、风控、记忆、Dashboard 和部署链路的前提下，逐步引入 Agent 工具调用、Token/上下文优化、工具安全审批、浏览器自动化和前端控制台能力。
>
> 核心原则：先做最小可用闭环，再扩工具；先 Dashboard 管理闭环，再放到 QQ；先只读工具，再危险工具；先可观测和可回滚，再追求功能完整。

---

## 0. 当前计划需要修正的问题

原计划方向正确，但太像“大爆炸式移植”：一次性把 ReAct、浏览器 34 个动作、Skill 市场、Dashboard Agent 控制台、上下文压缩、安全系统全部塞进 10 天，会带来三个问题：

1. **核心聊天链路风险太高**：`chat.js` 是 QQ 群人格、历史、风控、Vision、fallback 的汇合点，直接嵌 ReAct 循环容易影响普通聊天。
2. **浏览器全量移植过早**：QwenPaw 的 `browser_control.py` 是完整 Playwright 工作区状态机，包含 CDP、下载、快照、refs、idle watchdog。应先做 6~8 个核心动作，不应一开始移植 34 个动作。
3. **安全和前端审批必须先于危险工具**：`write_file`、`shell`、`browser run_code` 不能在没有 pending/审批/日志的情况下接入 QQ 群聊。
4. **Claude Code 的缓存优化不能简单理解为“工具只首轮发送”**：更可靠的做法是“静态 prompt 稳定化 + 动态 prompt 后置 + 工具 schema 分层/按需暴露 + 工具结果截断/外置”。
5. **Skill 市场不是第一阶段能力**：技能市场只有在 Agent 工具闭环稳定后才有价值，否则会扩大调试面。

因此本计划改为 6 个阶段，每个阶段都有验收、回滚点和不做事项。

## 1. 借鉴来源与可复用设计

### 1.1 QwenPaw 值得借鉴的点

| 来源 | 可借鉴设计 | 在莲莲Bot中的落地方式 |
|---|---|---|
| `src/qwenpaw/agents/tool_guard_mixin.py` | 工具调用前置拦截、执行等级、审批 pending、并发锁 | 做 `tool-safety.js`，先支持 block/confirm/auto 三档 |
| `src/qwenpaw/agents/context/light_context_manager.py` | 工具结果截断，超长结果保存到文件，消息压缩 | 做 `agent-context.js`，先实现结果截断和摘要，不急着完整压缩 |
| `src/qwenpaw/agents/tools/browser_control.py` | 单 browser 工具 + action 参数、workspace 隔离、refs、idle watchdog | 先实现核心 action 子集，逐步扩展 |
| `console/src/api/modules/tools.ts` | 前端工具列表、开关、配置接口 | Dashboard 增加工具管理 API 和面板 |
| ToolGuard UI/approval flow | 管理员审批危险动作 | Dashboard 审批优先，QQ 只做文本确认 |

### 1.2 Claude Code 值得借鉴的点

| 设计 | 正确理解 | 在莲莲Bot中的落地方式 |
|---|---|---|
| Prompt/cache 稳定性 | 静态系统提示尽量不变，动态上下文后置 | `chat.js` 拆分 core system 与 dynamic system |
| Tool schema 控制 | 不一定每轮都发全部工具；按场景暴露 | QQ 默认少工具，Dashboard 可多工具，危险工具需开关 |
| Tool result pruning | 长结果截断，必要时保存完整结果 | 工具执行层统一做 `truncateResult()` |
| Hooks/审批思想 | 工具执行前后可插安全策略 | `executeTool()` 前后插 safety、日志和统计 |
| 可观测性 | `/cost`、日志、状态面板 | Dashboard 显示工具调用、token 估算、审批记录 |

## 2. 总体架构：不要把 ReAct 直接塞进普通聊天主路径

推荐新增一个独立 Agent Engine，再由 QQ 和 Dashboard 两个入口调用。

```text
QQ 普通聊天
  index.js / handler.js
    └─ chat.js 原有聊天链路（默认不变）

QQ Agent 模式 / 管理命令
  handler.js
    └─ agent-engine.js
        ├─ api.js / callOpenAI
        ├─ tool-registry.js
        ├─ tool-safety.js
        └─ agent-context.js

Dashboard Agent 控制台
  standalone.js /dashboard/api/agent/*
    └─ agent-engine.js（同一套）
```

### 2.1 为什么不直接改 `chat.js` 主流程

`chat.js` 已经承载：人格 prompt、群聊历史、Vision、fallback、敏感词、共享上下文、随机回复、话题总结等。直接在主流程插 ReAct 循环会让普通聊天也承担工具调用风险。

建议：

- 第一阶段新增 `lib/agent-engine.js`，普通聊天不默认走 Agent。
- QQ 侧先通过显式命令进入 Agent，例如：
  - `莲莲工具 现在几点`
  - `莲莲查一下 xxx`
  - 后续再考虑自动意图识别。
- Dashboard Agent 控制台直接调用 Agent Engine。
- 等稳定后，再让 `chat.js` 在特定意图下委托 Agent Engine。

### 2.2 推荐新增文件

```text
packages/koishi-plugin-dongxuelian-ai/lib/agent/
├── engine.js              # ReAct 主循环，独立于普通 chat 主路径
├── messages.js            # system 分层、tool message 组装
├── context.js             # token 估算、工具结果截断、临时压缩
├── stats.js               # token/工具调用统计
├── pending.js             # pending 确认状态，按 channelKey+userId 隔离
├── safety.js              # block/confirm/auto、安全检查、频率限制
└── tools/
    ├── registry.js        # 工具注册、渠道过滤、执行分发
    ├── get-time.js
    ├── calculator.js
    ├── read-file.js
    ├── web-search.js
    ├── browser.js         # 第二/三阶段再扩
    ├── write-file.js      # 危险工具，默认 Dashboard confirm
    └── shell.js           # 危险工具，默认关闭或 Dashboard confirm
```

### 2.3 推荐修改文件

| 文件 | 改动原则 |
|---|---|
| `lib/api.js` | 增加可选 `tools/tool_choice` 参数，确保 tool_calls 不被 content 判空吞掉 |
| `lib/handler.js` | 新增 Agent 显式命令、工具模式、确认命令 |
| `lib/runtime-config.js` | 增加 toolConfig / agentConfig 读写 |
| `packages/koishi-plugin-dashboard/standalone.js` | 增加 `/dashboard/api/agent/*` 和 `/dashboard/api/tools/*` |
| Dashboard 前端 | 后期新增 Agent 控制台和工具管理页 |
| `lib/chat.js` | 第一阶段尽量不改；只在后期做 prompt 分层或 Agent 委托 |

## 3. Token 与缓存优化：先稳定 prompt，再做压缩

### 3.1 目标

不是为了“炫技压缩”，而是避免 Agent 工具循环把上下文撑爆，同时让模型 API 的前缀缓存更容易命中。

### 3.2 四层策略

| 层级 | 内容 | 第一阶段是否做 |
|---|---|---|
| L1 静态 System 稳定化 | core/persona/rules 尽量固定，动态时间/状态后置 | ✅ |
| L2 工具 schema 分层 | QQ 只暴露少量工具；Dashboard 可按开关暴露更多 | ✅ |
| L3 工具结果截断/外置 | 超长结果截断，完整结果存临时文件或日志 | ✅ |
| L4 上下文压缩 | 超阈值时摘要旧 tool 结果和中间轮次 | 第二阶段 |

### 3.3 不建议的做法

- 不建议“第 2 轮完全不发 tools，第 3 轮再恢复”作为硬规则。不同模型对 tool schema 记忆不稳定，容易出现第二轮想继续调工具却没 schema。
- 推荐“按需暴露 + 小工具集 + 工具结果压缩”：
  - QQ：默认只暴露 `get_time`、`calculator`、`web_search`。
  - Dashboard：可暴露文件/浏览器/危险工具，但通过配置开关。
  - ReAct 循环每轮仍传当前可用的小工具集，避免模型丢 schema。

### 3.4 `agent/context.js` 最小实现

```js
function estimateTokens(messages)          // 字符估算即可先上
function truncateToolResult(text, options) // 默认 4000~8000 chars
function compactAgentScratchpad(messages)  // 第二阶段再实现
function buildContextReport()              // Dashboard/命令查看
```

### 3.5 与现有 conversation.summary 的关系

- `conversation.js` 的摘要是长期对话记忆，继续保留。
- Agent 中间轮次、tool result 不写入普通 conversation 历史。
- Agent 完成后只把“用户请求 + 最终回答 + 必要工具摘要”写入历史或日志。

## 4. 工具系统：先做安全只读闭环

### 4.1 工具定义格式

```js
module.exports = {
  name: 'get_current_time',
  description: '获取当前时间',
  parameters: { type: 'object', properties: {} },
  dangerous: false,
  defaultChannels: ['dashboard', 'qq'],
  async execute(args, ctx) {
    return { text: '...' }
  }
}
```

### 4.2 工具注册器职责

`agent/tools/registry.js`：

- 收集工具定义。
- 按渠道过滤：`qq` / `dashboard`。
- 按配置过滤：禁用工具不暴露给 LLM。
- 执行前调用 `safety.check()`。
- 执行后统一截断结果、记录 stats。
- 失败返回结构化错误给 LLM，不让异常炸穿主流程。

### 4.3 工具分批实现

#### P0：只读最小闭环

| 工具 | 渠道 | 说明 |
|---|---|---|
| `get_current_time` | QQ + Dashboard | 验证 tool_calls 基础能力 |
| `calculator` | QQ + Dashboard | 验证参数 schema |
| `web_search` | Dashboard + QQ confirm/限频 | 先 HTTP 搜索，不急 Playwright |

#### P1：文件只读/有限读

| 工具 | 渠道 | 安全要求 |
|---|---|---|
| `read_file` | Dashboard 默认开，QQ 默认关 | 限定工作区，支持 offset/limit，结果截断 |
| `list_files` | Dashboard 默认开，QQ 默认关 | 限定目录深度和数量 |

#### P2：危险工具

| 工具 | 默认 | 安全要求 |
|---|---|---|
| `write_file` | Dashboard confirm，QQ 关 | 路径白名单、diff 预览、审批 |
| `edit_file` | Dashboard confirm，QQ 关 | 精确替换、备份/日志 |
| `execute_shell` | 默认关 | 命令 denylist、timeout、cwd 限制、审批 |
| `execute_javascript` | 默认关 | Node vm/worker timeout，不接触系统权限 |

#### P3：浏览器工具

先做核心 8 个 action，稳定后再扩。

## 5. 安全系统：先设计为不可绕过的执行前置层

### 5.1 安全模式

| 模式 | 行为 | 默认建议 |
|---|---|---|
| `block` | 危险工具一律拒绝 | QQ 默认 |
| `confirm` | 生成 pending，等待管理员确认 | Dashboard 默认 |
| `auto` | 自动执行，但仍做 denylist/路径/频率检查 | 仅本地开发手动开启 |

### 5.2 安全检查

| 检查 | 第一阶段 | 说明 |
|---|:---:|---|
| 未知工具拒绝 | ✅ | 防模型幻觉工具 |
| 渠道过滤 | ✅ | QQ/Dashboard 独立开关 |
| 路径越权 | ✅ | 文件工具必须限制在允许根目录 |
| 命令黑名单 | P2 | shell 上线前必须做 |
| 超时 | ✅ | 所有工具默认 timeout |
| 频率限制 | ✅ | 按 userId + channelKey |
| pending 绑定 | ✅ | 只有发起人/管理员能确认 |
| 审计日志 | ✅ | 记录工具名、参数摘要、结果摘要、耗时 |

### 5.3 pending 确认设计

```text
危险工具调用
  → safety 返回 pendingId
  → Agent 回复：需要确认，展示工具名、参数摘要、风险
  → QQ：用户回复“确认 <pendingId>”
  → Dashboard：点击批准/拒绝
  → 执行后把结果回填给 Agent 或直接返回用户
```

注意：QQ 群聊里不能只用单字“确认”，容易误触；建议必须带 pendingId 或在 60 秒内只允许一个 pending。

## 6. ReAct Engine：最小循环，不污染普通聊天

### 6.1 最小循环

```text
buildAgentMessages()
  ↓
requestChatCompletions(messages, { tools })
  ↓
如果没有 tool_calls → 返回最终文本
  ↓
执行每个 tool_call（默认先串行，后续再并行）
  ↓
把 tool result 追加到 scratchpad
  ↓
最多循环 3~5 次
```

### 6.2 边界

- 每次 Agent 请求最多 5 轮工具循环。
- 每轮最多执行 3 个工具。
- 每个工具有 timeout。
- 总耗时超过阈值则停止并总结已完成内容。
- tool result 不进入普通聊天长期历史，只记录摘要。

### 6.3 `api.js` 必改点

- `requestChatCompletions(messages, options)` 支持：
  - `tools`
  - `tool_choice`
  - `parallel_tool_calls`（可选，先 false）
- 响应解析不能因为 `content` 为空就认为失败。
- fallback 调用要透传 tools，否则主模型失败后工具能力丢失。

## 7. 浏览器自动化：不要第一天移植 34 个 action

### 7.1 参考 QwenPaw 实现

QwenPaw 的 `browser_control.py` 已经包含：

- action-based 单工具 API。
- per-workspace browser state。
- refs / accessibility snapshot。
- console/network 记录。
- 下载目录管理。
- idle watchdog。
- Windows reload 下 sync Playwright fallback。

莲莲Bot不需要一次性完整复制，先实现高频闭环。

### 7.2 第一批 action

| action | 必要性 | 说明 |
|---|---|---|
| `start` | 必需 | 启动浏览器/复用已有实例 |
| `stop` | 必需 | 关闭并清理 |
| `navigate` | 必需 | 打开网页 |
| `snapshot` | 必需 | 给 LLM 页面结构 |
| `click` | 必需 | 基础交互 |
| `type` | 必需 | 输入 |
| `screenshot` | 必需 | 调试/可视反馈 |
| `wait_for` | 常用 | 等待文本/选择器 |

第二批再加：`tabs`、`press_key`、`evaluate`、`console_messages`、`network_requests`、`file_upload/download`、`pdf`、`cookies_*`。

### 7.3 渠道默认

- Dashboard：browser 工具可开启，但默认 confirm 或仅管理员可用。
- QQ：默认关闭 browser；只允许管理员手动开启某些只读/搜索动作。

### 7.4 不能省的安全限制

- URL allow/deny 策略：默认拒绝内网敏感地址、file://、本机管理端口，除非管理员开启。
- 下载大小限制。
- 截图和页面文本结果截断。
- 浏览器进程 idle 自动关闭。

## 9. Skill 市场：后置，不进入 Agent P0

Skill 市场可以做，但不应阻塞 Agent MVP。

### 9.1 更合理定位

- 不是“工具市场”，而是“Agent 使用说明/工作流市场”。
- Skill 只提供 `SKILL.md` 和可选脚本。
- 安装 skill 不自动授予危险权限。

### 9.2 实施时机

等以下能力稳定后再做：

1. Agent Engine 可用。
2. 工具注册/开关可用。
3. 安全审批可用。
4. Dashboard 能展示工具和审批。

### 9.3 最小 CLI

```text
skill-hub list
skill-hub install <local-path|github-url>
skill-hub enable <name>
skill-hub disable <name>
skill-hub remove <name>
```

下载来源第一版只支持本地路径和 GitHub raw；不要一开始支持六个市场源。

## 10. 分阶段实施路线

### Phase 0：准备和保护网

目标：保证后续改动可测、可回滚。

- 梳理 `api.js` 当前响应解析和 fallback。
- 梳理 `chat.js` 当前 system/history/fallback 位置，但不立即改主流程。
- 新增测试夹具：模拟 tool_calls 响应、tool result 响应、fallback 响应。
- 明确数据文件位置：`data/agent-tool-config.json`、`data/agent-tool-audit.log`。

验收：现有 `npm run check`、`npm run test:quick` 通过。

### Phase 1：只读 Agent MVP

目标：Dashboard/QQ 都能通过显式入口调用只读工具。

- 新增 `agent/engine.js` 最小 ReAct 循环。
- `api.js` 支持 tools/tool_calls。
- 新增工具：`get_current_time`、`calculator`。
- `handler.js` 新增显式命令：`莲莲工具 <问题>`。
- 工具结果截断、审计日志、stats。

验收：

- QQ 发“莲莲工具 现在几点”能调工具并回答。
- 普通聊天不受影响。
- 工具调用失败时不会让 bot 崩。

### Phase 2：安全与配置闭环

目标：危险工具上线前先完成权限系统。

- `safety.js`：block/confirm/auto。
- `pending.js`：pendingId、过期、用户绑定。
- `runtime-config.js`：工具模式和渠道配置。
- `handler.js`：工具状态、模式切换、确认/拒绝命令。
- Dashboard API：工具列表、开关、pending 审批。

验收：

- QQ 默认危险工具不可见。
- Dashboard 可以开关工具。
- confirm 模式能生成 pending 并审批。

### Phase 3：文件和搜索工具

目标：让 Agent 有实际辅助能力，但仍以可控为主。

- `web_search`：先 HTTP 搜索，失败给清晰提示。
- `read_file` / `list_files`：限定工作区、offset/limit、结果截断。
- `write_file` / `edit_file`：仅 Dashboard confirm，带参数摘要和审计。

验收：

- Dashboard Agent 能读文件、搜索、给建议。
- 危险文件写入必须审批。
- 路径越权被拒绝。

### Phase 4：Dashboard Agent 控制台

目标：让工具可视化管理，不只靠 QQ 命令。

- 新增 `AgentPanel.vue`。
- 工具开关、审批队列、调用统计。
- Agent Chat 简版：输入 → 返回最终文本，不强求 SSE 流式。

验收：

- 前端能开关 QQ/Dashboard 工具渠道。
- 能批准/拒绝 pending。
- 能看到最近工具调用记录。

### Phase 5：浏览器核心动作

目标：引入浏览器自动化但控制范围。

- `browser.js` 实现核心 8 action。
- workspace browser state。
- snapshot refs。
- idle watchdog。
- URL 安全策略。

验收：

- Dashboard Agent 能打开页面、snapshot、点击、输入、截图。
- QQ 默认不可用。
- 浏览器 idle 后自动关闭。

### Phase 6：上下文压缩与 Skill 市场

目标：增强长任务能力和扩展性。

- `agent/context.js` 完整 compact。
- token 统计面板。
- skill-hub CLI 最小版。
- Skill 注入只注入 name/description，不把全文塞进 system。

验收：

- 长工具任务不会轻易爆上下文。
- skill 安装/启停可控。
- 不因 skill 注入导致 prompt 过大。

## 12. 风险与对策

| 风险 | 等级 | 对策 |
|---|---:|---|
| ReAct 影响普通聊天 | 高 | Agent Engine 独立，普通聊天默认不走 |
| tool_calls 被 fallback 吞掉 | 高 | `api.js` 测试覆盖，fallback 透传 tools |
| 危险工具越权 | 高 | 默认关闭，confirm，路径/命令检查 |
| QQ 群误触确认 | 中 | pendingId 或单 pending 窗口，绑定用户 |
| 工具结果撑爆上下文 | 中 | 统一截断，长结果外置 |
| 浏览器进程残留 | 中 | idle watchdog，stop 清理 |
| Dashboard 前端范围过大 | 中 | 先工具管理，再 Agent Chat |
| Skill 注入污染 prompt | 中 | 只注入索引，不注入全文 |
| 多模型 tool_call 兼容差异 | 中 | API 层适配，单元测试模拟 |

## 13. 最小可交付版本定义

不要把“完整 Agent 平台”作为第一目标。第一版只要满足：

1. QQ 显式命令能调用 `get_current_time` / `calculator`。
2. Dashboard 能看到工具列表、开关和调用日志。
3. `api.js` 正确支持 tool_calls 和 fallback。
4. 普通聊天完全不受影响。
5. 所有工具调用都有审计日志和结果截断。

做到这一步后，再决定是否继续文件工具、搜索、浏览器和 Skill 市场。

## 14. 与原计划的主要差异

| 原计划 | 修订后 |
|---|---|
| 直接在 `chat.js` 嵌 ReAct | 新增独立 `agent/engine.js`，普通聊天默认不变 |
| 10 天做完全部能力 | 6 个阶段，每阶段可验收可回滚 |
| 浏览器 34 action 全量移植 | 先 8 个核心 action，再扩展 |
| 工具首轮发送、第二轮不发 | 改为按渠道/开关暴露小工具集，每轮保持 schema 稳定 |
| Skill 市场较早进入 | 后置到 Agent、安全、Dashboard 稳定之后 |
| Dashboard 同时做 Chat + 管理 | 先工具管理/审批/统计，再做 Agent Chat |
| 危险工具列入早期工具表 | 默认关闭，必须安全系统完成后上线 |
