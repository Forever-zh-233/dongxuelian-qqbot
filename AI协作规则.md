# AI 协作规则

## 一、架构红线

1. **禁止重写或扩展 index.js 的中间件主业务逻辑。** 新增业务必须放独立模块；index.js 只允许做最小接线。确需改变主流程顺序时，必须先给方案并补 scenario。

2. **禁止循环依赖，尤其禁止 require('./index')。** 运行时配置从 runtime-config.js 引入；基础工具从 utils.js、constants.js 引入。

3. **禁止未经明确授权删除、放松、绕过 cascade-test.js 的防御性检查。** 新增模块、导出、语法检查和正负样例时，可以同步增强 cascade。

4. **测试默认不使用全局 fake timers。** 只有验证真实 setTimeout/setInterval 调度时，才允许显式 fakeTimers: true，并写明原因。

## 二、代码修改规范

0. **新增正则必须避免 ReDoS：**
   - 禁止嵌套量词（如 `(a+)+`、`(\w*\d*)+`）
   - `.*` 和 `.+` 必须配合尾部字面量锚定，或替换为 `.{0,N}` 有界版本
   - 用户输入正则优先用字面量串联，少用长 alternation

1. **文件行数预警。** lib/ 下文件超过 350 行时须指出风险并建议拆分。已超标的 index.js / chat.js 不追溯；新增逻辑超过 50 行须先提拆分方案。

2. **函数复杂度。** 超过 50 行的函数须分析是否职责单一；不单一则优先拆 helper 或独立模块。

3. **新增模块同步。** 新增生产模块时，同步更新：
   - package.json scripts.check 加 node -c 检查
   - cascade-test.js 的 modPaths、syntaxFiles、duplicateScanFiles
   - 函数导出放 expectedExports；非函数导出单独写类型断言

4. **代码风格。** 新增或本轮修改的 lib 代码禁止新增 var，统一 const/let。旧 var 不追溯，触碰相关函数时顺手清理。

5. **命名风格。** 文件名沿用现有 kebab-case/既有命名；函数变量 camelCase；常量 UPPER_SNAKE_CASE。

6. **utils.js 边界。** utils.js 只保留纯工具函数，不混入业务逻辑。

7. **配置访问。** 运行时配置（provider/model/baseURL/apiKey/thinking）从 runtime-config.js 获取；静态常量、路径、正则、阈值可以从 constants.js 引入。

8. **新增模块注释。** 新增 lib/*.js 模块时，必须在文件头部添加 MODULE 注释块，至少包含：
   - 职责范围（一句话）
   - 边界约束（禁止做什么，如"不调 AI API""不改 conversation"）
   - 状态说明（如有 Map/Cache）

9. **防止文件写坏、截断、误删。** 修改已有文件时默认使用小块 patch，不整文件重写，不用不可靠的“读一段再写回整文件”方式。
   - 改前先确认目标文件当前行数、关键上下文、相关测试 needle；尤其是 `test/scenarios/*.js` 和 `cascade-test.js`。
   - patch 必须带足上下文，只改目标片段；如果上下文匹配不稳，先重新读取附近 30-80 行再改。
   - 一次只改一个小区域；跨多个区域时分多次 patch，每次改后立即看局部 diff。
   - 改后立刻检查 `git diff -- <file>`、行数是否异常、关键测试名/needle 是否还在。
   - 如果文件行数突然大幅减少、代码块尾部消失、测试 needle 丢失，立刻停止，不继续叠加修改；先用 git diff 判断损坏范围，再按用户授权恢复或手工补回。
   - 禁止用 `git checkout -- file` 盲目恢复，除非确认该文件里没有用户或其他 agent 的未提交有效改动；否则先展示/说明风险。

## 三、测试规范

1. **新增行为 = 新增测试。** 任何新增用户可感知行为必须新增或扩展 test/scenarios 场景测试。

2. **修改 handler 后提醒。** 修改 handler.js 命令路由后，提醒运行 npm run test:quick + npm run test:scenario。

3. **测试独立性。** 每个场景测试用独立 withScenario 调用，确保状态不污染。

4. **安全 ruleset。** 改安全 ruleset 时，cascade 至少补一个"应拦截"样例和一个"应放行"样例。

## 四、AI 行为准则

1. **先摸底再动刀。** 改代码前先读：`rg` 搜入口/调用点/常量/测试覆盖，看现有风格，看 `package.json` scripts，看 dirty worktree。能从仓库确认的事实不靠猜。

2. **小施工面。** 按依赖顺序改：底层常量/配置 → 调用点 → 测试/fake/setup → cascade/文档。每步保持行为一致，不把无关重构混进来。

3. **最小 diff。** 最小替换调用点，不重排大段逻辑，不顺手格式化整个文件，不改编码，不改中文文案，不碰无关文件。

4. **行为测试优先。** 用户可见行为变更优先加 scenario；cascade 只守结构契约、导出、防线和架构红线，不用源码字符串扫描替代行为测试。

5. **不靠猜定位。** 测试挂了先分清楚是语法、导出、行为、环境 skip、断言脆弱，还是暴露真实旧 bug。阻塞当前任务或同改动面的明确小 bug 可以最小修复；无关大问题不顺手展开。

6. **验证闭环。** 相关文件先跑 `node -c`；常规收尾跑 `npm run check`、按范围跑 `npm run test:quick` / `npm run test:scenario` / `npm run test:plugins`，最后跑 `npm test` 和 `git diff --check`。环境限制导致 skip 要说明清楚。

7. **不越界。** 不做破坏性 git 操作，不回滚已有改动，不加 `_testOnly`，不引入新框架/新依赖/新数据库，不把小需求扩成大重构。

8. **部署守则。** 正式部署优先使用已有部署脚本；手工 SSH/SCP 修复时分批输送：传一个文件 → 确认 → 下一个。不一次覆盖整包，不覆盖服务器配置文件和 data 文件。

9. **新状态新 IO 就拆模块。** 小于 50 行但引入新 `Map`、文件 IO、定时器、协议、缓存所有权，优先独立模块。超过 80 行的新功能必须先提拆分方案。

10. **遇到未知先收窄，不扩大战场。** 测试或阅读代码时暴露问题，先分类再处理：
    - 阻塞当前任务的真实 bug：允许做最小修复，并在总结里说明。
    - 与当前改动同一调用链、同一文件或同一行为面，且修复很小、测试能覆盖的小 bug：允许顺手修。
    - 无关历史问题、需要产品取舍、会牵出大范围重构或迁移的问题：不顺手修，只记录、说明，必要时另开计划。

11. **把“最小修复”理解成“最小必要改动”，不是“只改一行”也不是“死守口号”。** 实际处理时按下面的情形走：
    - **语法、导出、类型、路径错误**：优先做最小修复，保证文件能加载、测试能继续跑；如果一个改动同时影响到多个相关文件，就把这条链路一起修完，不要留半截。
    - **当前功能直接失效**：先修阻塞点，再补对应测试或 scenario，确保能复现、能回归。
    - **同一条调用链上的连带问题**：如果修主问题必然要碰到相邻 helper、配置或桥接层，可以一起改，但只限同一行为面，不外溢到别的功能。
    - **资源泄露、内存飙升、僵尸进程、重复定时器**：不能只修表面症状，必须把成功、失败、超时、取消四条路径都收住，最后再做一次真实输入模拟跑或压测式验证。
    - **搜索、联网、浏览器、Agent 路由、上下文桥接**：测试绿不等于真的能用，必须用真实输入跑一轮，确认实际触发了工具链、返回了有效来源、日志也能对上。
    - **配置、权限、开关、部署脚本**：如果改动只影响一个开关或一个入口，就尽量单点修；如果同一问题已经暴露出配置和运行时都错了，就把这两个层一起修掉。
    - **无关历史债**：看见也先别顺手清，除非它正好卡住当前任务、而且修复很小、风险可控。否则只记录，不扩大战场。

12. **实际工作中优先级是：先让链路恢复，再让实现变干净，最后才考虑好看。**
    - 先恢复可用性：能跑、能连、能回包、能关资源。
    - 再补验证：把刚才修的真实行为写进测试或 scenario。
    - 最后再整理代码：拆 helper、收命名、清重复逻辑。
    - 如果你手里同时有“能救火的小修复”和“会牵出重构的大优化”，先选前者，除非用户明确要重构。

## 五、蒸馏版工作流：我平时真正会这么做

这部分不是口号集，而是把在这个仓库里反复证明有效的做法，整理成别人也能直接照着走的版本。

### 1. 先确认事实，再决定动作

- 先读仓库里能直接确认的东西：入口、调用点、测试、配置、日志、现有实现。
- 找不到证据就不要替现实补剧情；不确定就写成“待验证”或“推测”，不要装成已确认。
- 近期变化、环境状态、服务器资源、外部接口可用性，默认都需要重新核实，不靠旧记忆。

### 2. 先找到最短可验证路径

- 先做能编译、能跑、能复现的一小步，再考虑优化。
- 能用一个最小输入验证的，就不要上来改一大片。
- 如果涉及外部链路，优先做真实输入 smoke test，而不是只看测试名字或源码匹配。

### 3. 先修阻塞点，再补回归

- 当前功能直接失效时，先把链路救回来。
- 修复完成后立刻补对应测试或 scenario，把刚刚修过的行为锁住。
- 如果 bug 和当前改动同链路、修复很小，可以顺手一起修；如果会牵出重构，就先停。

### 4. 先保守落地，再决定要不要优化

- 默认保留现有接口、文案、返回结构、配置名、环境变量名。
- 只有当它们本身就是问题根源时，才改约定。
- 对行为敏感的改动，先以最小 diff 证明正确，再考虑抽 helper、拆模块、改命名。

### 5. 先关资源，再谈结果

- 浏览器、页面、reader、stream、timeout、child process、临时文件、缓存，都要在成功、失败、超时、取消路径里关干净。
- 看到内存泄露、僵尸进程、重复定时器，不只修症状，要检查整个生命周期。
- 修完以后要看真实资源表现，而不是只看页面有没有返回。

### 6. 先说清楚我知道什么，不知道什么

- 已确认的事实单独说，推测单独说，待验证的地方单独说。
- 如果结论依赖外部服务、实时状态或运行环境，就明确告诉自己这不是静态事实，必须再测一次。
- 不把“我觉得”“大概”“应该是”伪装成结论。

### 7. 先让别人能复用

- 以后可能再踩的坑，尽量写进教训总结、协作规则或测试里。
- 只属于一次性排障的细节，不要扩成通用规律。
- 规则要写成别人照着就能做的动作，而不是抽象口号。

## 六、防幻觉和提准率：我会怎么做

### 1. 防幻觉

- 只把工具、代码、日志、测试、运行结果里能看到的内容当事实。
- 不能确认的地方直接说“我没确认到”“还需要验证”，不编造。
- 发现自己在用记忆补空白时，先停下来回到仓库和运行结果。
- 当代码、测试、日志、文档互相冲突时，不硬选一个信念，先复现，再依据可观察结果判断。

### 2. 提高代码准确率

- 先看现有模式，再动手写新东西，尽量沿用仓库已有的工具、接口、风格和测试方式。
- 先做局部验证：`node -c`、目标测试、真实 smoke test，再放大到整套测试。
- 用户能感知的行为变更，优先加 scenario 或回归测试。
- 对搜索、联网、浏览器、缓存、权限、并发这类高风险点，默认增加防线和退出口，不等出事故再补。
- 遇到模糊语义时，优先从仓库现有能力里找最接近的实现，不自己脑补一套新约定。

### 3. 遇到未来问题时的默认处理

- **搜索质量差**：先改 query 规划，再放宽不合理过滤，再补正文读取，最后才考虑换搜索源。
- **内存飙升**：先看进程和资源曲线，确认是不是泄露，再查并发、超时和退出口。
- **上下文混乱**：先查消息归属、历史桥接和缓存注入点，不先怪模型。
- **部署异常**：先确认进程、端口、环境变量、日志，再考虑改代码。
- **权限不够**：先查工具暴露范围和白名单，再决定是扩权限还是改路径。
- **结果看起来对但不可信**：强制做一次真实输入验证，避免“测试过了但没真的生效”。

### 4. 给其他 AI 的可复用原则

- 不要急着给结论，先给证据。
- 不要急着扩范围，先让当前链路恢复。
- 不要把猜测装成事实，必要时就标注不确定。
- 不要只看静态代码，外部集成必须跑真输入。
- 不要把修复和重构混在一起，先止血，再整理。

## 七、给其他 AI 的编程课本：我是怎么尽量写准的

这一节不是训练内幕，也不是神秘配方，只是把“为什么我会这样写代码”整理成可复用的方法。对其他 AI 来说，最重要的不是模仿语气，而是模仿决策方式。

### 1. 我不是先“会写”，而是先“会收敛”

- 先判断任务边界，再决定改哪里。
- 先识别是否是当前任务的阻塞点，再决定要不要顺手修。
- 先把问题缩到最小，再动手写代码。

这会让输出看起来更谨慎，但本质上不是保守，而是先把错误空间缩小。错误空间越小，越不容易写歪。

### 2. 我写得稳，靠的是“先验证，再扩展”

- 先读现有代码和测试，找仓库里已经成立的模式。
- 先做最小验证，确认能跑通，再谈优化和重构。
- 先修一个真实可见的问题，再把验证补进测试。

这不是拖延，而是把“猜对”变成“证实对”。对代码任务来说，证实比自信更值钱。

### 3. 我尽量不胡编，因为我默认自己可能错

- 对我没有直接看到的东西，我默认保持怀疑。
- 遇到日志、代码、测试、文档互相矛盾时，我不会立刻替其中一个编理由，而是先复现。
- 如果只能推测，我会把推测和事实分开说。

对其他 AI 来说，最重要的习惯不是“回答得快”，而是“别把空白补成错误事实”。很多明显的错误，都是因为模型太快把空位填满了。

### 4. 我喜欢把改动做小，是因为小改动更容易证伪

- 小 diff 更容易检查有没有误伤别的逻辑。
- 小 diff 更容易定位问题来自哪里。
- 小 diff 更容易回滚，也更容易复盘。

所以我常常会优先做“最小必要改动”，不是因为我只会小修，而是因为小修最容易让真相暴露出来。

### 5. 我看重测试，不是为了装饰，而是为了把行为钉住

- 语法检查负责让代码先能活。
- 结构测试负责让接口别乱。
- 场景测试负责让用户能感知的行为别漂。
- 真实 smoke test 负责验证外部链路真的通了。

不同层的验证不是重复劳动，它们防的是不同类型的错。只靠一种测试，通常会漏掉另一种失误。

### 6. 我会主动收紧风险，而不是等事故教我

- 搜索、联网、浏览器、并发、缓存、权限、部署，这些地方我会默认先加退出口和限制。
- 一旦发现资源泄露、上下文污染、错误归属、缓存泄漏，我会优先修生命周期，而不是只修表面输出。
- 看起来“只是个功能增强”的东西，如果碰到 IO、状态、外部依赖，我会按高风险处理。

这不是悲观，是因为这些地方最容易出现“单测没错，线上炸了”的问题。

### 7. 我对“更像人”没有执念，对“更像对的代码”更有执念

- 不需要为了好看而扩写解释。
- 不需要为了显得聪明而写复杂结构。
- 不需要为了给出完整答案而补自己没确认的内容。

如果一个回答或者一段代码能更准确、更稳、更容易复现，我通常会选那个版本，即使它看起来没有那么花哨。

### 8. 给其他 AI 的学习建议

- 先学会读仓库，再学会改仓库。
- 先学会承认不确定，再学会补证据。
- 先学会做小而确定的修复，再学会做优雅重构。
- 先学会让代码能验证，再学会让代码漂亮。
- 先学会不乱扩散，再学会解决更大的系统问题。

### 9. 可以直接照抄的工作习惯

- 改代码前先找入口、调用点、测试、配置、日志。
- 看到异常先分类，不要上来就猜根因。
- 涉及真实外部链路时，必须跑真输入。
- 涉及资源和生命周期时，必须检查关闭路径。
- 涉及用户可见行为时，必须补回归测试。
- 涉及不确定结论时，必须把事实和推测分开。

### 10. 最后一句最重要的话

我写得比较稳，不是因为我天生知道答案，而是因为我把“不要瞎猜、不要乱扩、不要漏验证”当成默认动作。其他 AI 如果想学这一点，最有效的办法不是模仿措辞，而是把这三件事变成自己的习惯。

## 八、写代码口诀版

- 先摸底，再动刀。
- 先确认，再下结论。
- 先收窄，再扩展。
- 先修阻塞，再谈优化。
- 先保行为，再整理代码。
- 先做最小验证，再放大测试。
- 先看真实链路，再看静态结果。
- 先关资源，再看输出。
- 先写回归，再改实现。
- 先说清不确定，再补证据。
- 先让链路恢复，再让代码变美。
- 先止血，再缝合。

## 九、卡住时怎么发散：一条路不通就换路，不原地打转

### 1. 先判断是“路错了”还是“路太窄了”

- 如果同一个方向连续验证失败，不要默认继续加大力度，先判断是不是路径本身就错了。
- 如果一条路径只证明了“这条路不行”，不要把它误判成“整个问题无解”。
- 先区分是输入不对、环境不对、实现不对，还是验证方法不对。

### 2. 遇到 bug 时，我会怎么发散

- **先拆层看**：输入层、转换层、状态层、输出层、外部依赖层，逐层排查。
- **先换视角**：从报错、日志、测试、数据、配置、调用链五个角度交叉看，不只盯一个点。
- **先换路径**：如果直接修实现没效果，就转去查配置、调用顺序、缓存、边界条件、环境变量、外部接口。
- **先缩小范围**：构造最小复现，确认 bug 到底在哪一层出现。
- **先找反例**：找一条成功路径和一条失败路径对比，看看差异在哪里。

### 3. 遇到乱码时，我会怎么做

- **先确认是不是编码问题**：是文件编码、终端编码、传输编码、日志编码，还是内容本身就坏了。
- **先换读法**：同一个文件，用不同读取方式核对，不直接相信单次显示结果。
- **先看源头**：乱码如果出现在传输后，优先查传输方式和落盘编码，不先怀疑业务逻辑。
- **先做最小复现**：把乱码单独截出来，确认是读取、写入、还是显示层有问题。

### 4. 一条路不通时的发散规则

- 不要在同一错误假设上反复加码。
- 不要只从代码改代码，也要看配置、环境、数据、权限、部署和外部依赖。
- 不要把“没修好”理解成“继续原路加深就会修好”。
- 不要在没有新信息时重复同样动作，除非你是在验证时间因素或随机性。

### 5. 我会怎么选更好的路径

- 选能更快复现根因的路径。
- 选能更容易证伪假设的路径。
- 选对系统副作用更小的路径。
- 选能顺手留下测试或回归证据的路径。
- 选能把问题收束成局部修复的路径。

### 6. 发散不是乱猜

- 发散的前提是“每条新路径都带着一个清晰假设”。
- 每换一条路，都先说清楚这一条是在验证什么。
- 如果新路径也失败，就用结果反推假设，而不是把所有失败堆成一团。
- 发散的目标是找到更短的真路径，不是把问题讲得更热闹。

### 7. 给其他 AI 的卡住时口诀

- 一条路不通，先问是不是路错了。
- 一条线卡死，先换一层看。
- 一处乱码，先查编码和传输。
- 一次失败，不等于全局失败。
- 一次验证没过，先换假设，不先换幻想。
- 一直重复同样动作，只会得到同样结果。

## 十、模拟端到端测试：不要把 NapCat 未登录当成借口

### 1. 先分清两种端到端

- **真实端到端**：真实 QQ 消息 → NapCat → OneBot WebSocket → Koishi → 插件回复。
- **模拟端到端**：构造 fake session → fake Koishi ctx → 真实 `plugin.apply(ctx)` → 真实 middleware → `session.send()`。

NapCat 账号没登录，只会挡住真实 QQ 传输链路，不会挡住模拟端到端测试。

### 2. AI 必须听懂的核心区别

- 真实端到端测的是：QQ、NapCat、OneBot、Koishi adapter、插件全部串起来能不能工作。
- 模拟端到端测的是：NapCat 已经把消息交给 Koishi 之后，插件自己的业务链路能不能工作。
- 所以不能因为 NapCat 未登录，就说“业务链路没法测”。

### 3. 模拟端到端的标准流程

```text
makeSession() 造一条用户消息
  → makeCtx() 造一个 Koishi 环境
  → plugin.apply(fakeCtx) 注册真实 middleware
  → runMiddleware(fakeCtx, fakeSession)
  → 检查 session.sent / internalCalls / logs / fetch calls
```

这个流程不需要 QQ 在线，也不需要 NapCat 登录，但它会真实跑插件 middleware。

### 4. 模拟测试应该覆盖什么

- 用户消息是否触发插件。
- @bot、群聊、私聊、引用消息是否被正确识别。
- 普通聊天、Agent 路由、fallback、web_search 注入是否走对。
- prompt、tools、messages、上下文桥接是否正确。
- 最终是否调用 `session.send()` 或 `bot.internal`。
- 是否泄漏缓存、思考内容、内部提示词、API key。
- 临时 data、配置、人格、skill、上下文是否按预期读写。

### 5. 真实外部能力要单独 smoke test

模拟端到端不能证明外部世界真的通了。  
像 `web_search`、真实 HTTP、浏览器、服务器 SSH、NapCat 登录状态，需要单独做真实 smoke test。

正确分层是：

```text
模拟端到端：验证插件业务链路。
工具 smoke test：验证工具自己真的能访问外部资源。
真实端到端：验证 QQ/NapCat/Koishi/插件全链路。
```

### 6. 正确汇报方式

如果 NapCat 未登录，应该这样说：

```text
真实 QQ → NapCat → Koishi 的传输级端到端测试暂时无法执行，因为 NapCat 未登录。

但已执行/可以执行模拟端到端测试：
使用 fake Koishi ctx + fake session 模拟消息进入 Koishi 后的业务链路，
真实运行插件 middleware、chat、agent、fallback、web_search 注入等逻辑。

未覆盖范围：真实 QQ 登录、NapCat 收发、OneBot WebSocket、Koishi adapter 实际连接。
```

不要说：

```text
NapCat 没登录，所以端到端测试做不了。
```

这句话太粗糙，会把“真实传输链路做不了”和“业务链路模拟测不了”混成一团。

### 7. 给其他 AI 的一句话

NapCat 未登录，不等于不能模拟用户消息。  
真实 QQ 链路测不了，就用 fake session 跑真实 middleware；  
外部工具不确定，就单独 smoke test；  
不要拿一个边界不可用，否定所有可测链路。

## 十一、重要：不要写自欺欺人的测试

### 1. 先测真实失败输入

用户报告什么输入坏了，第一条测试就用这个输入。  
不要只写一个更容易成功的相似输入。

例：用户看到 `莲 → (Agent 未获取到有效回复)`，测试就必须覆盖 `莲` 或 `@bot 莲`，并断言不允许出现 `Agent 未获取`。

### 2. 不要只测 happy path

模型被 mock 成成功回复、工具被 mock 成成功结果，只能证明成功路径能发出去。  
这不能证明真实问题修好了。

必须同时测：

- 模型空回复。
- reasoning-only。
- 工具失败。
- 工具有结果但模型没最终回答。
- 短消息误进 Agent。
- fallback 文案是否泄漏给用户。

### 3. 不要 mock 掉被测对象

可以 mock 外部 API，但不要把正在验证的核心函数替换成固定成功结果。  
如果替换了 `webSearch.execute`，这个测试就不能宣称验证了真实 `web_search`。

### 4. 断言要对准事故

只断言 `session.sent.length > 0` 太弱，因为错误兜底也是回复。  
应该断言：

- 不包含 `Agent 未获取`。
- 不包含 reasoning。
- 不包含内部 prompt/cache 标记。
- 走 Agent 时必须有 tools。
- 不该走 Agent 时必须没有 tools。

### 5. 测试命名要诚实

用了 mock fetch，就叫 mocked HTTP。  
没连 QQ/NapCat，就叫模拟端到端。  
只测 Koishi middleware，就不要叫真实全链路。

### 6. 不要截断测试输出骗自己

不能只看前 30 条 OK 就宣布通过。  
必须看最终 summary，确认 `failed: 0`。  
如果只截取局部输出，必须说明这不是完整测试结果。

### 7. monkey patch 必须 finally 恢复

凡是改了 `global.fetch`、模块函数、环境变量、配置缓存，都必须放在 `try/finally` 里恢复。  
否则一个测试失败会污染后续测试，制造假通过或假失败。

### 8. 一句话

测试“我安排成功时系统能成功”不够。  
真正有意义的测试，要证明“用户真实失败时系统不会再失败”。

### 9. 测试报告四问

任何声称“已通过模拟端到端 / e2e / 回归测试”的报告，都必须回答这四个问题：

- 复现了用户哪条真实失败输入？
- 断言了哪个失败现象不会再出现？
- 哪些依赖被 mock 了？
- 因为 mock，哪些真实链路仍未覆盖？

答不上这四问，就不能说“问题已验证修复”。最多只能说“某条 happy path 测试通过”。
