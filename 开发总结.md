# 东雪莲 QQ 机器人开发成果总结

## 一、服务器部署与基础架构修复

从零开始在 120.55.246.12（Ubuntu 22.04）部署了完整的 QQ 机器人系统，架构为：

```
NapCat（QQ 登录, WS 服务端 :8080）
  → Koishi v4.18.11（adapter-onebot, WS 客户端）
  → dongxuelian-ai 插件
  → 多 API 后端（OpenCode Go / DeepSeek 官方 / 阿里云）
```

修复了三大基础设施问题：

**1. NapCat Worker 进程崩溃：** 添加 `--disable-gpu` 启动参数解决 GPU 兼容性问题。

**2. adapter-onebot 无法连接 NapCat：** 根因是 koishi.yml 格式错误，将 server 插件格式改为根级 port/selfUrl，协议从 ws-reverse 改为 ws 正向，因为 NapCat 是 WebSocket 服务端，Koishi 须以客户端身份连接。

**3. @satorijs/core 版本冲突：** Koishi 4.x 需要 core ^4.6.0，但 adapter-onebot 依赖 satori 3.x 锁死 core 3.7.0。通过 package.json 添加 overrides 锁定版本，同时创建 patch.js 通过 NODE_OPTIONS 预加载补齐 3.x 缺失的 Session API（resolve、stripped、send）。

---

## 二、识图功能（多模态 + OCR 双轨）

实现了完整的图片识别管线：

从 QQ 消息中提取图片 file 参数 → 调用 NapCat 的 get_image API 获取本地缓存路径 → 读取本地图片文件 → base64 编码 → 发送给多模态模型。

同时集成了 tesseract OCR（chi_sim+eng）作为非视觉模型的回退方案，当模型不支持多模态时自动走 OCR 管线。

中间件中添加了白名单和概率控制：只有白名单群中的图片才会触发识图，非白名单群和无 @ 的图片静默跳过。非多模态模型（如 deepseek-chat）自动跳过识图流程。

---

## 三、转发消息处理

实现了完整的合并转发消息识别管线：

**外层转发：** 检测 session.content 中的 `<forward id="xxx"/>` 或 CQ 码格式 → 调用 callGetForwardMsg 通过 NapCat API 拉取消息节点 → 转换为统一格式 → 通过 summarizeForwardNodes 格式化为带层级的可读文本 → 注入 AI 上下文。

**嵌套转发：** 检测消息段中的 forward 类型或 raw_message 中的 CQ 码 → 递归调用 callGetForwardMsg 尝试拉取 → NapCat 明确返回"内层消息无法获取"时降级显示 `[嵌套转发：内容暂不可见]`。

**格式优化：** 加入层级标记（`[对话]` / `└─ [内层转发]`），接入 MAX_FORWARD_NODES（50 条）和 MAX_FORWARD_DEPTH（4 层）做截断控制，超出部分提示"还有 N 条消息未显示"。

**昵称净化：** 过滤不可见字符（韩文填充符、零宽空格等 U+200B-U+11FF 范围），空昵称回退为"群友"。

---

## 四、引用消息处理

实现了引用消息的内容注入：直接从 session.quote 提取被引用的消息内容，解析文本/图片/表情/@/转发/视频/语音/文件等多种消息段类型，将引用内容以 `[引用内容]xxx[以上是对方说的话，不是在对你说]` 格式注入用户消息，同时去掉重复的系统消息注入，避免 AI 混淆。

---

## 五、模型切换系统（三分层父子菜单）

设计了三层切换体系：`切换模型` → 选择供应商 → 选择模型。支持三个供应商：

- **OpenCode Go**：14 个模型（千问 3.5、千问 3.6、GLM-5、GLM-5.1、Kimi K2.5、Kimi K2.6、DSv4pro、DSv4、MiMo-V2-Pro、MiMo-V2-Omni、MiMo-V2.5-Pro、MiMo-V2.5、MiniMax M2.7、MiniMax M2.5）
- **阿里云 DashScope**：qwen3.5-plus、qwen3.6-plus
- **DeepSeek 官方**：deepseek-chat、deepseek-v4-flash、deepseek-v4-pro

切换时自动切换 API 地址和对应密钥。为 OpenCode Go 的 DeepSeek 模型添加了 reasoning_content 回退处理（当 content 为空时使用 reasoning_content）。

## 六、联网搜索能力

在 getSearchCapability 函数中添加了 qwen 模型识别逻辑，使通过 OpenCode Go 调用的 qwen 模型也能使用 enable_search 参数进行联网搜索，突破了原有代码只识别 DashScope 和 OpenAI 官方 API 的限制。支持 dashscope-chat、openai-chat-search、openai-responses 三种搜索模式。

## 七、消息队列与稳定性

给 enqueueForChannel 添加了 60 秒超时机制，防止单个 API 请求卡死整个消息队列。修复了 config 作用域问题（config is not defined、vConfig is not defined），在视觉处理代码块中单独加载配置。修复了 helpAI 菜单重复发送的问题。

## 八、白名单与概率回复系统

实现了群聊白名单加概率触发的双层过滤机制。白名单群中的消息按设定概率回复，非白名单群全部跳过。每个群可单独设置概率（如 30%、5%），带有 warmup 机制（50 条消息内不递增，之后每条递增 2%）。

修改了 shouldSkipForRandomReply 逻辑，使图片也参与概率判断（修复了图片总是回复的 bug）。同时保留了 @ 提及和私聊的始终回复逻辑。随机回复时添加 `[群聊刷到]` 标记，让 AI 以群友插话口吻回应而非误以为被直接召唤。

## 十、提示词工程与人设优化

**政治红线：** 写入两套人格，包含红线示例（党和国家领导人、领土主权、政治运动、历史事件等）、隐喻/谐音/反讽识别提醒、标准拒绝话术。

**脏话替代词：** 嘴臭人格中加入屎→史、妈→老冯、傻逼→沙鼻、废物→飞舞、智障→智战等谐音替代规则，同时要求减少屎尿屁输出，并提醒模仿例子时切换代替词。友善人格中删除替代规则（因为友善模式不骂人）。

**字数控制：** 取消 trimReply 的硬截断，改为提示词层面的软性引导"评价类回复控制在两三句话以内"。

**评价类回复：** 加入引导"先看完内容再回应，有值得评论的地方直接说"，避免 AI 盲目输出。识图提示词改为"直接评价这张图，不要描述过程"。

**随机回复插话：** 加入"当前消息是你在群聊里看到的，对方不是在叫你"的上下文标记，防止 AI 用回应对方的口吻回复。

**回复字数放宽：** max_tokens 从 500 提升到 1000，支持更长的回复。

**摆烂机制：** 当 completion_tokens 超过 600 时，AI 自动输出"太麻烦不想了，摆烂了"等随机摆烂回复。

## 十三、服务器部署教训集

### 1. `pkill -f` 误杀 SSH
```bash
# 错误 ❌ — 会杀 SSH 守护进程导致服务器失联
pkill -f "start"

# 正确 ✅
pkill -f 'koishi/lib/worker'
```
`pkill -f` 用ERE匹配进程名，`"start"`会匹配到包含`start`的所有进程包括sshd。

### 2. SCP 编码损坏
PowerShell `Get-Content file | ssh ... Set-Content` 会输出 UTF-16 LE带BOM，损坏中文字符和API key。必须直接用 `scp` 二进制传输：
```bash
scp file user@host:/path/    # ✅ 正确
```

### 3. API Key 格式
| 供应商 | Key 前缀 | 文件 |
|---|---|---|
| DeepSeek | `sk-*` | `ai-deepseek-key.txt` |
| mimorium | `tp-*` | `ai-mimorium-key.txt` |
| opencode | `sk-*` | `ai-openai-key.txt` |
| dashscope | `sk-*` | `ai-dashscope-key.txt` |
| GLM | 无固定前缀 | `ai-glm-key.txt` |

服务端和本地的 key 相互不通用，服务端 key 不能复制到本地使用（反之亦然）。

### 4. 进程残留导致双实例
重启时若 `pkill` 没杀干净，旧进程孤儿化后继续监听端口，新进程启动失败。必须 `pkill -f 'koishi/lib/worker'` 后 `sleep 3` 确保释放。

### 5. 重复部署导致双实例
部署时如果第一次 `pkill + start` 已经成功（日志能看到 `loaded`），但没仔细确认就再次 `nohup start`，会跑起两个 bot 实例。两个实例同时处理同一条消息，群友会收到重复回复。

**避免方法**：部署后等 15 秒，确认日志出现 `dongxuelian-ai.*loaded` 且只有一条，不要盲目重试。清理时用 `pkill -f 'koishi/lib/worker'` 精确杀进程，不要用 `pkill -f 'koishi'`（过于宽泛）。

### 6. koishi.yml 格式错误
`server` 插件必须以根级 `port`/`selfUrl` 配置，`adapter-onebot` 必须用 `ws` 协议正向连接 NapCat（NapCat 是 WS 服务端）。

## 十五、部署脚本设计

项目根目录包含三类脚本：

| 脚本 | 用途 | 运行位置 |
|---|---|---|
| `setup.sh` | 首次部署 | 复制到服务器后 `bash setup.sh` |
| `scripts/deploy-package.sh` | 更新重启 | 单个插件: `bash scripts/ai.sh`，全量: 用 SSH 逐包部署 |

**setup.sh 用户流程：**
1. 编辑脚本顶部 QQ_NUMBER 和 ADMIN_QQ
2. `bash setup.sh` → 自动安装 NapCat + Koishi + 全部依赖
3. 看到二维码 → 掏出 QQ 扫码登录
4. 按提示输入 DeepSeek API key（首选必填，最流行的模型）
5. 可选输入 mimorium / opencode / dashscope / GLM key
6. 自动启动 bot

**设计原则：** 代码更新后必须同步更新所有脚本。新增 API 供应商、修改配置文件路径、新增依赖等，都要对应修改 setup.sh 和 scripts/ 下的部署脚本。

## 十七、世界观注入与人格匹配原则

**人格与世界观严格对应。** 每个世界观 lore 必须按照所属人格注入，禁止混入不相关人格：

| 世界观 | 适用人格 | 例外 |
|---|---|---|

### repeat.js 拆分失败教训

第 2 步：源文件加 import，跑 node -c 确认语法

第 4 步：测试通过后删掉注释掉的旧代码，再跑一次测试

每步之间 `node -c` + `npm run test:quick` 验证，不过不进入下一步。后三步中任何一步失败，退回上一步重来。

## 二十、群聊日报插件（koishi-plugin-daily-report）

### 功能概述

独立插件，为 QQ 群生成每日聊天分析报告图片。两条命令：
- `群聊日报` — 基础模式（话题总结 + 今日圣经，~1000 tokens）
- `群聊详细日报` — 完整模式（+ 群友画像 + 群聊锐评，~2500 tokens）

### 架构

```
index.js          中间件入口（命令匹配、白名单、冷却）
config.js         配置（DATA_DIR fallback、超时、模板调试）
data-collector.js 读取 today-cache-{key}.json，统计消息数据
ai-analyzer.js    复用主插件 runtime-config.js + api.js，compress→analyze 两阶段
html-renderer.js  模板加载、时段选择、占位符替换、Puppeteer 截图
models.js         数据模型工厂函数
analyzers/        独立分析器（topic/user-title/golden-quote/chat-quality）
templates/        HTML 模板（light.html 亮色 / dark.html 暗色）
test/             plugin-test.js（29 项单元测试）
```

### 关键设计决策

1. **复用主插件 API**：禁止自己写 AIClient，直接 require 主插件的 `runtime-config.js` + `api.js`，获得 fallback 链、超时、重试
2. **白名单共用**：读取主插件的 `summary-whitelist.json`，群里启用消息缓存即自动启用日报
3. **中间件拦截**：`ctx.middleware()` 匹配 `群聊日报`/`群聊详细日报` 后 bare `return` 阻止后续插件
4. **compress→analyze 两阶段**：先批量压缩消息（100条/批），再分析压缩结果，节省 60% tokens
5. **Puppeteer 信号量**：最多 2 个并发实例，30 秒超时，防止内存溢出
6. **模板时段选择**：6:00-18:00 用 light.html（亮色初音绿），18:00-6:00 用 dark.html（暗色深蓝黑）
7. **占位符系统**：HTML 模板用 `{{variable}}` 占位符（非 `${}` 避免 JS 冲突），运行时 split+join 替换
8. **QQ 头像**：今日圣经和群友画像均使用 `https://q1.qlogo.cn/g?b=qq&nk={userId}&s=100` 抓取真实头像
9. **DATA_DIR fallback**：本地未设 `DONGXUELIAN_AI_DATA_DIR` 时自动定位到主插件 data 目录

### 模板文件

| 模板 | 时段 | 风格 |
|---|---|---|
| `light.html` | 6:00-18:00 | 白底初音绿，SVG 柱状图 |
| `dark.html` | 18:00-6:00 | 深蓝黑底，渐变标题，LIAN 水印 |

### 开发中踩过的坑

1. **中间件 return 语义**：`return session.send()` 有返回值会终止但语义不清，必须 `await session.send()` + bare `return`
2. **`${}` 模板冲突**：HTML 模板用 `${}` 会和 JS 模板字符串冲突，改用 `{{}}`
3. **scp -r 不删文件**：本地删除的文件不会从远程删除，服务器残留 paper.html/miku.html 导致崩溃
4. **DATA_DIR 环境变量**：index.js 直接读 `process.env` 而非 config.js 的 DATA_DIR，导致本地白名单读不到
5. **dark.html CSS 损坏**：旧版 CSS 每条规则末尾多 `}}`，类名和渲染器不匹配，需完全重写
6. **Puppeteer 并发**：多个群同时请求会启动多个 Chromium（各 ~200MB），无信号量会 OOM

### 测试

`npm run test:plugins` 覆盖 29 项：模块加载、导出检查、models 单元测试、data-collector 校验、中间件注册、命令匹配。

友善（0-60） → 阴阳（60-90） → 嘴臭（90-100，管理员开关）
```

### 架构

| 文件 | 职责 |
|------|------|
| `retaliation.js`（新建） | `calculateRetaliationScore()` — AI 评分 0-100，正则命中后才调用 |

### 三态行为

| 等级 | 反击值 | 默认人格 | 自定义人格 |
|------|--------|---------|-----------|
| 0 = 友善 | ≤ 60 | 正常回复，忽略攻击 | 完全绕过，使用人格自有 skill |
| 1 = 阴阳 | 61-90 | 阴阳怪气、讽刺、不带脏话 | 完全绕过 |

3. **阴阳模式默认开启**：不需要开关文件，`YINYANG_MODE_FILE` 不存在即为开启状态

## 二十四、引用消息归属修复

### 问题

群友引用消息时，AI 把引用内容归到回复发送者名下，导致混淆（"这他妈这句话不是我说的"）。

<user>
昵称：水落烟雨          ← 回复发送者

```

### 修复

改 `chat.js` 的引用消息处理，从 `session.quote` 中提取原作者昵称：

```
[引用 ru 的原话]
ru的原话
[以上是引用内容，不是 水落烟雨 说的]
```

`session.quote.userId`、`session.quote.nickname` 等字段原本已存在但从未被读取。

## 二十八、开发教训总结

### Dashboard 硬编码问题

端口、路径、Shell 命令不能硬编码：

| 硬编码 | 修复 |
|--------|------|
| `const PORT = 5150` | `process.env.DASHBOARD_PORT \|\| 5150` |
| `bash /root/koishi-app/restart.sh` | `process.env.KOISHI_DIR \|\| __dirname` |
| `=== 'on'` 维护检测 | `!!readFileSync(...)` 非空检测 |

### NapCat Bearer Token

NapCat 有两个独立的认证体系：
1. **OneBot HTTP API** — token 来自配置文件
2. **WebUI（QQ 管理）** — 动态 token，启动时从 webui.json 读取，运行时可能被刷新

当前 WebUI 的 Bearer token 验证未通过。替代方案：Dashboard 通过 `/dashboard/napcat/*` 反向代理到 NapCat WebUI，在服务器本地访问。

### 多 worker 进程隔离

koishi 多 worker 模式下，每个 worker 有独立的内存空间和模块缓存。`channelTodayCache`、`channelSharedCache` 等 Map 实例在各 worker 之间不共享。跨 worker 数据共享必须通过磁盘文件。

### 重启流程必须用脚本

手动拼命令重启容易漏步骤。`restart.sh` 做 5 件事：杀进程 → 等端口 → 设环境变量 → 启动 → 验证日志。验证时通过时间戳标记区分新旧日志，防止旧日志中的成功记录导致误报。

## 二十九、Dashboard 视觉升级

### 毛玻璃效果

卡片使用半透明 `rgba` 背景 + `backdrop-filter: blur(12px)` 毛玻璃效果。CSS 变量化后支持 dark/light 双主题切换，切换按钮在右上角。

### 鼠标拖尾

使用 Canvas 渲染的渐变色线条拖尾（青→粉→紫渐变），加上星星点点的溢出粒子（随机颜色、闪烁、缓慢消散）。鼠标停止后线条通过 lerp 方式平滑缩短（旧点以 10% 速度移向下一个点，靠近后删除），不会突兀截断。

### 毛玻璃卡顿的教训

首次实现时所有卡片使用 `backdrop-filter: blur(16px)`，加上 `<Transition>` 动画切换组件，导致连续切换标签页时严重卡顿。解决方案：
- `blur` 从 16px 降到 12px（性能与美观平衡）
- `<Transition>` 改为 `<KeepAlive>`，组件缓存不卸载，切换瞬间恢复
- KeepAlive 缓存的组件需要通过 `onActivated` 钩子重置按钮状态，避免锁死

## 三十、管理员密码系统

### 二次验证密码体系

| 密码 | 默认值 | 用途 |
|------|--------|------|
| 访问密码 | `DASHBOARD_PASSWORD` | 登录 Dashboard |
| 管理员密码 | `DASHBOARD_ADMIN_PASSWORD` | 敏感操作二次验证；变量名保留兼容旧版本 |

### 受保护的操作

所有涉及 Bot 配置变更的操作都需要管理员密码二次验证：启动/停止 Koishi、修改 API 地址/Key、新增人格、修改白名单、更换 QQ 号、开/关维护模式、修改密码。

### 实现

- 前端统一使用 `window.showAdminDialog(msg, callback)` 全局弹窗，由 App.vue 的 `AdminDialog` 组件渲染（`z-index: 10000`，高于鼠标拖尾的 9999）
- 管理员 token 存储在 `localStorage`，有效期 1 小时，过期后自动清除
- 后端敏感 API 路由检查 `X-Admin-Token` 请求头
- 密码存储在 `data/dashboard-admin-pwd.txt` 和 `data/dashboard-access-pwd.txt`，环境变量 `DASHBOARD_PASSWORD` / `DASHBOARD_ADMIN_PASSWORD` 可覆盖

### 失败教训

首次实现时 `AdminDialog` 的 `visible` 默认设为 `true`，导致弹窗默认挡住整个界面。且弹窗 `z-index` 低于鼠标拖尾 Canvas（9999），按钮点不了。正确做法：`visible` 默认 `false`，在 API 返回 403 时才弹窗。

## 三十一、Dashboard 稳定性

### 独立进程

Dashboard 从 koishi 插件分离为独立进程（`standalone.js`），监听独立端口（默认 5150），不受 koishi 启停影响。`restart.sh` 先启动 Dashboard 再启动 koishi。

### 崩溃保护

- `process.on('uncaughtException')` 和 `process.on('unhandledRejection')` 全局捕获，单请求错误不拖垮整个进程
- `watchdog.sh` 守护脚本，每 10 秒检查端口 5150，崩溃后自动重启
- 所有按钮操作使用 `try/catch/finally` 兜底，确保 `acting` 状态永远解锁
- 所有 API 路由独立错误处理，单个 500 不影响其他请求

## 三十二、session.stripped 兼容修复与代码清理（2026-05-08 ~ 05-09）

### 背景

`@satorijs/core@3.7.0` 的 `Session.prototype` 缺少 `stripped`、`resolve`、`send` 四个方法，导致 Commander 中间件访问 `session.stripped` 时返回 `undefined`，整个中间件链断裂，所有插件收不到消息。

### 排查链路

1. 日志出现 `[W] session ... TypeError: Cannot destructure property 'hasAt' of 'session.stripped' as it is undefined.`
2. 直接原因：`session.stripped` 的 getter 在运行时 session 对象上不存在
3. 补丁打在 `require('@satorijs/core').Session.prototype` 但被 `require('koishi').Session`（undefined）误导
4. 辅助原因是重启脚本使用了全局 `/usr/bin/koishi`（含独立 `@satorijs/core@4.6.0`），而非本地 binary

### 修复内容

| 修复 | 文件 | 说明 |
|------|------|------|
| Session 补丁 | `index.js` 顶部 | 从 `@satorijs/core` 直接引用 Session/Bot，安装 `stripped`/`parsed` 访问器 |
| Bot.prototype.session 包装 | `index.js` | 确保每个新创建的 session 实例都有 stripped 兜底 |
| resolve 补丁 | `index.js` | 补 `Session.prototype.resolve` |
| send 补丁 | `index.js` | 补 `Session.prototype.send`，加 `h.normalize()` 确保 CQ 码被正确解析 |
| safeSendReply | `index.js`（YUN 已有） | 发送失败封控：挫折计数、管理员通知、1 小时熔断 |
| chat().catch 兜底 | `index.js` | 从裸 `session.send(msg)` 改为 `safeSendReply` |
| 删除 patch.js | 根目录 | 补丁已合并到 index.js，不再需要独立文件 + NODE_OPTIONS |
| 恢复 start.js | 根目录 | 去掉 NODE_OPTIONS 拼接，恢复 `require('koishi/bin')` |
| 修复 restart-bot.sh | `scripts/` | 用本地 `node_modules/koishi/bin.js` 替换全局 `/usr/bin/koishi`，去掉 NODE_OPTIONS |
| 修复 restart.sh BOM | 服务器 | 移除 UTF-8 BOM（`ef bb bf`），防止 `#!/bin/bash` 无法识别 |
| 修复 setup.sh 编码 | `setup.sh` | 从二进制损坏的 20966 字节恢复到正常 UTF-8 文本 10835 字节 |
| 删除 standalone.js 密码 debug 日志 | `standalone.js` | `log('login failed: input_len=...')` 改为 `log('login failed')` |

### 关键发现

1. **`require('koishi').Session` 是 `undefined`**，必须用 `require('@satorijs/core').Session`
2. **全局 `/usr/bin/koishi` 有自己的 `@satorijs/core@4.6.0`**，与 app 的 `@satorijs/core@3.7.0` 不兼容，adapter-onebot 的 `http.ws()` 静默失败
3. **PowerShell 管道写入文件会损坏二进制**：`git show ... > file` 或 `| Set-Content` 会输出 UTF-16 导致文件编码损坏。必须用 `cmd /c "git show ... > file"` 或 SCP 传输

- **`YUN`**（备选基线）：模块拆分更彻底，setup.sh 编码已修。已恢复原状 `eb391fe`
- **`main`**（生产分支）：稳定的旧版，未更新
