# 东雪莲 QQ 机器人开发成果总结

## 一、服务器部署与基础架构修复

从零开始在 120.55.246.12（Ubuntu 22.04）部署了完整的 QQ 机器人系统，架构为：

```
NapCat（QQ 登录, WS 服务端 :8080）
  → Koishi v4.18.11（adapter-onebot, WS 客户端）
  → dongxuelian-ai 插件
  → 多 API 后端（OpenCode Go / DeepSeek 官方 / 阿里云）
```

修复了三大基础设施问题：

**1. NapCat Worker 进程崩溃：** 添加 `--disable-gpu` 启动参数解决 GPU 兼容性问题。

**2. adapter-onebot 无法连接 NapCat：** 根因是 koishi.yml 格式错误，将 server 插件格式改为根级 port/selfUrl，协议从 ws-reverse 改为 ws 正向，因为 NapCat 是 WebSocket 服务端，Koishi 须以客户端身份连接。

**3. @satorijs/core 版本冲突：** Koishi 4.x 需要 core ^4.6.0，但 adapter-onebot 依赖 satori 3.x 锁死 core 3.7.0。通过 package.json 添加 overrides 锁定版本，同时创建 patch.js 通过 NODE_OPTIONS 预加载补齐 3.x 缺失的 Session API（resolve、stripped、send）。

---

## 二、识图功能（多模态 + OCR 双轨）

实现了完整的图片识别管线：

从 QQ 消息中提取图片 file 参数 → 调用 NapCat 的 get_image API 获取本地缓存路径 → 读取本地图片文件 → base64 编码 → 发送给多模态模型。

同时集成了 tesseract OCR（chi_sim+eng）作为非视觉模型的回退方案，当模型不支持多模态时自动走 OCR 管线。

中间件中添加了白名单和概率控制：只有白名单群中的图片才会触发识图，非白名单群和无 @ 的图片静默跳过。非多模态模型（如 deepseek-chat）自动跳过识图流程。

---

## 三、转发消息处理

实现了完整的合并转发消息识别管线：

**外层转发：** 检测 session.content 中的 `<forward id="xxx"/>` 或 CQ 码格式 → 调用 callGetForwardMsg 通过 NapCat API 拉取消息节点 → 转换为统一格式 → 通过 summarizeForwardNodes 格式化为带层级的可读文本 → 注入 AI 上下文。

**嵌套转发：** 检测消息段中的 forward 类型或 raw_message 中的 CQ 码 → 递归调用 callGetForwardMsg 尝试拉取 → NapCat 明确返回"内层消息无法获取"时降级显示 `[嵌套转发：内容暂不可见]`。

**格式优化：** 加入层级标记（`[对话]` / `└─ [内层转发]`），接入 MAX_FORWARD_NODES（50 条）和 MAX_FORWARD_DEPTH（4 层）做截断控制，超出部分提示"还有 N 条消息未显示"。

**昵称净化：** 过滤不可见字符（韩文填充符、零宽空格等 U+200B-U+11FF 范围），空昵称回退为"群友"。

---

## 四、引用消息处理

实现了引用消息的内容注入：直接从 session.quote 提取被引用的消息内容，解析文本/图片/表情/@/转发/视频/语音/文件等多种消息段类型，将引用内容以 `[引用内容]xxx[以上是对方说的话，不是在对你说]` 格式注入用户消息，同时去掉重复的系统消息注入，避免 AI 混淆。

---

## 五、模型切换系统（三分层父子菜单）

设计了三层切换体系：`切换模型` → 选择供应商 → 选择模型。支持三个供应商：

- **OpenCode Go**：14 个模型（千问 3.5、千问 3.6、GLM-5、GLM-5.1、Kimi K2.5、Kimi K2.6、DSv4pro、DSv4、MiMo-V2-Pro、MiMo-V2-Omni、MiMo-V2.5-Pro、MiMo-V2.5、MiniMax M2.7、MiniMax M2.5）
- **阿里云 DashScope**：qwen3.5-plus、qwen3.6-plus
- **DeepSeek 官方**：deepseek-chat、deepseek-v4-flash、deepseek-v4-pro

切换时自动切换 API 地址和对应密钥。为 OpenCode Go 的 DeepSeek 模型添加了 reasoning_content 回退处理（当 content 为空时使用 reasoning_content）。

## 六、联网搜索能力

在 getSearchCapability 函数中添加了 qwen 模型识别逻辑，使通过 OpenCode Go 调用的 qwen 模型也能使用 enable_search 参数进行联网搜索，突破了原有代码只识别 DashScope 和 OpenAI 官方 API 的限制。支持 dashscope-chat、openai-chat-search、openai-responses 三种搜索模式。

## 七、消息队列与稳定性

给 enqueueForChannel 添加了 60 秒超时机制，防止单个 API 请求卡死整个消息队列。修复了 config 作用域问题（config is not defined、vConfig is not defined），在视觉处理代码块中单独加载配置。修复了 helpAI 菜单重复发送的问题。

## 八、白名单与概率回复系统

实现了群聊白名单加概率触发的双层过滤机制。白名单群中的消息按设定概率回复，非白名单群全部跳过。每个群可单独设置概率（如 30%、5%），带有 warmup 机制（50 条消息内不递增，之后每条递增 2%）。

修改了 shouldSkipForRandomReply 逻辑，使图片也参与概率判断（修复了图片总是回复的 bug）。同时保留了 @ 提及和私聊的始终回复逻辑。随机回复时添加 `[群聊刷到]` 标记，让 AI 以群友插话口吻回应而非误以为被直接召唤。

## 十、提示词工程与人设优化

**政治红线：** 写入两套人格，包含红线示例（党和国家领导人、领土主权、政治运动、历史事件等）、隐喻/谐音/反讽识别提醒、标准拒绝话术。

**脏话替代词：** 嘴臭人格中加入屎→史、妈→老冯、傻逼→沙鼻、废物→飞舞、智障→智战等谐音替代规则，同时要求减少屎尿屁输出，并提醒模仿例子时切换代替词。友善人格中删除替代规则（因为友善模式不骂人）。

**字数控制：** 取消 trimReply 的硬截断，改为提示词层面的软性引导"评价类回复控制在两三句话以内"。

**评价类回复：** 加入引导"先看完内容再回应，有值得评论的地方直接说"，避免 AI 盲目输出。识图提示词改为"直接评价这张图，不要描述过程"。

**随机回复插话：** 加入"当前消息是你在群聊里看到的，对方不是在叫你"的上下文标记，防止 AI 用回应对方的口吻回复。

**回复字数放宽：** max_tokens 从 500 提升到 1000，支持更长的回复。

**摆烂机制：** 当 completion_tokens 超过 600 时，AI 自动输出"太麻烦不想了，摆烂了"等随机摆烂回复。

## 十三、服务器部署教训集

### 1. `pkill -f` 误杀 SSH
```bash
# 错误 ❌ — 会杀 SSH 守护进程导致服务器失联
pkill -f "start"

# 正确 ✅
pkill -f 'koishi/lib/worker'
```
`pkill -f` 用ERE匹配进程名，`"start"`会匹配到包含`start`的所有进程包括sshd。

### 2. SCP 编码损坏
PowerShell `Get-Content file | ssh ... Set-Content` 会输出 UTF-16 LE带BOM，损坏中文字符和API key。必须直接用 `scp` 二进制传输：
```bash
scp file user@host:/path/    # ✅ 正确
```

### 3. API Key 格式
| 供应商 | Key 前缀 | 文件 |
|---|---|---|
| DeepSeek | `sk-*` | `ai-deepseek-key.txt` |
| mimorium | `tp-*` | `ai-mimorium-key.txt` |
| opencode | `sk-*` | `ai-openai-key.txt` |
| dashscope | `sk-*` | `ai-dashscope-key.txt` |
| GLM | 无固定前缀 | `ai-glm-key.txt` |

服务端和本地的 key 相互不通用，服务端 key 不能复制到本地使用（反之亦然）。

### 4. 进程残留导致双实例
重启时若 `pkill` 没杀干净，旧进程孤儿化后继续监听端口，新进程启动失败。必须 `pkill -f 'koishi/lib/worker'` 后 `sleep 3` 确保释放。

### 5. 重复部署导致双实例
部署时如果第一次 `pkill + start` 已经成功（日志能看到 `loaded`），但没仔细确认就再次 `nohup start`，会跑起两个 bot 实例。两个实例同时处理同一条消息，群友会收到重复回复。

**避免方法**：部署后等 15 秒，确认日志出现 `dongxuelian-ai.*loaded` 且只有一条，不要盲目重试。清理时用 `pkill -f 'koishi/lib/worker'` 精确杀进程，不要用 `pkill -f 'koishi'`（过于宽泛）。

### 6. koishi.yml 格式错误
`server` 插件必须以根级 `port`/`selfUrl` 配置，`adapter-onebot` 必须用 `ws` 协议正向连接 NapCat（NapCat 是 WS 服务端）。

## 十五、部署脚本设计

项目根目录包含三类脚本：

| 脚本 | 用途 | 运行位置 |
|---|---|---|
| `setup.sh` | 首次部署 | 复制到服务器后 `bash setup.sh` |
| `scripts/deploy-package.sh` | 更新重启 | 单个插件: `bash scripts/ai.sh`，全量: 用 SSH 逐包部署 |

**setup.sh 用户流程：**
1. 编辑脚本顶部 QQ_NUMBER 和 ADMIN_QQ
2. `bash setup.sh` → 自动安装 NapCat + Koishi + 全部依赖
3. 看到二维码 → 掏出 QQ 扫码登录
4. 按提示输入 DeepSeek API key（首选必填，最流行的模型）
5. 可选输入 mimorium / opencode / dashscope / GLM key
6. 自动启动 bot

**设计原则：** 代码更新后必须同步更新所有脚本。新增 API 供应商、修改配置文件路径、新增依赖等，都要对应修改 setup.sh 和 scripts/ 下的部署脚本。

## 十七、世界观注入与人格匹配原则

**人格与世界观严格对应。** 每个世界观 lore 必须按照所属人格注入，禁止混入不相关人格：

| 世界观 | 适用人格 | 例外 |
|---|---|---|
| 鸣潮（wuwa-lore） | 默认人格、长离、椿 | 特蕾西娅不注入，避免 OOC |
| 泰拉（terra-lore） | 仅特蕾西娅 | 鸣潮角色不注入 |

## 十八、代码拆分教训

### repeat.js 拆分失败教训

**目标**：将 index.js 里的复读检测 8 个函数 + 3 个状态 Map 搬到新建 repeat.js。

**失败原因**：一次性改了 4 个文件（index.js、utils.js、chat.js、cascade-test.js），删旧代码和加新代码同时进行，测试报错时分不清是哪个操作引入的。

**具体失误**：

| 操作 | 后果 |
|---|---|
| 删 index.js 函数定义 + 加 import 同时做 | 漏了一个未闭合的大括号，语法检查通过前已污染了文件 |
| cascade-test.js 改了 section 16.5 的源码匹配 | 改到一半的测试引用了不存在的 `index.buildRepeatCandidate` |
| `getSegmentData` 加到了 utils.js 但没及时删 index.js 里的旧定义 | 函数去重检查报错 |

**正确步骤（后续拆文件必须遵守）**：

```
第 1 步：只创建目标文件，完整复制函数 + 状态，跑 node -c 确认语法
第 2 步：源文件加 import，跑 node -c 确认语法
第 3 步：注释掉源文件里的旧定义（不删），跑测试验证 import 生效
第 4 步：测试通过后删掉注释掉的旧代码，再跑一次测试
第 5 步：更新 cascade-test.js 的引用路径（如果需要），再跑测试
```

每步之间 `node -c` + `npm run test:quick` 验证，不过不进入下一步。后三步中任何一步失败，退回上一步重来。

## 十九、chat.js 回复防线拆分

### 本轮目标

只拆 `reply-guard.js`，不拆 `prompt-builder.js`。原因是回复安全、防泄漏、重复回复 retry 的边界清楚，收益高、风险低；prompt 组合牵涉人格、lore、history、memory、quoted、forward、vision 注入顺序，暂时不碰。

### reply-guard.js 边界

- 只做判断和 retry/fallback prompt 生成。
- 不调用模型，不修改 `messages`。
- 不保存 conversation，不 import `chat.js` 或 `index.js`。
- `chat.js` 继续拥有 `callOpenAI()`、retry 循环、日志和最终保存顺序。

### 测试结果

- `node -c packages/koishi-plugin-dongxuelian-ai/lib/reply-guard.js` 通过。
- `node -c packages/koishi-plugin-dongxuelian-ai/lib/chat.js` 通过。
- `npm run check` 通过。
- `npm run test:quick` 通过。
- `npm run test:scenario` 通过。
- `npm test` 通过。
- `git diff --check` 通过。

### 后续规则

- 改 thinking leak、禁词、重复回复、兜底选择时，优先维护 `reply-guard.js`。
- 改 prompt、人格、lore、history 注入时，再评估是否拆 `prompt-builder.js`。
- 不为了减少行数硬拆；只在边界自然且有测试护栏时拆。
