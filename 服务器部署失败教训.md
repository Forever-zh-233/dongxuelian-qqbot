# 服务器部署失败教训记录

## 日期
2026-05-01

## 一、直接原因：`pkill -f "start|koishi"` 误杀系统进程

当时使用的命令：
```bash
ssh root@120.55.246.12 'pkill -f "start|koishi" 2>/dev/null; sleep 2; cd /root/koishi-app && nohup npm exec koishi start >> koishi.log 2>&1 &'
```

**教训**：`|` 在 pkill 的 ERE 正则中会被解析为**或**操作符，匹配所有含 "start" **或** "koishi" 的进程。
"start" 关键词匹配了大量系统进程（sshd 子进程、systemd 相关进程等），导致 SSH 服务被意外杀死。

**正确做法**：只匹配 koishi 精确的关键词：
```bash
pkill -f "koishi/lib/worker"
# 或
kill $(pgrep -f "node.*koishi")
```

## 二、根因：`@koishijs/core` 与 `@satorijs/core` 版本错位

`package.json` 中有 `"overrides": { "@satorijs/core": "3.7.0" }` 强制锁定旧版。

`@satorijs/core@3.7.0` 的 `Session.prototype` 缺少 `stripped`、`resolve`、`send` 三个方法。
项目中本来有 `patch.js` 和 `start.js` 在启动时打补丁，但 `npm exec koishi start` 不会加载这些补丁。

**教训**：项目依赖了外部补丁文件（patch.js）来修复运行时兼容性问题，但补丁文件在 `npm exec koishi start` 启动方式下不生效。
修正方式是在插件本身的 index.js 里加载补丁，不依赖外部启动脚本。

## 三、补丁链式问题

过程中我曾手动修改 `@koishijs/core` 的 node_modules 文件来修复 `session.stripped` 报错，
但由于补丁方式过于粗暴（`try {} catch {}` 整段包裹），导致 Commander 的 `before("attach")` 初始化逻辑不完整，
进而导致 `session.send` 不可用。

**教训**：永远不要直接修改 node_modules 文件。应该在项目层面通过插件加载补丁。

## 四、变量作用域 Bug：`const currentUserId` 的 TDZ（暂时性死区）

`const currentUserId = session.userId || ...` 声明在 line 3530，但人格指令处理器在 line 3249 就开始使用。
由于 `const` 的暂时性死区，声明前访问会抛出 `ReferenceError`，导致 handler 静默崩溃。

**教训**：重构时要检查变量定义位置是否在首次使用之前。

## 六、正确的部署流程

```bash
# 1. 确保 NapCat 在运行且有有效的 QQ 登录会话
# 2. 上传文件
scp index.js root@server:/root/koishi-app/packages/...
# 3. 仅杀 koishi 相关进程
pkill -f "koishi/lib/worker"
# 4. 启动
ssh root@server "cd /root/koishi-app && npm exec koishi start"
```
