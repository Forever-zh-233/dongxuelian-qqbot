东雪莲 Bot 测试文件维护指南
==============================

一、测试入口与职责
--------------------

| 入口 | 命令 | 职责 | 适用场景 |
|---|---|---|---|
| 语法门卫 | npm run check | 所有生产 JS 文件语法检查 | 新增/修改 .js 文件后必须跑 |
| 结构契约 | npm run test:quick | 导出、常量、脚本、静态防线、覆盖映射 | 改接口/导出/常量时跑 |
| 行为链路 | npm run test:scenario | fake Koishi 集成测试 | 改命令/回复/中间件行为时优先跑 |
| 辅助插件 | npm run test:plugins | 辅助插件独立测试 | 改 group-name-at / local-video-sender 时跑 |
| 信任入口 | npm test | test:quick + test:scenario | 提价前跑这个就够了 |


先看这个：什么时候要改测试
--------------------------

不要每次改代码都把测试文件翻一遍。先按这个表判断，能不改就不改。

| 你这次改了什么 | 要不要改测试 | 通常要动哪里 | 最少跑什么 |
|---|---|---|---|
| 只改注释、README、总结文档 | 不用 | 不动测试 | 可不跑；不放心跑 `npm test` |
| 只改普通实现细节，用户行为不变，导出不变 | 通常不用 | 不动测试 | `npm run check`，再跑相关 scenario 或 `npm test` |
| 改了用户能看到的命令行为、权限、回复是否发送 | 要 | 对应 scenario，优先不改 cascade | `npm run test:scenario` |
| 只改回复文案，但语义不变 | 通常不用；除非测试锁了这句文案 | 优先放宽断言为 non-empty / includes 关键语义 / no-leak | 相关 scenario |
| 新增 `lib/*.js` 生产模块 | 必须 | `package.json` check + cascade 的模块/导出/语法/重复扫描 | `npm run check && npm run test:quick` |
| 新增 `lib/*/*.js` 子目录模块（如 rulesets） | 必须 | `package.json` check + cascade 路径用 `path.join(LIB, '子目录', '文件')`；非函数导出单独写类型断言 | `npm run check && npm run test:quick` |
| 只是从旧文件搬函数到新文件，行为不变 | 必须维护结构测试；行为测试不一定新增 | cascade 路径/导出；已有 scenario 保持通过 | `npm run check && npm run test:quick && npm run test:scenario` |
| 新增一个真实消息链路功能 | 要 | 新增或扩展 scenario；只在替代源码扫描时改 `COVERAGE_MAP` | `npm run test:scenario` |
| 改常量、provider、fallback、导出名 | 要 | cascade + 对应 fallback/chat scenario | `npm run test:quick && npm run test:scenario` |
| 改越狱/敏感/安全规则正则 | 要 | 优先改 ruleset 分组 + cascade 正负样例；不要再塞巨型正则 | `npm run test:quick`，影响聊天链路时加 `npm run test:scenario` |
| 改 `setup.sh`、部署脚本、插件列表 | 要 | cascade 部署脚本区；必要时 setup scenario | `npm run test:quick`，有 bash 再跑 scenario |
| 改辅助插件 `group-name-at` / `local-video-sender` | 要 | 对应 `plugin-test.js` | `npm run test:plugins` |
| 只是调试临时日志，之后会删 | 不建议改测试 | 不动测试 | 删除日志后跑 `npm run check` |

一句话判断：

```
用户行为变了 -> 改 scenario
模块边界变了 -> 改 cascade
只是内部写法变了 -> 通常不改测试
新增生产文件 -> check + cascade 必改
辅助插件变了 -> test:plugins
```


各类行为应该改哪个 scenario
---------------------------

| 行为范围 | 优先改这里 | 不要优先改这里 |
|---|---|---|
| AI 状态、重载、开关、管理员权限 | `command.test.js` | cascade 源码字符串扫描 |
| 聊天回复、thinking leak、conversation 保存 | `chat.test.js` | `_testOnly` / 直接调用私有函数 |
| API fallback、HTTP 错误、无效 JSON、key 泄漏 | `fallback.test.js` | 锁死 fallback 文案 |
| 复读、冷却、窗口、face 复读 | `repeat.test.js`；边界优先纯函数 | 全局 fake timers |
| sticker 文本/图片顺序、fallback、冷却 | `sticker.test.js` | 扫源码里的发送顺序 |
| 敏感检测开关、通知、关闭后不触发 | `sensitive.test.js` / `concurrency.test.js` | 手查内部 Map |
| 主动随机回复 | `random.test.js` | 改 `Math.random` 全局 |
| 合并转发 | `forward.test.js` | 真实 NapCat |
| 越狱/安全输入规则 | `lib/rulesets/*.js` + cascade 正负样例 | 单条巨型正则、只测源码字符串 |
| 识图 session 标记、引用图片 | `vision.test.js` | 直接碰 `_vision*` 字段 |
| 人格 prompt 组合和 lore 注入 | `persona-prompt.test.js` | 全文大快照 |
| 文件写入并发、安全落盘 | `persistence.test.js` | 人工检查临时文件 |
| setup.sh 模拟安装 | `setup.test.js` | 在 Windows 假装 bash 一定存在 |


什么时候不要新增测试
--------------------

- 已有 scenario 已经覆盖同一行为，只是换了实现方式。
- 只是把断言从一个文件搬到另一个文件，行为没有增加。
- 只是修错别字、注释、日志格式，且不会影响用户可见行为。
- 为了让某个源码字符串扫描“看起来更安全”而新增扫描；优先问能不能用 scenario 覆盖真实行为。
- 为了追求 100% 文案稳定而锁死整句回复。除非这句就是协议或安全兜底，否则只断言关键语义、非空、无泄漏。


二、变更类型与必改项
--------------------

| 变更类型 | 必改 | 位置 |
|---|---|---|
| 新增生产 JS 模块（如 forward.js） | `package.json` scripts.check 加一行 `node -c` | `package.json` |
| | `cascade-test.js` 的 `modPaths` 加模块路径 | `cascade-test.js` 模块加载区 |
| | `cascade-test.js` 的 `expectedExports` 加导出检查 | `cascade-test.js` 导出列表 |
| | `cascade-test.js` 的 `syntaxFiles` 数组 | `cascade-test.js` 语法扫描区 |
| | `cascade-test.js` 的 `duplicateScanFiles` 数组 | `cascade-test.js` 去重扫描区 |
| 新增行为测试文件（scenario） | `scenarios/index.js` 加 `require` + `SCENARIOS` | `test/scenarios/index.js` |
| | `cascade-test.js` 顶部的 `COVERAGE_MAP`（可选，仅当替代源码扫描时） | `cascade-test.js` 顶部 |
| 改命令/回复/中间件行为 | 优先改或新增 scenario，不要加源码字符串扫描 | `test/scenarios/` |
| 改结构契约/导出/脚本/部署防线 | 改 cascade 对应区块 | `cascade-test.js` |
| 改辅助插件 | 跑并维护 `test:plugins`，暂不并入 `npm test` | `plugin-test.js` |

固定执行顺序：
```
package.json scripts.check → cascade modPaths → expectedExports → syntaxFiles → duplicateScanFiles
```
这样可以避免 quick 能 require 但 check 没管语法的漏网模块。


三、核心原则
----------

1. 行为变更优先加 scenario，不要优先加源码字符串扫描。cascade 的源码扫描只防结构契约破坏。

2. cascade 的 `COVERAGE_MAP` 要当索引维护。行为被 scenario 覆盖后，源码扫描才可以删或放松。如果只是换了文件名，也要同步 `COVERAGE_MAP` 的文件和 needle。

3. 每次大拆模块后，至少跑 `npm run check && npm run test:quick`。只要碰到真实消息链路，再加跑 `npm run test:scenario`。

4. 提交前跑 `npm test`（test:quick + test:scenario）就够了。改辅助插件时额外跑 `npm run test:plugins`。


四、代码拆分步骤（防炸清单）
------------------------------

从 index.js 拆出新文件（如 repeat.js）时必须遵守以下步骤，不可跳过：

```
第 1 步：只创建目标文件，完整复制函数 + 状态，跑 node -c 确认语法
第 2 步：源文件加 import，跑 node -c 确认语法
第 3 步：注释掉源文件里的旧定义（不删），跑 npm run test:quick 验证 import 生效
第 4 步：测试通过后删掉注释掉的旧代码，再跑一次 npm run test:quick
第 5 步：更新 cascade-test.js 的引用路径（modPaths/exports/syntaxFiles/duplicateScanFiles 等），再跑测试
```

每步之间 `node -c` + `npm run test:quick` 验证，不过不进入下一步。后三步中任何一步失败，退回上一步重来。

六、新增模块 checklist
----------------------

新增 `lib/xxx.js` 时：

- [ ] `package.json` scripts.check 加 `node -c packages/koishi-plugin-dongxuelian-ai/lib/xxx.js`
- [ ] `cascade-test.js` `modPaths` 加 `xxx: path.join(LIB, 'xxx')`
- [ ] `cascade-test.js` `expectedExports` 加导出函数列表
- [ ] `cascade-test.js` `syntaxFiles` 加 `'lib/xxx.js'`
- [ ] `cascade-test.js` `duplicateScanFiles` 加 `'xxx.js'`
- [ ] 跑 `npm run check`
- [ ] 跑 `npm run test:quick`

新增 `lib/子目录/xxx.js` 时：

- [ ] `package.json` scripts.check 加完整 `node -c packages/.../lib/子目录/xxx.js`
- [ ] `cascade-test.js` `modPaths` 用 `path.join(LIB, '子目录', 'xxx')`
- [ ] 函数导出放 `expectedExports`；数组、对象、RegExp 这类非函数导出在模块加载后单独 `check(...)`
- [ ] `cascade-test.js` `syntaxFiles` 加完整路径
- [ ] `cascade-test.js` `duplicateScanFiles` 加 `'子目录/xxx.js'`
- [ ] 跑 `npm run check && npm run test:quick`

改安全 ruleset 时：

- [ ] 不直接恢复成一条巨型正则；按规则类别加 pattern
- [ ] cascade 至少加一个“应拦截”样例和一个“应放行”样例，防漏拦和误伤
- [ ] 保持旧导出名（例如 `JAILBREAK_INPUT_RE`）不变，除非同步改所有调用方

新增 `test/scenarios/xxx.test.js` 时：

- [ ] `scenarios/index.js` 加 `const xxx = require('./xxx')` + `SCENARIOS 加 'xxx'`
- [ ] 如果替代了某条源码扫描防线 → 更新 `COVERAGE_MAP`
- [ ] 跑 `npm run test:scenario`


七、常见陷阱
----------

1. 新增模块后只改了 cascade 没改 `package.json` check 脚本 → 语法检查漏网
2. 删了 index.js 里的旧定义但没更新 cascade 的引用 → `buildRepeatCandidate is not a function`
3. 同时在两个文件里定义了同名函数 → cascade 去重检查会报 FAIL
4. 拆文件时一次性改了 4 个文件然后跑测试 → 出问题时不知道哪个操作引入的。正确做法是先加后删、每步验证

八、时间依赖测试规则
--------------------

曾经默认安装 fake timers，是为了稳定测试复读冷却、sticker 冷却、敏感检测缓存过期这类依赖 `Date.now()` / `setTimeout()` 的路径。这个做法能避免真实等待 30-120 秒，但全局 patch `Date.now`、`setTimeout`、`setInterval` 会扩大污染面，后续维护时容易误判测试为什么通过。

现在的规则：

- 默认 scenario 使用真实时间，不默认安装 fake timers。
- 新增时间依赖代码时，优先用参数注入 `now`、局部 `time.now()`，或用 `expiresAt` 惰性过期，不要要求测试 patch 全局时间。
- 只有确实要验证真实 `setTimeout` / `setInterval` 调度行为时，才在场景里显式 `fakeTimers: true`，并在测试名或注释里说明原因。
- `global.fetch`、`process.env`、`require.cache` 属于测试隔离手段，可以继续使用，但必须用 `try/finally` 或 teardown 恢复。
- 如果某个场景需要推进时间，优先检查能不能下沉到纯函数层，例如 `checkGroupRepeat(..., now)`，不要先动 middleware 全局时钟。
