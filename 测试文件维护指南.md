东雪莲 Bot 测试文件维护指南
==============================

一、测试入口与职责
--------------------

| 入口 | 命令 | 职责 | 适用场景 |
|---|---|---|---|
| 语法门卫 | npm run check | 所有生产 JS 文件语法检查 | 新增/修改 .js 文件后必须跑 |
| 结构契约 | npm run test:quick | 导出、常量、脚本、静态防线、覆盖映射 | 改接口/导出/常量时跑 |
| 行为链路 | npm run test:scenario | fake Koishi 集成测试 | 改命令/回复/中间件行为时优先跑 |
| 辅助插件 | npm run test:plugins | 辅助插件独立测试 | 改 group-name-at / local-video-sender 时跑 |
| 信任入口 | npm test | test:quick + test:scenario | 提价前跑这个就够了 |


二、变更类型与必改项
--------------------

| 变更类型 | 必改 | 位置 |
|---|---|---|
| 新增生产 JS 模块（如 forward.js） | `package.json` scripts.check 加一行 `node -c` | `package.json` |
| | `cascade-test.js` 的 `modPaths` 加模块路径 | `cascade-test.js` 模块加载区 |
| | `cascade-test.js` 的 `expectedExports` 加导出检查 | `cascade-test.js` 导出列表 |
| | `cascade-test.js` 的 `syntaxFiles` 数组 | `cascade-test.js` 语法扫描区 |
| | `cascade-test.js` 的 `duplicateScanFiles` 数组 | `cascade-test.js` 去重扫描区 |
| 新增行为测试文件（scenario） | `scenarios/index.js` 加 `require` + `SCENARIOS` | `test/scenarios/index.js` |
| | `cascade-test.js` 顶部的 `COVERAGE_MAP`（可选，仅当替代源码扫描时） | `cascade-test.js` 顶部 |
| 改命令/回复/中间件行为 | 优先改或新增 scenario，不要加源码字符串扫描 | `test/scenarios/` |
| 改结构契约/导出/脚本/部署防线 | 改 cascade 对应区块 | `cascade-test.js` |
| 改辅助插件 | 跑并维护 `test:plugins`，暂不并入 `npm test` | `plugin-test.js` |

固定执行顺序：
```
package.json scripts.check → cascade modPaths → expectedExports → syntaxFiles → duplicateScanFiles
```
这样可以避免 quick 能 require 但 check 没管语法的漏网模块。


三、核心原则
----------

1. 行为变更优先加 scenario，不要优先加源码字符串扫描。cascade 的源码扫描只防结构契约破坏。

2. cascade 的 `COVERAGE_MAP` 要当索引维护。行为被 scenario 覆盖后，源码扫描才可以删或放松。如果只是换了文件名，也要同步 `COVERAGE_MAP` 的文件和 needle。

3. 每次大拆模块后，至少跑 `npm run check && npm run test:quick`。只要碰到真实消息链路，再加跑 `npm run test:scenario`。

4. 提交前跑 `npm test`（test:quick + test:scenario）就够了。改辅助插件时额外跑 `npm run test:plugins`。


四、代码拆分步骤（防炸清单）
------------------------------

从 index.js 拆出新文件（如 repeat.js）时必须遵守以下步骤，不可跳过：

```
第 1 步：只创建目标文件，完整复制函数 + 状态，跑 node -c 确认语法
第 2 步：源文件加 import，跑 node -c 确认语法
第 3 步：注释掉源文件里的旧定义（不删），跑 npm run test:quick 验证 import 生效
第 4 步：测试通过后删掉注释掉的旧代码，再跑一次 npm run test:quick
第 5 步：更新 cascade-test.js 的引用路径（modPaths/exports/syntaxFiles/duplicateScanFiles 等），再跑测试
```

每步之间 `node -c` + `npm run test:quick` 验证，不过不进入下一步。后三步中任何一步失败，退回上一步重来。

六、新增模块 checklist
----------------------

新增 `lib/xxx.js` 时：

- [ ] `package.json` scripts.check 加 `node -c packages/koishi-plugin-dongxuelian-ai/lib/xxx.js`
- [ ] `cascade-test.js` `modPaths` 加 `xxx: path.join(LIB, 'xxx')`
- [ ] `cascade-test.js` `expectedExports` 加导出函数列表
- [ ] `cascade-test.js` `syntaxFiles` 加 `'lib/xxx.js'`
- [ ] `cascade-test.js` `duplicateScanFiles` 加 `'xxx.js'`
- [ ] 跑 `npm run check`
- [ ] 跑 `npm run test:quick`

新增 `test/scenarios/xxx.test.js` 时：

- [ ] `scenarios/index.js` 加 `const xxx = require('./xxx')` + `SCENARIOS 加 'xxx'`
- [ ] 如果替代了某条源码扫描防线 → 更新 `COVERAGE_MAP`
- [ ] 跑 `npm run test:scenario`


七、常见陷阱
----------

1. 新增模块后只改了 cascade 没改 `package.json` check 脚本 → 语法检查漏网
2. 删了 index.js 里的旧定义但没更新 cascade 的引用 → `buildRepeatCandidate is not a function`
3. 同时在两个文件里定义了同名函数 → cascade 去重检查会报 FAIL
4. 拆文件时一次性改了 4 个文件然后跑测试 → 出问题时不知道哪个操作引入的。正确做法是先加后删、每步验证
