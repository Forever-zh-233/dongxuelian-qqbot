# 莲莲Bot Agent — 第五期开发计划

> 总目标：在第三期已完成的 Agent 队列、计划模式、主动推送、cron、memory、浏览器补齐基础上，把莲莲Bot Agent 从"工具调用型助手"推进到"自我进化、可扩展的智能体系统"。
>
> 核心方向：**Auto-Memory + Dream 自我进化** + **Skill 市场**（远程下载）+ **安全扫描器**。

### 0.1 前期已具备的基础

- Agent ReAct 引擎 + 队列 + 计划模式 + 主动推送 + cron + memory。
- Skill 体系：本地 `data/ai-skills/` 三层目录（core/modes/personas），`read_agent_skill` 工具按需加载。

## 3. 任务 —：多轮搜索上下文摘要

### 3.1 问题

Agent 连续搜索多轮时，前几轮的完整对话（含 tool 结果、搜索结果）会不断累积到 `sanitizeAgentHistory` 中。第 4 轮起，前 3 轮的完整对话历史加上工具调用详情占用大量无用 token。

典型场景：模型调用 web_search → 返回结果 → 模型换 query 再调 web_search → 又返回结果 → 模型终于回复。这个过程产生的中间 tool 结果和历史消息会被下一轮完整保留。

### 3.2 方案

从第 4 轮开始，对前 3 轮的对话用一句话摘要替换：

```diff
// engine.js continueAgent 循环中
+ if (round >= 3) {
+   // 将 messages 中的工具调用记录和旧助理回复压缩成摘要
+   // 保留最近 1 轮完整上下文，更早的用摘要替换
+ }
```

具体规则：
- **保留最近 1 轮**完整对话（最新 assistant + user + tool 结果）
- **第 1-3 轮**直接删除，改为插入一条 system 摘要："前几轮已搜索过 X、Y、Z，结果摘要：..."
- **只影响 `continueAgent` 内部的 messages 数组**，不影响已发出的回复
- 摘要由模型自动生成（用 `compactWithLLM`）

### 3.3 改动

| 文件 | 改动 | 行数 |
|------|------|:---:|
| `engine.js` | `continueAgent` 循环中 round >= 3 时触发摘要 | +10 |
| `engine.js` | 引用 `compactWithLLM`（已有） | 0 |

### 3.4 效果

| 场景 | 当前 | 优化后 |
|------|:---:|:---:|
| 第 1-3 轮 | N/A | N/A |
| 第 4 轮 | ~2000 char 历史 | ~200 char 摘要 |
| 第 5 轮 | ~3000 char 历史 | ~300 char 摘要 |

省约 1800 char/轮（第 4 轮起），约 **450 tok/轮**。

### 3.5 限制

- 不影响单轮搜索（无历史 → 无摘要）
- 摘要由 LLM 生成（`compactWithLLM`），用轻量模型，失败时回退保留原文

## 4. 任务 B：Auto-Memory 自动提取

### 4.1 QwenPaw 参考

QwenPaw 的 Auto-Memory：
- 每 N 条用户消息触发一次
- 只处理最近窗口（不重复处理历史）
- 后台异步执行，不阻塞回复
- 提取结果写入 `memory/YYYY-MM-DD.md`
- 去重由 LLM 提示控制（"避免重复已记录的信息"），非哈希/嵌入

### 4.2 莲莲Bot 实现方案

#### 4.2.1 渠道隔离

**只对 `channel === 'dashboard'` 生效。**
QQ 渠道即使触发 Auto-Memory 也不提取。
engine.js 中 `onAgentReplyComplete` 检查 `channel !== 'dashboard'` 时直接返回。

#### 4.2.2 存储结构

```
data/
├── agent-memory/                         ← QQ 渠道 memory（现有，不动）
└── agent-memory-dashboard/               ← 前端 Auto-Memory（新建）
    ├── {userId}.md                        ← 精选长期记忆（Dream 维护）
    └── daily/
        └── {userId}.YYYY-MM-DD.md         ← 每日原始记录（Auto-Memory 写入）
```

两条路径完全独立，`searchMemory`/`writeMemory` 也不会跨界。

长期文件 `{userId}.md` 格式：自然文本，无结构化限制。
每日文件 `{userId}.YYYY-MM-DD.md` 格式：当日提取的原始记忆，按时间顺序。

#### 4.2.3 触发条件

```javascript
const AUTO_MEMORY_INTERVAL = 8  // 每 8 条用户消息提取一次
const AUTO_MEMORY_WINDOW = 8    // 只看最近 8 条
```

触发时机：Agent 回复完成后，检查该用户在 dashboard 渠道的消息计数。

#### 4.2.4 提取流程

```text
agent.run() 返回后
  → check channel === 'dashboard'？ 不是 → 跳过
  → 检查该用户消息计数 % 8 === 0？ 不是 → 跳过
  → 异步（不阻塞）：
      取最近 8 轮对话
      → 调主模型，max_tokens=500
      → 模型输出自然文本记忆条目
      → 写入 daily/{userId}.YYYY-MM-DD.md（追加）
      → 如果 daily/ 目录下该用户所有文件总大小 > 20KB → 触发 Dream
```

#### 4.2.5 提取 prompt

```
从以下对话中提取值得长期记住的信息。只提取：
- 用户明确表达的偏好、身份、习惯
- 用户分享的重要事实（生日、职业、常用工具等）
- 对话中确认的决策或约定
- 工作流偏好（用户喜欢的操作方式、常用工具）

不要提取：临时话题、闲聊、已知常识、工具调用细节。
如果 daily 文件已存在，先读取已有内容，避免重复记录。
如果没有值得记住的，输出空行。
```

#### 4.2.6 模型和成本

- 使用 **主模型** 做提取（轻量模型 key 可能过期，不做额外依赖）
- 前端用户每天 10-20 条对话 → 触发 1-2 次
- 输入：最近 8 轮，截断 500 字/轮
- 输出：max_tokens=500
- 单个用户日成本可忽略

#### 4.2.7 新增模块

`lib/agent/auto-memory.js`

```javascript
/**
 * MODULE: Agent 自动记忆提取（仅 dashboard 渠道）。
 * 职责: 对话后异步提取关键信息写入 daily 记忆文件。
 * 边界: 不阻塞回复、不修改对话历史、不影响当前回复。
 * 状态: userMessageCounters (Map)。
 */
```

核心 API：
```javascript
onAgentReplyComplete({ userId, messages })
resetAutoMemoryCounter(userId)
getAutoMemoryStats()
```

### 4.3 验证标准

- 用户连续对话 8 轮后，daily/ 文件中出现提取的信息。
- 提取不阻塞回复（异步）。
- 重复信息不重复写入（LLM 提示控制）。
- 主模型调用失败时静默跳过。

## 5. 任务 C：Dream 记忆整理

### 5.1 QwenPaw 参考

QwenPaw 的 Dream：
- 创建 MEMORY.md 备份
- 启动 DreamOptimizer Agent（带 read_file/write_file/edit_file）
- 四原则：极简主义、状态覆盖、归纳合并、过期淘汰
- 输出整理后的 MEMORY.md
- 整理完成后删除已合并的 daily/ 文件

### 5.2 莲莲Bot 实现方案

#### 5.2.1 简化版 Dream（不启动子 Agent）

QwenPaw 用完整 ReAct Agent 做 Dream。莲莲Bot 方案：单次 LLM 调用整理，不启动子 Agent。

触发时机：**每次 Auto-Memory 后检查文件大小**，当 `daily/` 目录下该用户所有文件总大小 > 20KB 时触发。不比等 cron——前端用户每天对话少，cron 可能几天都触发不了。

```text
Auto-Memory 写入完成
  ├─ 常规路径 → 继续（日常对话）
  │
  └─ 检查 daily/ 该用户文件总大小 > 20KB？
      ↓ 是
    备份 {userId}.md → {userId}.md.bak
      ↓
    读取所有 daily/{userId}.*.md + 当前 {userId}.md
      ↓
    调主模型，四原则整理：
      1. 极简主义：只保留核心偏好、确认的事实、高价值经验
      2. 状态覆盖：新信息覆盖旧信息
      3. 归纳合并：相似条目合并为一条通用描述
      4. 过期淘汰：超过 30 天且从未被搜索命中的条目可删除
      ↓
    模型输出精简后的文本内容（自然语言，非结构化）
      ↓
    覆盖写入 {userId}.md
      ↓
    删除所有 daily/{userId}.*.md（已合并到长期文件）
```

#### 5.2.2 整理 prompt

```
你是记忆整理助手。以下是用%d用户的每日记忆文件和长期记忆文件内容。

请按以下原则整理：

1. 极简主义：只保留核心偏好、确认的事实、高价值经验
2. 状态覆盖：新信息覆盖旧信息（如"喜欢A"后来变成"喜欢B"，只保留B）
3. 归纳合并：相似条目合并为一条通用描述
4. 过期淘汰：超过 30 天且从未被搜索命中的条目可删除

输出精简后的内容，使用自然文本格式，每行一条或一个段落。
不要使用 JSON 格式，直接写给人看的记忆描述。
```

#### 5.2.3 删除机制

Dream 本身就是删除机制。不需要单独的删除功能：

| 删除对象 | 时机 | 方法 |
|----------|------|------|
| daily/ 文件 | Dream 整理完成后 | 全部删除（内容已合并到长期文件） |
| 长期记忆中的重复条目 | Dream 整理时 | LLM 合并多条目为一条 |
| 过时信息（30 天未被搜索命中） | Dream 整理时 | LLM 提示要求丢弃 |
| 长期文件超过 20KB | Dream 触发条件 | 触发一次整理，自然压缩 |

如果用户想手动删除某条记忆：Agent 有 `read_file` 和 `edit_file` 工具，用户可以要求 Agent 读取 `.md` 文件后编辑/删除特定条目，不需要额外写代码。

#### 5.2.4 Token 成本

- 只在文件 > 20KB 时触发（文件小不需要整理）
- 使用主模型
- 输入：daily 文件 + 长期文件，每条限 200 字
- 输出：max_tokens=500
- 前端用户每天 1-2 条提取，几十天后才会触发一次 Dream

#### 5.2.5 新增模块

`lib/agent/dream.js`

```javascript
/**
 * MODULE: Agent 记忆整理（Dream）。
 * 职责: 合并/daily 文件到长期记忆，去重/压缩/淘汰过期条目。
 * 边界: 不影响实时对话、不删除未备份的长期文件、失败时保留原文件。
 * 状态: 无（由 auto-memory 完成后触发）。
 */
```

核心 API：
```javascript
runDream({ userId })
runAutoDreamIfNeeded()        // auto-memory 完成后调用的检查函数
getDreamStatus()              // 返回上次整理时间和文件大小
```

### 5.3 验证标准

- daily/ 文件总大小 > 20KB 后自动触发 Dream。
- 整理前 {userId}.md 被备份为 {userId}.md.bak。
- 整理后 daily/ 文件被删除，长期文件体积缩小。
- 整理失败时保留原 {userId}.md + daily 文件不变。
- 不删除 30 天内被搜索命中的条目。

## 6. 任务 D：Skill 市场 3 层架构

### 6.1 QwenPaw 参考

QwenPaw 的 3 层架构：
- **Builtin**：随代码发布的内置 Skill（docx/pptx/xlsx/pdf 等）
- **Pool**：本地共享仓库 `skill_pool/`，含 manifest `skill.json`
- **Workspace**：每个 Agent 实例的活跃 Skill，从 Pool 安装

流程：Builtin → Pool（sync）→ Workspace（download）→ 启用/禁用

### 6.2 莲莲Bot 实现方案

#### 6.2.1 三层映射

| QwenPaw 层 | 莲莲Bot 对应 | 目录 |
|---|---|---|
| Builtin | 内置实用 Skill（随代码发布） | `data/ai-skills/docs/` |
| Pool | 本地共享仓库（所有已下载 Skill 源文件） | `data/skill-pool/` |
| Workspace | 当前工作区安装记录 | `data/ai-skills/workspace/manifest.json` |

**各层职责**：
- **Builtin**：随代码发布的原始文件。`syncBuiltinSkills()` 只同步 `docs/` 下的实用 Skill，不碰 `core/`、`modes/`、`personas/`、`lore/`（这些是人格/模式，不属于 Skill 市场）
- **Pool**：所有已下载到本地的 Skill 源文件，含同步的内置 Skill + 从 GitHub 下载的 Skill。入 Pool 前必须通过安全扫描
- **Workspace**：记录哪些 Pool 中的 Skill 已安装到当前工作区。不存启用状态

**启用态继续由 `ai-tool-config.json.enabledSkills` 管理**，不和 workspace manifest 混用：

| 数据源 | 管什么 | 示例 |
|--------|--------|------|
| `data/ai-skills/workspace/manifest.json` | 安装了哪些 Skill | `web-search-strategy` 已安装，来源 builtin |
| `ai-tool-config.json.enabledSkills` | 其中哪些是启用的 | `web-search-strategy` 已启用 |

操作流程：Pool → `installToWorkspace()` → 写 workspace manifest → `enableSkill()` → 写 ai-tool-config.json。两套配置不冲突——workspace 管"安装/未安装"，ai-tool-config 管"启用/禁用"。

**技能发现过滤链**：

```text
data/skill-pool/（所有已下载的源文件）   ← 新增 SKILL_DIRS 扫描目录
  → skills.listAgentSkills() 扫描全部
    → 按 workspace/manifest.json 过滤 → 只保留已安装到工作区的
      → buildAgentSkillSummary(enabledNames) → 只保留已启用的
        → Agent 上下文（仅 name + description 索引，不注全文）
```

下载的 Skill 和内置 Skill 走完全相同的索引机制——**不强制注入全文**。无论内置还是下载的 Skill，索引中只出现一行 name + description，需要完整内容时 Agent 自己调 `read_agent_skill(name)` 按需读取。

#### 6.2.2 Skill 元数据格式

每个 Skill 是一个目录，包含 `SKILL.*.md` 文件：

```yaml
---
name: skill-name
description: 一句话描述
version: 1.0.0
author: 作者
source: builtin | pool | hub
tags: [search, document, automation]
channels: [qq, dashboard]  # 适用渠道，all 表示全部
requires:
  tools: [web_search, read_file]  # 依赖的工具
  bins: []  # 依赖的系统命令
---
[Skill 正文：LLM 指令]
```

#### 6.2.3 Manifest 格式

`data/skill-pool/manifest.json`：

```json
{
  "schema": "skill-pool-manifest.v1",
  "skills": {
    "web-search-strategy": {
      "name": "web-search-strategy",
      "description": "搜索策略和结果评估",
      "version": "1.0.0",
      "source": "builtin",
      "installedAt": "2026-05-17T00:00:00.000Z",
      "path": "web-search-strategy/"
    },
    "pptx-generator": {
      "name": "pptx-generator",
      "description": "生成 PPTX 演示文稿",
      "version": "1.0.0",
      "source": "hub:github",
      "installedAt": "2026-05-17T00:00:00.000Z",
      "path": "pptx-generator/",
      "scanResult": { "safe": true, "scannedAt": "2026-05-17T00:00:00.000Z" }
    }
  }
}
```

`data/ai-skills/workspace/manifest.json`：

```json
{
  "schema": "workspace-skill-manifest.v1",
  "skills": {
    "web-search-strategy": {
      "enabled": true,
      "channels": ["all"],
      "source": "builtin",
      "installedFrom": "pool"
    }
  }
}
```

#### 6.2.4 新增模块

```text
lib/agent/skills/
├── pool-service.js      # Skill Pool 管理（安装/卸载/列表/同步内置）
├── workspace-service.js # Workspace 管理（启用/禁用/安装从 Pool）
├── hub.js               # 远程下载（GitHub + 未来扩展）
├── scanner.js           # 安全扫描器
└── store.js             # 文件存储（路径安全、原子写入）
```

#### 6.2.5 核心 API

```javascript
// pool-service.js
syncBuiltinSkills()           // 内置 Skill 同步到 Pool
installToPool(skillDir)       // 从目录安装到 Pool
removeFromPool(name)          // 从 Pool 移除
listPoolSkills()              // 列出 Pool 所有 Skill
getPoolSkillInfo(name)        // 获取 Pool Skill 详情

// workspace-service.js
installFromPool(name)         // 从 Pool 安装到 Workspace
enableSkill(name, channels)   // 启用 Skill
disableSkill(name)            // 禁用 Skill
removeFromWorkspace(name)     // 从 Workspace 移除
listWorkspaceSkills()         // 列出 Workspace Skill
getEffectiveSkills(channel)   // 获取当前渠道生效的 Skill

// hub.js
searchHub(query)              // 搜索远程 Skill
installFromHub(url)           // 从远程 URL 安装
```

### 6.3 验证标准

- 内置 Skill 能同步到 Pool。
- Pool Skill 能安装到 Workspace 并启用。
- 禁用的 Skill 不出现在 Agent 上下文中。
- Workspace manifest 损坏时能从 Pool 恢复。

## 7. 任务 E：远程 Skill 下载

### 7.1 QwenPaw 参考

QwenPaw 支持 6 个远程源（ClawHub、GitHub、LobeHub、ModelScope、skills.sh、skillsmp.com），有完整的 HTTP 重试、超时、取消机制。

### 7.2 莲莲Bot 实现方案

#### 7.2.1 三种来源

| 来源 | 方式 | Token 需求 |
|------|------|------------|
| **GitHub** | GitHub Contents API 下载目录/文件 | 可选（无 token 限速 60 req/h） |
| **直链 URL** | `fetch(url)` 下载 `.zip` 或 `SKILL.md` | 无 |
| **本地上传** | 用户上传 `.zip` 文件到服务器，解压后扫描 | 无 |

#### 7.2.2 GitHub 下载

```text
用户输入 URL（如 github.com/user/repo/tree/main/skills/my-skill）
  ↓
解析 URL → owner/repo/branch/path
  ↓
调用 GitHub Contents API 获取目录列表
  ↓
下载 SKILL.*.md + 附属文件（限制总大小 1MB）
  ↓
写入临时目录 → 安全扫描 → 通过 → 移入 Pool
```

支持格式：
- `https://github.com/user/repo/tree/branch/path/to/skill`（目录）
- `https://github.com/user/repo/blob/branch/path/to/SKILL.md`（单文件）
- `github:user/repo/path`（简写）

Token 配置：`process.env.GITHUB_TOKEN` 或 `data/github-token.txt`，无 token 时公开仓库仍可下载（速率限制 60 req/h）。

#### 7.2.3 直链 URL 下载

```text
用户输入 https://example.com/my-skill.zip
  ↓
fetch(url) 下载到临时文件（限制 1MB，15s 超时）
  ↓
是 .zip → 解压到临时目录
是 .md → 直接作为 SKILL.md
  ↓
验证：必须包含 SKILL.md，不得有可执行文件
  ↓
安全扫描 → 通过 → 移入 Pool
```

URL 来源不需要 adapter。GitHub、GitCode、个人网站等任何公开 URL 都支持，只要返回有效 Skill 文件。

#### 7.2.4 ZIP 上传

```text
用户在 Agent Console 选择 .zip 文件上传
  ↓
standalone.js 文件上传接口接收（已有）
  ↓
保存到临时目录 → 解压
  ↓
验证：必须包含 SKILL.md，不能有可执行文件、符号链接
  ↓
安全扫描 → 通过 → 移入 Pool → 可选安装到 Workspace
```

#### 7.2.5 自建技能

用户在 Agent Console 直接创建 Skill：

```text
Agent Console → 技能页 → "新建技能"按钮
  → 弹出表单：
    名称 (name)
    描述 (description)
    适用渠道 (channels: qq / dashboard / all)
    标签 (tags: 可选)
    SKILL.md 正文 (markdown 编辑区)
  → 保存到 data/skill-pool/{name}/SKILL.md
  → 写入 pool manifest
  → 安全扫描（同外部来源）
  → 通过 → 可选安装到 Workspace
```

**自建技能同样走安全扫描**，但不强制拦截。扫描结果仅提示用户，用户可以自行选择继续创建或放弃。尽职尽责，选择权在用户。

#### 7.2.6 统一下载流程

所有来源走同一套管道，但自建技能的扫描结果为**警告而非拒绝**：

```text
来源（GitHub / URL / ZIP / 自建）
  ↓
下载/接收 → 临时目录
  ↓
验证：SKILL.md 存在、无可执行文件、无路径穿越、大小合规
  ↓
安全扫描（scanner.js）
  ↓
CRITICAL/HIGH → 外部来源拒绝安装；自建技能提示用户，用户确认后可继续
MEDIUM/LOW → 警告，允许安装
SAFE → 直接通过
  ↓
移入 data/skill-pool/{name}/
  ↓
写入 pool manifest
  ↓
可选：安装到 Workspace → 启用
```

#### 7.2.7 安全措施

| 措施 | 说明 |
|------|------|
| 大小限制 | 单文件 500KB，总计 1MB |
| 文件数限制 | 最多 20 个文件 |
| 路径安全 | 拒绝 `../`、绝对路径、符号链接 |
| 超时 | GitHub: 15s；URL/解压: 30s；总计 60s |
| 重试 | GitHub 3 次，指数退避 |
| 速率限制 | 检测 GitHub 429，提示用户配置 token |
| 内容扫描 | 所有来源必须通过扫描才能入 Pool，无例外 |
| 可执行文件 | 拒绝 `.sh`、`.exe`、`.bat`、`.bin` 等 |

### 7.3 验证标准

- 能从 GitHub 公开仓库下载 Skill。
- 能从直链 URL 下载 `.zip` 或 `.md` Skill。
- 用户上传 `.zip` 成功安装。
- 自建技能保存后出现在 Pool 中。
- 恶意 Skill（含注入代码）被扫描器拦截，不能入 Pool。
- 路径穿越尝试被拒绝。

## 8. 任务 F：Skill 安全扫描器

### 8.1 QwenPaw 参考

QwenPaw 的安全扫描器：
- 17 个威胁类别（prompt_injection、command_injection、data_exfiltration 等）
- YAML 签名规则文件
- 严重度分级（CRITICAL/HIGH/MEDIUM/LOW/INFO/SAFE）
- 白名单 + 内容哈希固定
- 符号链接检测、路径边界检查、文件大小限制
- LRU 缓存 + mtime 失效

### 8.2 莲莲Bot 实现方案

#### 8.2.1 威胁类别（第一版 8 类）

| 类别 | 说明 | 严重度 |
|---|---|---|
| prompt_injection | 试图覆盖 system prompt 或注入角色 | CRITICAL |
| command_injection | 包含 shell 命令或代码执行指令 | CRITICAL |
| data_exfiltration | 试图外泄数据（URL、webhook、fetch） | HIGH |
| unauthorized_tool_use | 指示使用未授权工具 | HIGH |
| obfuscation | Base64 编码、Unicode 隐写、零宽字符 | HIGH |
| hardcoded_secrets | 包含 API key、token、密码模式 | MEDIUM |
| resource_abuse | 无限循环、大量文件操作指令 | MEDIUM |
| social_engineering | 诱导用户执行危险操作 | MEDIUM |

#### 8.2.2 扫描规则格式

```javascript
const SCAN_RULES = [
  {
    id: 'PI-001',
    category: 'prompt_injection',
    severity: 'CRITICAL',
    pattern: /(?:ignore|disregard|forget)\s+(?:all\s+)?(?:previous|above|prior)\s+(?:instructions|rules|prompts)/i,
    description: 'Attempts to override system instructions',
  },
  {
    id: 'PI-002',
    category: 'prompt_injection',
    severity: 'CRITICAL',
    pattern: /you\s+are\s+now\s+(?:a|an)\s+/i,
    description: 'Attempts to reassign agent identity',
  },
  {
    id: 'CI-001',
    category: 'command_injection',
    severity: 'CRITICAL',
    pattern: /(?:rm\s+-rf|sudo|chmod\s+777|curl.*\|\s*(?:bash|sh)|wget.*\|\s*(?:bash|sh))/i,
    description: 'Dangerous shell commands in skill content',
  },
  {
    id: 'DE-001',
    category: 'data_exfiltration',
    severity: 'HIGH',
    pattern: /(?:fetch|axios|http\.request|XMLHttpRequest)\s*\(/i,
    description: 'Network request code in skill',
  },
  // ... 更多规则
]
```

#### 8.2.3 扫描流程

```javascript
function scanSkillDirectory(skillDir) {
  const findings = []
  const files = listSkillFiles(skillDir)  // 限制数量和大小

  for (const file of files) {
    // 1. 元检查
    if (isSymlink(file)) findings.push({ severity: 'CRITICAL', ... })
    if (fileSize(file) > MAX_SKILL_FILE_SIZE) findings.push({ severity: 'HIGH', ... })

    // 2. 内容扫描
    const content = readFileSync(file, 'utf8')
    for (const rule of SCAN_RULES) {
      if (rule.pattern.test(content)) {
        findings.push({
          file: path.relative(skillDir, file),
          rule: rule.id,
          category: rule.category,
          severity: rule.severity,
          description: rule.description,
        })
      }
    }
  }

  const maxSeverity = getMaxSeverity(findings)
  return {
    safe: !findings.some(f => f.severity === 'CRITICAL' || f.severity === 'HIGH'),
    findings,
    maxSeverity,
    scannedAt: new Date().toISOString(),
  }
}
```

#### 8.2.4 扫描时机

- `installFromHub()` 下载后、入 Pool 前
- `installFromPool()` 安装到 Workspace 前
- `enableSkill()` 启用前（如果距上次扫描 > 24h）
- 管理员手动触发 `rescanAll()`

#### 8.2.5 白名单机制

```json
{
  "whitelist": {
    "web-search-strategy": {
      "hash": "sha256:abc123...",
      "reason": "内置 Skill，已人工审核",
      "addedBy": "admin",
      "addedAt": "2026-05-17T00:00:00.000Z"
    }
  }
}
```

白名单条目包含内容哈希，文件修改后哈希不匹配则白名单失效。

### 8.3 验证标准

- 包含 `ignore previous instructions` 的 Skill 被标记 CRITICAL。
- 包含 `curl | bash` 的 Skill 被标记 CRITICAL。
- 包含 `fetch()` 的 Skill 被标记 HIGH。
- 正常 Skill（纯 Markdown 指令）通过扫描。
- 白名单 Skill 跳过扫描。
- 白名单 Skill 被修改后重新触发扫描。

## 12. 风险控制

- 远程下载的 Skill 必须经过安全扫描，CRITICAL/HIGH 发现直接拒绝安装。
- Auto-Memory 使用主模型，失败静默跳过，不影响主流程。
- Dream 整理前必须备份，整理失败保留原文件。
- GitHub 下载有大小/数量/超时限制，防止资源滥用。
- 安全扫描规则必须有放行样例，避免误伤正常 Skill。
- 白名单使用内容哈希固定，防止篡改后绕过。
- 不允许远程 Skill 包含可执行代码（.js/.sh），只允许 Markdown 指令。
