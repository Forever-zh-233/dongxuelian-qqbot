# 表情包功能开发记录

## 现状

42 张表情包图片已部署到服务器，`stickerBase64Cache` 启动时加载到内存。代码层支持两种触发方式：

1. **AI 主动调用**：AI回复中包含 `[图:xxx]` 格式 → 代码替换为图片
2. **关键词自动匹配**：AI回复命中 `STICKER_MAP` 关键词 → 40%概率自动配图

## 遇到的难题及解决手段

### 难题 1：图片文件路径无法渲染

**表现：** `[CQ:image,file=/路径/图.jpg]` 被当成纯文本输出

**尝试过的修复：**
| 方案 | 结果 |
|---|---|
| 绝对路径 `file=/root/.../图.jpg` | ❌ 输出纯文本 |
| `file=file:///root/.../图.jpg` | ❌ 输出纯文本 |
| `file=base64://...`（嵌入字符串） | ❌ 消息体过大导致 API 报错，base64 被当作 text 段发送 |
| `[{type:'image',data:{file:'base64://...'}}]` 消息段对象 | ✅ 当前方案，但大图容易超限制 |

**结论：** `[CQ:image]` 的多种文件引用格式在 NapCat + OneBot adapter v6.9.3 上均未被正确解析，最终用消息段对象格式发送才可行。

### 难题 2：base64 数据过大导致发送失败

**表现：** 50KB 的 JPG → base64 约 70KB → 超过 QQ 单条消息体限制 → API 报错

**当前方案：** 在 `sendReply` 中用 `pendingStickers` 数组收集 base64，在文字消息发送完毕后，逐一调用 `session.send([{type:'image',data:{file:'base64://...'}}])` 单独发送图片消息。

**瓶颈：** 大图仍可能超限，需前端压缩或改用更小的图。

### 难题 3：AI 不主动输出 `[图:xxx]` 格式

**表现：** 用户说"调用一张图片"，AI 回复纯文本"好的，已调用"但不带 `[图:绷]`。

**尝试过的修复：**
| prompt 指令强度 | 结果 |
|---|---|
| `可以写 [图:xxx]`（建议语气） | ❌ 模型几乎不用 |
| `用户要求发图时必须用 [图:xxx]`（命令语气） | 未测试（刚改的版本） |
| 测试模式单独加指令 | 已加，但模型仍不遵守 |

**根因：** 模型需要在生成文本的同时决策插入 `[图:xxx]` 标记，属于额外的"工具调用"行为。大部分模型不会主动做这件事，除非有很强的 prompt 约束。

**关键词自动匹配（备选方案）：** 即使 AI 不输出 `[图:xxx]`，只要 AI 说了"绷""寄了""无语"等词，代码自动配图。这个方案目前正常工作（40%概率）。

### 难题 4：`[图:xxx]` 标记与真实关键词的关系

AI 输出的 `[图:xxx]` 中的 xxx 必须是 `STICKER_MAP` 中的 `kw` 值。如果 AI 输出 `[图:搞笑图片]` 而 map 中没有对应的 kw，就不会配图。当前 map 覆盖了 42 张图的关键词。

### 难题 5：怼人模式下不应发图

**方案：** 在 `chat()` 中 `saveConversationTurn` 前，如果 `hostile` 为 true，移除所有 `[图:xxx]` 标记。已实现。

## v0.6 更新 (2026-04-30)

### Bug 修复
- OneBot adapter v6.9.3 消息段 `file` 字段被 `attrs.file = attrs.src || attrs.url` 覆盖 → 改用 `h('image', { src })` 修复
- `\[图:\w+\]` 不匹配中文 → 改为 `\[图:[^\]]+\]`
- 重复检测误杀 `[图:xxx]` → 剥离标签后再比较
- 关键词自动匹配可能重复发送同一张贴图 → 添加 `pendingStickers.includes()` 去重
- `loadStickerCache()` 模块级冗余调用 → 删除

### 新功能
- 42 张表情包图片压缩部署（3.4MB → 303KB，全部 <15KB）
- 表情包通过关键词自动触发（60% 概率）
- 优化了日常聊天的人格，互动更自然
