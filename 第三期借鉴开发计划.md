# 莲莲Bot Agent — 第三期借鉴开发计划

> 合并来源：原《第三期借鉴开发计划》《Shell安全策略移植计划》《Agent独立前端建设计划》。本文件作为第三期唯一主计划，后续不再维护拆散的临时计划文件。
>
> 总目标：在第二期已完成的 Agent 引擎、工具体系、安全策略、Dashboard API、缓存优化和前端修复基础上，把莲莲Bot Agent 从“能调用工具的聊天助手”推进到“可规划、可并发、可审计、可主动执行、有独立控制台的 Agent 系统”。

---

## 0. 已有基础与本期边界

### 0.1 第二期已具备的基础

- Agent ReAct 引擎：`lib/agent/engine.js` 支持多轮模型调用、工具调用、pending 确认续跑。
- 工具体系：已有时间、计算、搜索、文件读写、日志查询、浏览器、Shell、JS 执行、发送文件、token usage 等工具。
- 安全体系：`safety.js` 管工具维度的 `auto / confirm / block`；`path-guard.js` 管文件路径；Shell 已接入 `shell-guard.js` 内容级规则。
- Dashboard 后端：`standalone.js` 已有 `/dashboard/api/agent/*`、pending、sessions、config 等接口。
- Dashboard 前端：现有 `AgentPanel.vue` 可配置工具、审批、会话、Skill、测试聊天，但仍嵌在大 Dashboard 里。
- 测试体系：`npm run check`、`npm run test:quick`、`npm run test:scenario` 已覆盖核心结构和行为链路。

### 0.2 第三期不做的事

- 不重写 `index.js` 主中间件，只做最小接线；新增能力放独立模块。
- 不引入远程 Skill 市场下载，不做未授权外部代码执行。
- 不把 Shell 安全策略降级为纯提示词；必须保留程序化拦截。
- 不让 Agent 长任务阻塞普通聊天。
- 不默认开启高风险工具或主动推送。
- 不把本地 data、密钥、运行日志作为前端静态资源暴露。

### 0.3 必守项目约束

- 新增生产 JS 模块必须同步：`package.json scripts.check`、`cascade-test.js` 的模块加载、导出、语法、重复扫描。
- 新增用户可感知行为必须优先补 scenario；cascade 只守结构契约和安全防线。
- 新增正则必须避免 ReDoS，安全规则必须有“应拦截”和“应放行”样例。
- 新增 `lib/*.js` 模块必须有 MODULE 注释块，说明职责、边界、状态。
- 部署仍优先走已有脚本，不覆盖服务器 data/config。

## 3. 任务 A：Shell 安全策略收口

### 3.1 当前状态

Shell 安全策略已完成核心移植：

- `lib/agent/tools/shell-guard.js`：独立规则模块。
- `lib/agent/tools/shell.js`：执行前调用 `checkShellCommand()`。
- `safety.js`：继续以工具维度管理 `execute_shell` 的 `auto / confirm / block`。

Shell 防线分两层：

```text
LLM 调 execute_shell
  ↓
safety.js：工具维度是否允许 execute_shell
  ↓
shell.js：参数和 cwd 基础校验
  ↓
shell-guard.js：命令内容级安全规则
  ↓
path-guard.js：路径边界
  ↓
execFile + timeout 执行
```

### 3.2 已移植规则类别

| 类别 | 规则数 | 例子 | 处理 |
|---|---:|---|---|
| 数据销毁 | 4 | `rm -rf`、`mv`、`mkfs`、`dd of=/dev` | 拦截 |
| 系统破坏 | 4 | `reboot`、`systemctl stop`、`kill -9`、fork bomb | 拦截 |
| 代码执行绕过 | 8 | `curl|bash`、`base64 -d|sh`、`$IFS`、危险 jq/zsh | 拦截 |
| 网络滥用 | 1 | reverse shell、`/dev/tcp`、`nc -e` | 拦截 |
| 敏感文件访问 | 2 | `/proc/*/environ`、`authorized_keys`、sudoers | 拦截 |
| 权限提升 | 2 | `sudo`、`su`、`pkexec`、危险 chmod/chattr | 拦截 |
| Shell 规避 | 6 | 命令替换、注释引号错位、反斜杠操作符、异常换行 | 拦截 |
| Backtick 注入 | 1 | 单引号外反引号 | 程序化检测 |
| 反斜杠空格 | 1 | `\ `、`\t` | 程序化检测 |

### 3.3 第三期需要补齐的收口项

1. **规则可展示**
   - `shell-guard.js` 导出规则摘要：`listShellGuardRules()`。
   - Agent Console 的工具详情页展示规则类别、严重度、说明。
   - Dashboard 旧 AgentPanel 可只显示“Shell Guard 已启用 + 规则数”。

2. **规则测试体系**
   - cascade：检查 `shell-guard.js` 导出、规则数量、规则 ID 唯一。
   - scenario 或纯模块测试：每类至少 1 个拦截样例、1 个放行样例。
   - 放行样例必须覆盖正常 `node -v`、`npm --version`、`ls`、`pwd`、只读 `git status`。

3. **误伤治理**
   - 对 `find -exec` 暂不移植，避免误伤正常查找；如后续加入，必须先做白名单化。
   - 对 `mv` 保守拦截；如确需允许白名单目录内 rename，必须走新工具而不是放宽 Shell。
   - 对 service/pkill 继续拦截；部署脚本自身不经过 Agent Shell 工具。

4. **结果形态统一**
   - `shell.js` 被 guard 拦截时返回结构化文案：规则 ID、严重度、原因、建议替代工具。
   - 不回显完整敏感命令中的密钥形态；命令摘要截断并脱敏。

### 3.4 不移植项说明

| QwenPaw 项 | 本期处理 | 原因 |
|---|---|---|
| SENSITIVE_FILE_BLOCK 数据集 | 暂不移植 | 本项目路径体系不同，已有 path-guard 和 data 保护 |
| TOOL_ABUSE_* 技能上传扫描 | 暂不移植 | 当前无远程 Skill 上传通道 |
| FIND_EXEC_PATTERN | 暂不移植 | 太保守，容易误伤正常只读查找 |

## 4. 任务 B：Agent/聊天并发拆队列

### 4.1 当前问题

现有 QQ 消息路径中，普通聊天和 Agent 请求可能共用同一个群队列。Agent 多轮工具调用可能耗时 20-60 秒，会让同群普通聊天排队等待。

```text
A：莲莲工具 查鸣潮最新角色 → Agent web_search/browser 30s
B：你好 → 不应等待 A 的 Agent 完成
C：现在几点 → 可走 Agent 或普通聊天，但不能被无关长任务拖死
```

### 4.2 目标行为

- 普通聊天队列和 Agent 队列分离。
- 同群普通聊天仍保持原有顺序和深度控制。
- Agent 队列按 `channelKey + userId` 隔离，避免同一用户重复堆积。
- Agent 全局并发有限制，避免 API 和浏览器资源被打爆。
- `browser_action` 默认互斥或 per-session 隔离，避免多个 Agent 同时操作同一个 page。
- pending 继续使用 `channelKey + ':' + userId`，确认不会串到别人任务。

### 4.3 设计

新增模块：`lib/agent/queue.js`

```js
/**
 * MODULE: Agent 队列调度。
 * 职责: 管理 Agent 长任务的 per-user 队列、群维度深度和全局并发。
 * 边界: 不调用模型、不发送 QQ 消息、不改聊天队列。
 * 状态: agentQueues / activeCount / depth counters。
 */
```

核心 API：

```js
enqueueAgentTask({ channelKey, userId, fn, maxDepth, timeoutMs })
getAgentQueueStats()
clearAgentQueue(channelKey, userId)
```

推荐策略：

| 维度 | 策略 |
|---|---|
| per-user | 同一群同一用户串行，最多排 1 个等待任务 |
| per-channel | 同一群 Agent 最多 3 个活跃/等待 |
| global | 全局 Agent 并发 2-4，可配置 |
| timeout | 单个 Agent 任务 90 秒硬超时 |
| 拒绝文案 | “Agent 正在处理上一个工具任务，稍后再试” |

### 4.4 index.js 接线原则

- 不重写中间件主流程。
- 只在路由判定 `useAgent` 后，把 `engine.run()` 包进 `enqueueAgentTask()`。
- 普通 `chat()` 继续走原 `enqueueForChannel()`。
- 路由判定本身要轻量，避免 LLM route decision 阻塞普通聊天；弱判断可以先进入 Agent 队列内部再做。

### 4.5 浏览器并发

`browser-action.js` 当前若复用模块级 browser/page，必须补一种保护：

- 第一版：全局浏览器互斥锁，所有 `browser_action` 串行。
- 第二版：按 Agent session 分配 page，限制最多 N 个 page。

第一版更稳，第三期优先第一版。

### 4.6 验证标准

- A 触发 30 秒 Agent，B 同群发“你好”，B 不等待 A 完成。
- A 和 C 同时触发 Agent，分别进入队列/并发，不互相覆盖 pending。
- 两个 browser_action 同时触发时不会抢同一个 page 状态。
- 超出队列深度时返回明确提示，不静默丢消息。

## 5. 任务 C：独立 Agent Console 前端

### 5.1 建设目标

从 Dashboard 中剥离 Agent 控制台，建设独立、现代、全功能的 Agent 前端 SPA。

现有 Dashboard 保留一个入口：点击进入 `/agent/`。旧 `AgentPanel.vue` 暂时保留作为降级入口或轻量入口，但不继续堆复杂 UI。

### 5.2 当前 Dashboard AgentPanel 的问题

| 问题 | 影响 |
|---|---|
| CSS 受 Dashboard 全局样式污染 | Agent 工具卡、聊天区很难做复杂交互 |
| 布局受限 | 左侧 Dashboard 导航占屏，聊天和工具面板拥挤 |
| 组件耦合 | 依赖 Dashboard AdminModal/PasswordField/主题变量 |
| 开发体验差 | 改 Agent UI 要构建整个 Dashboard 前端 |
| 信息密度过高 | 工具、审批、Skill、会话、聊天挤在一个 Vue 文件 |

### 5.3 技术选型

| 层 | 选型 | 说明 |
|---|---|---|
| 框架 | React 18 + TypeScript | 对齐 QwenPaw Console；适合复杂 Agent UI |
| 构建 | Vite 6 | 独立 HMR 和生产构建 |
| 样式 | Tailwind CSS 4 + shadcn/ui | 避免 Dashboard CSS 冲突 |
| 状态 | Zustand | 管 chat/config/pending/sessions |
| 请求 | fetch 封装 | 复用 `/dashboard/api/agent/*` |
| 部署 | standalone.js 挂载 `/agent/` | 与现有 Dashboard 同域，复用 token/cookie/localStorage |

> 若依赖兼容性或安装成本过高，Tailwind/shadcn 可分两步：先 React + CSS Modules，后续再引入组件库。不得因为 UI 库阻塞核心 Agent 功能。

### 5.4 QwenPaw 风格仿照要求

参考图里的 QwenPaw Console 不是简单的“工具面板 + 聊天框”，而是一套完整工作台。莲莲Bot 独立前端要按这个信息架构仿照，但替换成莲莲自己的品牌、接口和安全语义。

#### 5.4.1 总体视觉

| 视觉项 | QwenPaw 参考 | 莲莲Bot Agent Console 要求 |
|---|---|---|
| 整体风格 | 浅色、留白大、圆角卡片、细边框 | 默认浅色主题，保留暗色主题；主色用莲莲现有强调色或橙色系，不用 Dashboard 玻璃拟态 |
| 页面结构 | 左侧固定导航 + 中央大内容卡 + 顶部导航 | 采用同款三层结构：顶栏、左侧导航、主工作区；主工作区用大圆角白色容器 |
| 品牌区 | 左上 logo + 版本号 | 左上显示“莲莲 Agent”/项目 logo + 当前版本号；版本来自 package.json 或后端状态 |
| 激活态 | 灰底高亮 + 橙色点缀 | 当前菜单用浅灰背景，重要操作按钮用橙色/主色 |
| 卡片 | 工具/技能为 4 列卡片网格 | 工具、Skill、文件、统计都用卡片化展示，避免表格挤压 |
| 输入框 | 底部悬浮大输入框 | Chat 页底部使用 QwenPaw 式宽输入框，含附件、语音/快捷按钮、字数统计、发送按钮 |

#### 5.4.3 顶栏与右侧快捷栏

- 顶栏右侧仿照 QwenPaw：更新日志、文档、常见问题、GitHub、语言切换、主题/设备预览。
- 若没有对应链接，第一版只保留“文档/返回 Dashboard/主题切换/刷新”。不要生成不存在的外链。
- 右侧窄快捷栏可提供：搜索、创建新会话、设置；第一版可只做设置和主题切换。

#### 5.4.4 Chat 页面

QwenPaw Chat 的关键是“对话正文居中 + 底部大输入框 + 顶部模型/操作条”。莲莲实现要求：

- 顶部显示“新聊天”、当前模型、工具策略、刷新/复制/历史等小图标。
- 对话内容最大宽度居中，长文本排版类似文档阅读。
- assistant 消息带头像/名称“莲莲 Agent”，工具调用以可折叠卡片嵌在消息流中。
- pending 审批时消息流出现“等待确认”卡片，按钮为“确认 / 拒绝”。
- 输入框 placeholder 支持快捷提示：`/plan`、`/approve`、`/deny`、`@文件`、`#技能`。
- 多轮 Agent 未产出最终回复时不显示空气泡，只显示工具进度/等待状态。

#### 5.4.5 收件箱 Inbox

参考 QwenPaw 的“审批 / 推送消息”双 Tab：

- 审批 Tab：展示 pending 工具请求、计划确认、cron 高风险动作确认。
- 推送消息 Tab：展示主动推送历史、失败记录、今日限频额度。
- 空状态使用居中插画/图标 + “暂无待审批项”。
- pending 卡片必须显示：来源、工具名、参数摘要、过期倒计时、风险等级、确认/拒绝按钮。

#### 5.4.6 文件 Workspace

参考 QwenPaw 的文件页：左侧文件列表卡片，右侧编辑/预览区。

莲莲实现范围：

- 显示 effective read roots 和允许工作区。
- 文件列表支持搜索、刷新、上传、下载。
- 文件卡片显示文件名、大小、更新时间、是否允许注入上下文。
- 右侧预览支持 Markdown/文本；二进制文件只展示元信息。
- 写入/编辑必须走现有工具安全策略，不允许前端直接越权写任意路径。

#### 5.4.7 技能与技能池

参考图里有“技能”和“技能池”两个层级：

- **技能**：当前 Agent 已安装/可用技能，卡片展示启用状态、适用频道、更新时间、标签、描述，支持禁用/删除。
- **技能池**：本地可导入的 Skill 资源池，支持从内置技能加载、同步到技能池、zip 上传、导入 Hub（Hub 第一版仅保留入口，不做远程下载）。
- 卡片网格默认 4 列，窄屏自适应 1-2 列。
- 技能状态用绿色“已启用/最新”、橙色“已过期/需更新”、灰色“已禁用”。
- 批量操作按钮放页面右上，危险操作二次确认。

#### 5.4.8 工具页

参考 QwenPaw 的“内置工具”卡片网格：

- 1366px 以上：左侧导航 + 主内容完整展示。

#### 5.4.12 莲莲Bot 不照搬的部分

- 不照搬远程 Hub 下载，避免供应链风险；Hub 入口第一版只支持本地 zip/内置 Skill。
- 不照搬登录第三方论坛/平台的账号密码托管能力；涉及账号密码时必须提示用户手动登录或使用公开 API。
- 不开放未受控的 MCP/ACP 远程代理能力；先做本地多 Agent 协作预留。
- 不让前端直接执行工具；所有动作必须通过后端 Agent API 和安全策略。

### 5.7 核心组件要求

#### ChatLayout

- user 右侧、assistant 左侧。

- 工具调用卡片显示：tool name、args、status、duration、result 摘要。

- 支持 Ctrl+Enter 发送、Esc 取消输入。
- 本地保留最近 30 条 UI history，但不把 pending raw tool result 作为 assistant 历史回传。

#### ToolPanel

- 展示工具名称、描述、危险/外部网络/只读/写入标记。

- 安全模式 `auto / confirm / block`。

#### ApprovalQueue

- 实时 pending 列表：pendingId、toolName、参数摘要、channelKey/userId、过期倒计时。
- 支持确认/拒绝。
- 确认后展示 Agent 续跑最终答复，而不是只展示 raw tool result。
- ID 不匹配时显示明确错误。

#### SkillManager

- 搜索、过滤、启用/禁用。
- 展示 name/kind/description。
- 详情展开只展示本地 Skill 摘要，不暴露未经启用的完整正文。

#### BrowserPanel

- 显示 `browser_action` 是否启用、安全策略。
- 显示最近一次 URL/title/screenshot 摘要。
- 快捷动作只生成结构化请求，不绕过 Agent 安全策略直接操作浏览器。

#### ContextMonitor

- 展示 token 估算、工具调用次数、上下文压缩记录。
- 展示 Agent 队列状态：当前活跃数、等待数、最近超时。

#### HistoryPanel

- 使用 `GET /agent/sessions` 和 `GET /agent/sessions/:id`。
- 支持会话回放：userMessage、reply、toolCalls、pendingId。
- 不允许从历史中直接重放危险工具，必须重新走确认。

## 6. 任务 D：计划模式

### 6.1 目标

让 Agent 能把多步骤任务拆成计划并持久执行，而不是只靠一次 ReAct 上下文临时推进。

示例：

```text
用户：/plan 帮我准备一份鸣潮版本更新说明 PPT，生成后发到群里

计划：
1. 搜索版本更新内容
2. 整理关键更新点
3. 生成 PPTX 文件
4. 发送文件到当前群
5. 汇报完成
```

### 6.2 新增模块

```text
lib/agent/plan/
├── plan-engine.js     # 计划状态机和执行调度
├── plan-store.js      # 文件持久化 data/agent-plans/
├── plan-prompts.js    # 计划提示词模板
└── plan-tools.js      # create/update/finish/check 工具定义
```

### 6.3 数据模型

```json
{
  "id": "plan_20260515_001",
  "title": "准备鸣潮版本更新说明 PPT",
  "state": "executing",
  "channel": "qq",
  "channelKey": "1087798197",
  "userId": "532701045",
  "tasks": [
    { "id": "t1", "desc": "搜索版本更新内容", "state": "done", "outcome": "找到官方公告...", "toolCallCount": 1 },
    { "id": "t2", "desc": "整理关键更新点", "state": "in_progress", "outcome": null, "toolCallCount": 0 },
    { "id": "t3", "desc": "生成 PPTX 文件", "state": "todo", "outcome": null, "toolCallCount": 0 }
  ],
  "createdAt": "2026-05-15T00:00:00.000Z",
  "updatedAt": "2026-05-15T00:02:00.000Z"
}
```

状态机：

```text
todo → in_progress → done
                    ↘ abandoned
executing → done / abandoned / failed
```

### 6.4 计划工具

| 工具 | 说明 | 危险性 |
|---|---|---|
| `create_plan` | 创建计划和任务列表 | 安全，只写 data/agent-plans |
| `update_task_status` | 更新任务状态和 outcome | 安全，只写计划文件 |
| `check_plan_status` | 查询计划状态 | 只读 |
| `finish_plan` | 完成计划并生成总结 | 安全，可能触发 push |
| `abandon_plan` | 放弃计划 | 安全，需记录原因 |

### 6.5 触发方式

- 显式：`/plan xxx`、`莲莲计划 xxx`。
- 半自动：Agent 判断任务超过 2 步时，回复建议用户使用 `/plan`。
- 自动：配置 `agentConfig.planMode.autoCreate = true` 后才允许。

第一版默认只启用显式触发，避免 Agent 擅自长任务。

### 6.6 执行策略

- 计划执行走 Agent 队列，不走普通聊天队列。
- 每次只推进一个 task，完成后保存状态。
- 需要危险工具时进入 pending；确认后继续当前 task。
- 重启后从 `active.json` 恢复可继续计划。
- 计划完成后调用 `push.taskComplete()`，受推送限频控制。

### 6.7 测试

- `agent-plan.test.js`：创建计划、更新任务、完成计划、放弃计划。
- scenario：`/plan` 创建计划后，不阻塞普通聊天。
- pending 场景：计划中危险工具 pending → 确认 → 继续任务。

## 7. 任务 E：主动推送

### 7.1 目标

Agent 可在任务完成、定时任务、异常告警时主动发消息，但必须限频、可审计、可关闭。

### 7.2 设计决策

| 项 | 决策 |
|---|---|
| 推送风格 | 走人格润色，保持东雪莲语气 |
| 推送前缀 | 不默认加 `[Agent]`，避免破坏人设 |
| 频率限制 | 每群每天最多 5 条 Agent 主动推送 |
| 默认状态 | 关闭，管理员开启 |
| 触发来源 | 计划完成、cron、异常告警、管理员手动触发 |

### 7.3 新增模块

`lib/agent/push.js`

```js
push.send({ channelKey, text, bot, personalize = true, reason })
push.sendToAdmin({ text, bot, reason })
push.taskComplete({ planId, channelKey, summary, bot })
push.cronResult({ cronId, channelKey, text, bot })
push.getQuota(channelKey)
```

### 7.4 安全和审计

- 写入 `data/agent-push-log.jsonl`：时间、channelKey、reason、长度、是否成功。
- 不记录完整敏感内容；长文本截断。
- 推送失败不重试刷屏，只记录失败。
- 管理员命令可查看今日 quota。

### 7.5 接线

- `plan-engine.js` 完成计划 → `push.taskComplete()`。
- `cron.js` 执行结果 → `push.cronResult()`。
- 日志异常告警可后续接入，不作为第一版强制。

## 8. 任务 F：上下文压缩升级

### 8.1 目标

把当前基础裁剪升级为 LLM 结构化摘要，减少长任务中间状态丢失。

### 8.2 摘要格式

```markdown
## 目标
用户要完成什么任务。

## 进度
- 已完成：...
- 正在做：...
- 待做：...

## 关键事实
- 搜索/文件/日志得到的事实。
- 文件路径、生成物路径、pendingId 等。

## 决策
- 采用了哪些方案，为什么。

## 下一步
下一轮 Agent 应继续做什么。
```

### 8.3 实现

`context.js` 新增：

```js
compactWithLLM(messages, config)
buildStructuredSummaryPrompt(messages)
mergeSummaryIntoMessages(summary, recentMessages)
```

策略：

- 超过 60K token 触发。
- 保留最近 N 条完整消息和最近 M 条工具结果。
- 较早工具结果改为摘要，不保留大 raw result。
- LLM 摘要失败时回退现有 `compactMessages()`，不能中断 Agent。

### 8.4 测试

- LLM 摘要成功：输出含目标/进度/下一步。
- LLM 摘要失败：回退裁剪。
- 大工具结果：旧结果被摘要替换，最近结果保留。

## 11. 任务 I：浏览器全量补齐

### 11.1 目标

在现有 SSRF 和隐私防线基础上，把 `browser_action` 补齐到高频 30+ action。

### 11.2 优先动作

| 优先级 | action | 说明 |
|---|---|---|
| P0 | `evaluate` | 受限 JS，拦 localStorage/sessionStorage/内网 fetch/document.write/window.open |
| P0 | `batch` | 批量动作，单步失败继续，最后汇总 |
| P0 | `pdf` | 导出 PDF 到 `data/agent-browser/` |
| P0 | `network_requests` | 最近 50 条请求摘要，脱敏 header/cookie |
| P0 | `console_messages` | 最近 50 条 console，截断 |
| P0 | `cookies_set` | 需确认，限制字段和域名 |
| P0 | `fill_form` | 批量填表，限制 fields 数量 |
| P0 | `navigate_forward` / `reload` | 导航补齐 |
| P1 | `hover` / `select_option` / `drag` | 交互补齐 |
| P1 | `file_upload` / `file_download` | 受控目录 |
| P1 | `resize` / `clear_cache` / `cookies_clear` | 环境控制 |

### 11.3 安全要求

- URL 继续拒绝 localhost、loopback、私网、link-local、metadata、带 username/password 的 URL。
- `evaluate` 默认危险，Dashboard 开启也需 confirm，QQ 默认关闭。
- `cookies_get` 默认脱敏；`cookies_set` 必须确认。
- `batch` 不允许绕过单个 action 的安全策略。
- 所有返回结果有长度上限。
- 文件输出只能写入 `data/agent-browser/`。

### 11.4 并发要求

- 第一版浏览器动作全局互斥。
- batch 内部串行执行。
- 超时后释放锁。

## 16. 风险控制

- Agent Console 是新目录，不应污染 Dashboard 现有样式和构建。
- Shell Guard 不因用户体验放宽高危规则；需要新能力时新增专用工具，而不是放开 Shell。
- Agent 长任务必须有队列深度和超时，避免刷爆 API。
- 主动推送默认关闭，开启后限频并写审计日志。
- Cron 禁止秒级/分钟级刷屏任务，默认最小间隔 10 分钟。
- Memory 仅管理员默认可用，必须可删除，不自动记录敏感信息。
- Browser evaluate/cookies/file 动作保持 confirm 和脱敏。
- 文档生成不引入 LibreOffice 这类重运行时，先用纯 JS 方案。
- 任何新增依赖必须确认 Windows 本地部署器和 Linux 服务器都能安装。

## 18. 最终验收标准

1. QQ 群里 Agent 长任务不会阻塞普通聊天。
2. Shell 高危命令被内容级规则稳定拦截，并能在 Agent Console 查看规则摘要。
3. `/agent/` 独立前端可完成：聊天、工具开关、pending 审批、Skill 管理、sessions 查看、stats 查看。

5. 计划完成、cron 执行结果能在限频下主动推送。
6. 上下文超大时能生成结构化摘要，不丢目标/进度/下一步。
7. Cron 和 Memory 默认受管理员权限保护，可查看、删除、关闭。
8. Browser 新动作可用，且 SSRF、cookies、evaluate、文件边界防线有效。
9. PPTX Skill 能生成文件并通过 `send_file_to_user` 发送或给出明确降级提示。
10. 全部新增模块通过 check/quick/scenario，前端构建通过，`git diff --check` 无错误。
