# 东雪莲 QQ 机器人开发成果总结

## 一、服务器部署与基础架构修复

从零开始在 120.55.246.12（Ubuntu 22.04）部署了完整的 QQ 机器人系统，架构为：

```
NapCat（QQ 登录, WS 服务端 :8080）
  → Koishi v4.18.11（adapter-onebot, WS 客户端）
  → dongxuelian-ai 插件
  → 多 API 后端（OpenCode Go / DeepSeek 官方 / 阿里云）
```

修复了三大基础设施问题：

**1. NapCat Worker 进程崩溃：** 添加 `--disable-gpu` 启动参数解决 GPU 兼容性问题。

**2. adapter-onebot 无法连接 NapCat：** 根因是 koishi.yml 格式错误，将 server 插件格式改为根级 port/selfUrl，协议从 ws-reverse 改为 ws 正向，因为 NapCat 是 WebSocket 服务端，Koishi 须以客户端身份连接。

**3. @satorijs/core 版本冲突：** Koishi 4.x 需要 core ^4.6.0，但 adapter-onebot 依赖 satori 3.x 锁死 core 3.7.0。通过 package.json 添加 overrides 锁定版本，同时创建 patch.js 通过 NODE_OPTIONS 预加载补齐 3.x 缺失的 Session API（resolve、stripped、send）。

---

## 二、识图功能（多模态 + OCR 双轨）

实现了完整的图片识别管线：

从 QQ 消息中提取图片 file 参数 → 调用 NapCat 的 get_image API 获取本地缓存路径 → 读取本地图片文件 → base64 编码 → 发送给多模态模型。

同时集成了 tesseract OCR（chi_sim+eng）作为非视觉模型的回退方案，当模型不支持多模态时自动走 OCR 管线。

中间件中添加了白名单和概率控制：只有白名单群中的图片才会触发识图，非白名单群和无 @ 的图片静默跳过。非多模态模型（如 deepseek-chat）自动跳过识图流程。

---

## 三、转发消息处理

实现了完整的合并转发消息识别管线：

**外层转发：** 检测 session.content 中的 `<forward id="xxx"/>` 或 CQ 码格式 → 调用 callGetForwardMsg 通过 NapCat API 拉取消息节点 → 转换为统一格式 → 通过 summarizeForwardNodes 格式化为带层级的可读文本 → 注入 AI 上下文。

**嵌套转发：** 检测消息段中的 forward 类型或 raw_message 中的 CQ 码 → 递归调用 callGetForwardMsg 尝试拉取 → NapCat 明确返回"内层消息无法获取"时降级显示 `[嵌套转发：内容暂不可见]`。

**格式优化：** 加入层级标记（`[对话]` / `└─ [内层转发]`），接入 MAX_FORWARD_NODES（50 条）和 MAX_FORWARD_DEPTH（4 层）做截断控制，超出部分提示"还有 N 条消息未显示"。

**昵称净化：** 过滤不可见字符（韩文填充符、零宽空格等 U+200B-U+11FF 范围），空昵称回退为"群友"。

---

## 四、引用消息处理

实现了引用消息的内容注入：直接从 session.quote 提取被引用的消息内容，解析文本/图片/表情/@/转发/视频/语音/文件等多种消息段类型，将引用内容以 `[引用内容]xxx[以上是对方说的话，不是在对你说]` 格式注入用户消息，同时去掉重复的系统消息注入，避免 AI 混淆。

---

## 五、模型切换系统（三分层父子菜单）

设计了三层切换体系：`切换模型` → 选择供应商 → 选择模型。支持三个供应商：

- **OpenCode Go**：14 个模型（千问 3.5、千问 3.6、GLM-5、GLM-5.1、Kimi K2.5、Kimi K2.6、DSv4pro、DSv4、MiMo-V2-Pro、MiMo-V2-Omni、MiMo-V2.5-Pro、MiMo-V2.5、MiniMax M2.7、MiniMax M2.5）
- **阿里云 DashScope**：qwen3.5-plus、qwen3.6-plus
- **DeepSeek 官方**：deepseek-chat、deepseek-v4-flash、deepseek-v4-pro

切换时自动切换 API 地址和对应密钥。为 OpenCode Go 的 DeepSeek 模型添加了 reasoning_content 回退处理（当 content 为空时使用 reasoning_content）。

---

## 六、联网搜索能力

在 getSearchCapability 函数中添加了 qwen 模型识别逻辑，使通过 OpenCode Go 调用的 qwen 模型也能使用 enable_search 参数进行联网搜索，突破了原有代码只识别 DashScope 和 OpenAI 官方 API 的限制。支持 dashscope-chat、openai-chat-search、openai-responses 三种搜索模式。

---

## 七、消息队列与稳定性

给 enqueueForChannel 添加了 60 秒超时机制，防止单个 API 请求卡死整个消息队列。修复了 config 作用域问题（config is not defined、vConfig is not defined），在视觉处理代码块中单独加载配置。修复了 helpAI 菜单重复发送的问题。

---

## 八、白名单与概率回复系统

实现了群聊白名单加概率触发的双层过滤机制。白名单群中的消息按设定概率回复，非白名单群全部跳过。每个群可单独设置概率（如 30%、5%），带有 warmup 机制（50 条消息内不递增，之后每条递增 2%）。

修改了 shouldSkipForRandomReply 逻辑，使图片也参与概率判断（修复了图片总是回复的 bug）。同时保留了 @ 提及和私聊的始终回复逻辑。随机回复时添加 `[群聊刷到]` 标记，让 AI 以群友插话口吻回应而非误以为被直接召唤。

---

## 九、多 API 密钥管理

实现了三套 API 密钥的自动切换机制：ai-openai-key.txt（OpenCode Go）、ai-deepseek-key.txt（DeepSeek 官方）、ai-dashscope-key.txt（阿里云），根据当前激活的供应商自动选择对应的密钥和 API 地址。

---

## 十、提示词工程与人设优化

**政治红线：** 写入两套人格，包含红线示例（党和国家领导人、领土主权、政治运动、历史事件等）、隐喻/谐音/反讽识别提醒、标准拒绝话术。

**脏话替代词：** 嘴臭人格中加入屎→史、妈→老冯、傻逼→沙鼻、废物→飞舞、智障→智战等谐音替代规则，同时要求减少屎尿屁输出，并提醒模仿例子时切换代替词。友善人格中删除替代规则（因为友善模式不骂人）。

**字数控制：** 取消 trimReply 的硬截断，改为提示词层面的软性引导"评价类回复控制在两三句话以内"。

**评价类回复：** 加入引导"先看完内容再回应，有值得评论的地方直接说"，避免 AI 盲目输出。识图提示词改为"直接评价这张图，不要描述过程"。

**随机回复插话：** 加入"当前消息是你在群聊里看到的，对方不是在叫你"的上下文标记，防止 AI 用回应对方的口吻回复。

**回复字数放宽：** max_tokens 从 500 提升到 1000，支持更长的回复。

**摆烂机制：** 当 completion_tokens 超过 600 时，AI 自动输出"太麻烦不想了，摆烂了"等随机摆烂回复。

## 十一、测试与维护工具

**测试人格：** 管理员通过"测试开/测试关"命令切换，开启后 AI 进入绝对服从模式，原有所有人格设定被覆盖。

**维护模式：** 通过 ai-paused.txt 文件开关，开启后所有消息统一回复"优化中，别急"，不触发任何 AI 调用。

**代码同步：** E 盘莲莲Bot 与服务器代码定期同步，通过 git 管理并推送到 GitHub bot_ZZY 分支（不碰 main 分支）。

**冗余清理：** 清理了调试日志、未使用的常量等冗余代码，备份了被删除的代码行以便回退。

---

## 十二、QQ 机器人识图配置

QQ 图片有本地缓存路径，位于服务器 `/root/.config/QQ/nt_qq_*/nt_data/Pic/YYYY-MM/Ori/`。识图流程通过 NapCat 的 get_image API 获取本地文件路径，再读取并编码为 base64。非视觉模型自动回退到 tesseract OCR（chi_sim+eng）。
