# NapCat + Koishi 换号操作手册

这份文档不是“排障备忘录”，而是从零开始的换号操作手册。

目标是把下面整条链路按顺序切过去：

```text
停掉旧 QQ 号
-> 用 NapCat 登录新 QQ 号
-> 让新号自己的 OneBot 配置生效
-> Koishi 绑定新 selfId
-> NapCat 的 3001 WebSocket 正常监听
-> Koishi 成功连上新号
-> 群里能正常收发消息
```

这份文档尽量只写能直接执行和直接判断的命令，不靠猜。

---

## 一、先说这次真实根因

这次不是 `Koishi` 主配置写错了，也不是 AI 插件有问题。

真实根因是：

1. `Koishi` 已经把 `selfId` 改成新号了。
2. `NapCat` 也确实登录到了新号。
3. 但新号自己的配置文件 `onebot11_<新QQ号>.json` 里，`websocketServers` 是空的。
4. 所以新号虽然在线，但没把 `3001` 的 `OneBot WebSocket Server` 起起来。
5. 结果就是 `Koishi` 一直报：

```text
adapter connect ECONNREFUSED 127.0.0.1:3001
```

以后换号，千万不要只改 `Koishi selfId`，还要确认新号自己的 `NapCat OneBot` 配置是不是完整继承过去了。

```text
/root/Napcat/opt/QQ/resources/app/app_launcher/napcat/config
```

这次实际涉及的配置文件：

```text
/root/Napcat/opt/QQ/resources/app/app_launcher/napcat/config/onebot11_3098291287.json

新号：
/root/Napcat/opt/QQ/resources/app/app_launcher/napcat/config/onebot11_3651312852.json
```

## 三、换号前先做备份

先别动服务，先把关键文件备份出来。

### 第 1 步：备份 Koishi 配置

```bash
/bin/cp /root/koishi-app/koishi.yml /root/koishi-app/koishi.yml.bak-before-switch
```

### 第 2 步：备份 NapCat 配置目录

```bash
mkdir -p /root/Napcat-config-backup
/bin/cp -r /root/Napcat/opt/QQ/resources/app/app_launcher/napcat/config /root/Napcat-config-backup/config-$(date +%F-%H%M%S)
```

### 第 3 步：记录当前有哪些账号配置

```bash
ls -lh /root/Napcat/opt/QQ/resources/app/app_launcher/napcat/config
ls -lh /root/Napcat/log
```

这一步的目的不是修东西，只是先把现场留住，防止你后面改乱了没法回退。

## 四、先识别 NapCat 是怎么启动的

这一段很重要。

你不能一上来就写死 `systemctl restart napcat`，因为不同机器的 NapCat 启动方式可能不一样。

先查，再动。

### 第 1 步：先查 systemd 里有没有 NapCat 服务

```bash
systemctl list-unit-files --type=service | grep -i napcat
systemctl list-units --type=service | grep -i napcat
```

如果看到了像下面这种东西：

```text
napcat.service
```

或者：

```text
NapCat.service
```

记住这个服务名，后面就按这个名字操作。

### 第 2 步：再查当前 QQ / NapCat 相关进程

```bash
ps -ef | grep -Ei "napcat|qq|onebot" | grep -v grep
```

你要重点看：

1. 有没有 `qq` 进程
2. 有没有 `napcat` 相关进程
3. 这些进程是不是你预期的那套 NapCat

### 第 3 步：如果你不确定谁拉起了 QQ，就查父子关系

```bash
ps -eo pid,ppid,cmd | grep -Ei "napcat|qq" | grep -v grep
```

如果 `qq` 进程是被某个 `systemd service` 拉起来的，通常这里能看出痕迹。

### 判断规则

#### 情况 A：有明确 `napcat.service`

后面优先用：

```bash
systemctl stop <服务名>
systemctl start <服务名>
systemctl restart <服务名>
```

#### 情况 B：没有 systemd 服务，但有独立启动脚本或面板

后面就按你原来的启动方式重启，不要乱 invent 新命令。

#### 情况 C：只看到裸 `qq` 进程，看不出谁启动的

那你先别硬杀，先找到你平时是怎么启动 NapCat 的，再继续。

这一条不是偷懒，是为了防止把机器上别的东西误停。

## 五、从旧号切到新号

这一段才是“怎么给 NapCat 换号”。

换号本质上分成两件事：

1. 让 NapCat 不再跑旧号
2. 让 NapCat 重新启动后登录到新号

### 第 1 步：先停掉当前 NapCat / QQ

如果你的 NapCat 是 `systemd` 管理，比如服务名是 `napcat`：

```bash
systemctl stop napcat
systemctl status napcat --no-pager
```

如果服务名不是 `napcat`，把名字换成你查到的那个。

如果你不是 `systemd` 管理，而是自己手动启动的，那就按你平时的方式停掉它。

停完以后，用下面命令确认旧的 QQ 进程已经没了：

```bash
ps -ef | grep -Ei "napcat|qq|onebot" | grep -v grep
```

如果还有残留 `qq` 进程，再判断是不是旧 NapCat 没退干净。

### 第 2 步：启动 NapCat，让它进入登录流程

如果是 `systemd`：

```bash
systemctl start napcat
systemctl status napcat --no-pager
```

如果不是 `systemd`，用你原来的启动命令。

这一段文档没法替你猜具体启动脚本名，但原则是固定的：

```text
你必须用你机器上原本的那套启动方式，把 NapCat 再起一次
```

### 第 3 步：打开 NapCat 登录入口，确认当前登录的是谁

这一段一定要做，不然你根本不知道它到底登的是旧号还是新号。

先找 NapCat 日志里对应新号或旧号的文件：

```bash
ls -lh /root/Napcat/log
```

再持续看最新日志变化：

```bash
tail -f /root/Napcat/log/napcat_*.log
```

如果你已经知道候选账号，比如旧号 `3098291287`、新号 `3651312852`，那就分别看：

```bash
tail -n 80 /root/Napcat/log/napcat_3098291287.log
tail -n 80 /root/Napcat/log/napcat_3651312852.log
```

你要确认的是：

1. 新号日志文件是否开始刷新
2. 旧号日志是否不再继续刷新

### 第 4 步：通过日志确认 NapCat 实际登录的是新号

换号不是“我以为我换了”，而是要看证据。

最直接的方法就是：

```bash
ls -ltr /root/Napcat/log | tail -n 10
```

然后看最新正在写入的是不是：

```text
napcat_<新QQ号>.log
```

再用：

```bash
tail -n 80 /root/Napcat/log/napcat_<新QQ号>.log
```

如果这个文件一直在新增日志，就说明 NapCat 当前活跃账号确实是新号。

这一段的目标只有一个：

```text
先确认 NapCat 登录的新号到底是不是你想要的新号
```

这里还不是确认 `OneBot` 一定通了。

---

## 六、确认 Koishi 已经绑定到新号

NapCat 登录新号以后，再检查 `Koishi`。

### 第 1 步：查看 Koishi 的 OneBot 配置

```bash
grep -nE "adapter-onebot|selfId|endpoint" /root/koishi-app/koishi.yml
```

你要确认至少两件事：

1. `selfId` 已经是新号
2. `endpoint` 还是你要连的地址

这次实际正确值就是：

```text
selfId: '3651312852'
endpoint: ws://127.0.0.1:3001
```

### 第 2 步：如果 `selfId` 还是旧号，先改它

不要先急着重启全套服务，先把配置改对。

改完以后再确认一遍：

```bash
grep -nE "adapter-onebot|selfId|endpoint" /root/koishi-app/koishi.yml
```

---

## 七、第一次联调：先看 Koishi 报不报 3001 refused

### 第 1 步：重启 Koishi

```bash
systemctl restart koishi
```

### 第 2 步：直接看最近日志

```bash
journalctl -u koishi -n 80 --no-pager
```

如果你看到：

```text
adapter connect ECONNREFUSED 127.0.0.1:3001
adapter failed to connect to ws://127.0.0.1:3001/, code: 1006
```

说明当前问题不是插件没加载，而是：

```text
Koishi 想连 3001
但 NapCat 没把 3001 的 OneBot WebSocket 服务起起来
```

这时别再盯着 `Koishi` 了，直接回头查 `NapCat`。

---

## 八、确认 3001 到底有没有监听

### 第 1 步：直接查端口

```bash
ss -lntp | grep 3001
```

### 判断规则

#### 情况 A：没有任何输出

说明 `3001` 没监听。

这时根因大概率在：

1. 新号对应的 `onebot11_<新QQ号>.json` 不完整
2. NapCat 没读到正确 OneBot 配置
3. NapCat 启动了，但 OneBot WebSocket Server 没开

#### 情况 B：有输出

例如：

```text
LISTEN 0 511 *:3001 *:* users:(("qq",pid=97134,fd=180))
```

说明 NapCat 已经在 `3001` 监听了。

这时如果 `Koishi` 还连不上，再回头看：

1. `endpoint` 是否写错
2. `selfId` 是否写错
3. 是否刚重启，服务还没完全起来

---

## 九、检查新号自己的 OneBot 配置文件

这一段是本次事故的核心。

### 第 1 步：列出 NapCat 配置目录

```bash
cd /root/Napcat/opt/QQ/resources/app/app_launcher/napcat/config
ls -lh
```

你要重点看：

```text
onebot11_<QQ号>.json
```

### 第 2 步：先看旧号配置

```bash
cat /root/Napcat/opt/QQ/resources/app/app_launcher/napcat/config/onebot11_3098291287.json
```

### 第 3 步：再看新号配置

```bash
cat /root/Napcat/opt/QQ/resources/app/app_launcher/napcat/config/onebot11_3651312852.json
```

### 第 4 步：重点核对 `websocketServers`

你期待看到的应该是这种结构：

```json
"websocketServers": [
  {
    "enable": true,
    "host": "0.0.0.0",
    "port": 3001,
    "reportSelfMessage": false,
    "enableForcePushEvent": true,
    "messagePostFormat": "array",
    "token": "zxcvbnmzxcvbnm",
    "debug": true,
    "heartInterval": 30000
  }
]
```

如果你在新号文件里看到的是：

```json
"websocketServers": []
```

那根因就找到了。

这次真实事故就是这个。

## 十、修新号的 OneBot 配置

### 第 1 步：先备份新号配置

```bash
/bin/cp /root/Napcat/opt/QQ/resources/app/app_launcher/napcat/config/onebot11_3651312852.json /root/Napcat/opt/QQ/resources/app/app_launcher/napcat/config/onebot11_3651312852.json.bak
```

如果你机器上的 `cp` 被 alias 成交互模式，优先用 `/bin/cp`，不要跟它来回扯。

### 第 2 步：直接用旧号可用配置覆盖新号配置

如果旧号那份是你已经验证过可用的配置，最稳的办法不是手抄 JSON，而是直接复制：

```bash
/bin/cp /root/Napcat/opt/QQ/resources/app/app_launcher/napcat/config/onebot11_3098291287.json /root/Napcat/opt/QQ/resources/app/app_launcher/napcat/config/onebot11_3651312852.json
```

### 第 3 步：覆盖后立刻再看一遍新号配置

```bash
cat /root/Napcat/opt/QQ/resources/app/app_launcher/napcat/config/onebot11_3651312852.json
```

你至少要确认这几个关键字段还在：

```json
{
  "network": {
    "websocketServers": [
      {
        "enable": true,
        "host": "0.0.0.0",
        "port": 3001,
        "messagePostFormat": "array"
      }
    ]
  }
}
```

如果你不想整份复制，也可以自己手改，但风险更高，容易漏掉 `token`、`messagePostFormat`、`enableForcePushEvent` 这些字段。

所以这套机器上，优先推荐“复制旧号可用配置给新号”。

## 十一、让 NapCat 重新读取新号配置

你改完 JSON 还不算完，必须让 NapCat 重新加载它。

### 第 1 步：重启 NapCat

如果服务名是 `napcat`：

```bash
systemctl restart napcat
```

如果你的服务名不是这个，就换成你前面查到的名字。

如果你不是 `systemd` 管理，就用你自己的启动方式重启。

这里不要瞎猜服务名，也不要把文档里的 `napcat` 三个字当绝对值。

### 第 2 步：重启后先查进程

```bash
ps -ef | grep -Ei "napcat|qq|onebot" | grep -v grep
```

确认 `qq` 进程已经重新起来了。

---

## 十二、验证 NapCat 的新号 OneBot 确实起来了

### 第 1 步：再查 3001 监听

```bash
ss -lntp | grep 3001
```

正确结果通常类似：

```text
LISTEN 0 511 *:3001 *:* users:(("qq",pid=97134,fd=180))
```

### 第 2 步：看新号日志里有没有 WebSocket 启动记录

```bash
grep -niE "3001|websocket|listen|OneBot11" /root/Napcat/log/napcat_3651312852.log | tail -n 80
```

你要看到类似：

```text
[OneBot11] [network] 配置加载
WebSocket服务: 0.0.0.0:3001, : 已启动
[OneBot] [WebSocket Server] Server Started :::3001
```

只要这几行出来，就说明：

```text
NapCat 新号自己的 OneBot WebSocket Server 已经真的起来了
```

这一步完成后，`NapCat -> OneBot` 这一段才算真正打通。

---

## 十三、再让 Koishi 连一次

### 第 1 步：重启 Koishi

```bash
systemctl restart koishi
```

### 第 2 步：立刻看新日志

```bash
journalctl -u koishi -n 50 --no-pager
```

必要时实时盯：

```bash
journalctl -u koishi -f
```

### 判断规则

#### 坏信号

```text
adapter connect ECONNREFUSED 127.0.0.1:3001
```

如果还在刷，说明：

1. NapCat 端口仍未就绪
2. 你看的不是当前新号日志
3. `endpoint` 写错
4. 服务重启顺序不对

#### 好信号

1. 不再持续刷 `3001 refused`
2. Koishi 适配器恢复正常收发

有些版本不一定明确打印 `connected`，但只要不再刷拒绝连接，并且群里能正常回复，就可以认定链路恢复。

直接 `@机器人` 发一句话。

如果机器人能正常回你，就说明：

```text
Koishi -> NapCat -> QQ
```

发消息链路也正常。

## 十五、这次事故的快速判定模板

以后只要看到下面这个组合，基本就可以直接怀疑“新号的 OneBot 配置没继承过去”：

1. `Koishi selfId` 已经改成新号
2. NapCat 也已经登录新号
3. `Koishi` 持续报：

```text
adapter connect ECONNREFUSED 127.0.0.1:3001
```

4. `ss -lntp | grep 3001` 没监听
5. 新号日志里只有：

```text
[OneBot11] [network] 配置加载
```

没有：

```text
WebSocket服务: 0.0.0.0:3001, : 已启动
```

6. 新号的 `onebot11_<新QQ号>.json` 里看到：

```json
"websocketServers": []
```

出现这一套，基本不用先怀疑 AI 插件、帮助插件、昵称插件。

先修 NapCat 新号的 OneBot 配置。

## 十六、以后换号的标准顺序

以后固定按这个顺序走，不要跳：

1. 备份 `koishi.yml`
2. 备份 NapCat `config` 目录
3. 查清楚 NapCat 的启动方式
4. 停掉当前旧号对应的 NapCat / QQ
5. 重新启动 NapCat，让它登录新号
6. 用日志确认当前活跃账号确实是新号
7. 把 `Koishi selfId` 改成新号
8. 检查新号自己的 `onebot11_<新号>.json`
9. 如果新号 `websocketServers` 为空，直接复制旧号可用配置给新号
10. 重启 NapCat
11. 用 `ss -lntp | grep 3001` 确认监听
12. 用新号日志确认 `WebSocket Server Started`
13. 重启 Koishi
14. 群里实际收发消息验收

## 十八、关于 `Error: No open ports available`

这条日志和 `3001 refused` 不是同一个问题。

这次实际情况是：

1. 主故障是新号 `onebot11_3651312852.json` 没开 `websocketServers`
2. 把新号 OneBot 配置补齐后，`3001` 就恢复监听了
3. 这说明 QQ 收发主链路已经恢复

所以排障优先级必须是：

1. 先修 `NapCat -> OneBot -> Koishi`
2. 机器人恢复收发后
3. 再单独处理 `No open ports available`

不要一开始被这条日志带偏。

## 二十、后续维护建议

### 建议 1：每次换号后，第一时间复制旧号可用配置给新号

也就是登录新号后立刻执行：

```bash
/bin/cp /root/Napcat/opt/QQ/resources/app/app_launcher/napcat/config/onebot11_<旧号>.json /root/Napcat/opt/QQ/resources/app/app_launcher/napcat/config/onebot11_<新号>.json
```

然后再重启 NapCat。

### 建议 2：换号完成只认 3 个硬指标

必须同时满足：

1. `ss -lntp | grep 3001` 有监听
2. 新号日志里出现 `WebSocket Server Started`
3. 群里能实际收发消息

少一个，都不算换号完成。
