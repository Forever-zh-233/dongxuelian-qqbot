## 二、算法 / 性能优化点

### 2.1 LCS（最长公共子串）：阈值早停 + 短串作内层

`isReplyTooSimilar` 的判定本质是「LCS 是否 ≥ 较短串长度的 50%」。原实现每次都跑完整的 O(n·m) DP，在长回复上是热点。改动：

- 让 `dp` 数组的长度跟随**较短串**而不是 `b`，常数空间从 max 降到 min。
- 增加 `threshold` 参数，DP 内层一旦 `maxLen >= threshold` 立刻 `return`。
- 用 `charCodeAt` 替代字符串索引比较，减少临时 string 分配。

```js
function longestCommonSubstringLength(a, b, threshold = Infinity) {
  // 让较短串作内层，dp 长度更小
  let outer = a, inner = b
  if (inner.length > outer.length) { outer = b; inner = a }
  const dp = new Array(inner.length + 1).fill(0)
  let maxLen = 0
  for (let i = 1; i <= outer.length; i++) {
    let prev = 0
    const ci = outer.charCodeAt(i - 1)
    for (let j = 1; j <= inner.length; j++) {
      const temp = dp[j]
      if (ci === inner.charCodeAt(j - 1)) {
        const cur = prev + 1
        dp[j] = cur
        if (cur > maxLen) {
          maxLen = cur
          if (maxLen >= threshold) return maxLen   // 阈值早停
        }
      } else if (temp !== 0) {
        dp[j] = 0
      }
      prev = temp
    }
  }
  return maxLen
}
```

### 2.2 `isReplyTooSimilar`：字符集 Jaccard 预过滤

LCS 长度 ≤ 两串字符集交集大小。这是 O(n+m) 可以算的上界。
所以在调用 LCS 之前先做一次廉价检查：

```js
function isReplyTooSimilar(left, right) {
```

效果：当两条回复字符集差异大时，**不再触发 LCS**，直接返回 `false`。

### 2.3 `extractAtIds` / `countAtIdOccurrences`：常量化正则 + Set 去重

原实现：

- 每次调用都在函数体内用字面量构造数组（`patterns = [...]`）。
- 用 `Array.includes` 去重，长消息里 `O(n)` 重复线性扫描。

改动：

- 把两个 `/g` 正则提为模块级常量 `AT_ID_PATTERN_XML` / `AT_ID_PATTERN_CQ`。
  **注意**：每次使用前显式 `pattern.lastIndex = 0`（共享状态约定）。
- `Set` 去重，命中和写入都是 O(1)。
- 删去多余的 `String(match[1])`（`exec` 返回的捕获组本就是 string）。

## 三、自测结果

`test_optimizations.js` 同时引入了**旧实现（参考）**和**新实现**做差分测试。

```
--- LCS 等价性 ---             12/12 通过
--- LCS 阈值早停一致性 ---     48/48 通过
--- isReplyTooSimilar 等价性 ---  9/9 通过
--- extractAtIds 等价性 ---      6/6 通过

汇总: 75 通过 / 0 失败
```

性能采样（200 次循环）：

| 场景 | 旧 | 新 | 改善 |
|------|----|----|------|
| `isReplyTooSimilar`（长串、字符多有重叠） | 105.2 ms | 3.6 ms | 约 29× |
| `isReplyTooSimilar`（长串、字符集几乎不重叠） | 57.4 ms | 1.2 ms | 约 48× |
| 裸 `LCS`（长串、阈值不可达） | 101.7 ms | 112.2 ms | 基本持平 |

> 说明：裸 `LCS` 在阈值不可达时不应快多少（最坏仍 O(n·m)），新版与旧版基本持平。
> 真正的收益在调用方 `isReplyTooSimilar`：先做 O(n+m) 的 Jaccard 上界，能在大多数无关回复上**完全跳过 LCS**。

## 四、文档优化要点（README.md）

老文档的问题：

1. NapCat 那段写「请按官方方式安装」太空，用户不知道实际要点哪个按钮。
2. `koishi.yml` 给的是 `ws-reverse`，但实际仓库里运行的 `koishi.yml` 用的是 `ws`（正向），口径不一致。
3. 「执行 `scripts/xxx.sh`」没说清是 git clone 后跑，还是粘贴脚本内容跑。
4. 视频插件、AI 插件的依赖文件路径散落在好几个地方，新人容易漏。

新文档做了什么：

- 第三章给一张「**6 步成功标准**」的总表，每一步都规定「怎么算成功」。
- 第四章 → 服务器 + Node.js（带版本检查命令）。
- 第五章 → NapCat：**写明用一键脚本、写明管理页面端口、写明 OneBot 反向 / 正向选哪个、给出 `ss -ltnp | grep 8080` 验证命令**。
- 第六章 → Koishi：`koishi.yml` 改回与仓库一致的 `protocol: ws`，并提供单行 `sed -i` 替换命令避免手改文件。
- 第七章 → 插件：明确「**直接复制 `scripts/*.sh` 内容粘贴执行**」，给一张总表（哪个必装、验证指令、是否要前置依赖）。
- 视频插件、AI 插件分别有独立小节，把「装 yt-dlp / ffmpeg / 上传 cookies / 写 key 文件 / 写模型名 / 写 base url」拆成可逐条粘贴的命令。
- 第八章给 pm2 的安装、启动、查日志、重启、开机自启完整命令。
