# 制定计划

当你面对复杂任务时，不要一上来就执行。先用 `create_plan` 创建计划，拆成明确步骤，再逐步执行。

## 何时需要制定计划

- 任务需要多步拆解
- 步骤之间有依赖关系
- 需要明确先后顺序、检查点
- 涉及多个工具、文件、系统
- 用户明确要求先给出计划
- **当前路径受阻，需要换方法**

## 遇阻不放弃规则（重要）

当工具返回空结果、失败、或不理想时：

1. **不要直接放弃**。分析失败原因。
2. **换工具**——一个工具不行换另一个。
   - 例：`query_logs` 返回空 → 改用 `execute_shell tail` 或 `read_file` 直接读日志
   - 例：搜索无结果 → 换关键词、换搜索引擎、读候选页正文
3. **换路径**——当前方法走不通，找替代方案。
4. **不确定时**——向用户说明尝试了什么、为什么失败、建议什么替代方案。
5. **空结果 ≠ 问题不存在**。空结果只说明当前查询没命中。

## 如何制定计划

1. 分析任务：确定最终目标
2. 拆步骤：每个步骤是可执行的独立动作
3. 定顺序：标注依赖关系和推荐顺序
4. 设检查点：每一步完成后如何验证
5. 使用 `create_plan` 创建持久化计划，然后逐步执行 `update_task_status`

## 诊断任务的推荐步骤

当用户问 "服务器状态"、"查日志"、"有什么问题"：

1. `execute_shell` 查进程 — `ps aux | grep koishi`
2. `execute_shell` 查端口 — `ss -tlnp | grep -E '5140|5150'`
3. `read_file` 或 `execute_shell tail` 读最近日志
4. `query_logs` 精确搜错误
5. `execute_shell` 查资源 — `free -h`
6. 综合以上给出判断。不要只凭一个工具的结果下结论。

## 不要这样用

以下情况不要用：
- 你其实可以把任务直接完成
- 真正缺的是一个小事实，不是计划
- 你连任务目标都没理解清楚

## 计划质量标准

一份合格的计划应满足：
- 有明确步骤，而不是泛泛建议
- 步骤顺序清楚
- 每步是可执行动作
- 有必要的验证点
- 遇阻时有替代方案
