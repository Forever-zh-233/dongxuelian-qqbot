🧩 莲莲Bot 借鉴 QwenPaw & Claude Code：Agent 能力渐进式开发计划

目标：在不破坏莲莲Bot现有 QQ 聊天人格、风控、记忆、Dashboard 和部署链路的前提下，逐步引入 Agent 工具调用、Token/上下文优化、工具安全审批、浏览器自动化和前端控制台能力。

核心原则：先做最小可用闭环，再扩工具；先 Dashboard 管理闭环，再放到 QQ；先只读工具，再危险工具；先可观测和可回滚，再追求功能完整。

0. 当前计划需要修正的问题

原计划方向正确，但太像"大爆炸式移植"。改为 6 个阶段，每个阶段都有验收、回滚点和不做事项。

  1. 核心聊天链路风险太高：chat.js 直接嵌 ReAct 循环容易影响普通聊天。
  2. 浏览器全量移植过早：应先做 6~8 个核心动作。
  3. 安全和前端审批必须先于危险工具。
  4. Claude Code 的缓存优化：更可靠的做法是静态 prompt 稳定化 + 动态 prompt 后置 + 工具 schema 分层。
  5. Skill 市场不是第一阶段能力。

2. 总体架构

核心原则：不要把 ReAct 直接塞进普通聊天主路径。新增独立 Agent Engine。
QQ 普通聊天 → chat.js 原有链路（默认不变）
QQ Agent 模式 → agent-engine.js → api.js / tool-registry.js / tool-safety.js
Dashboard Agent → standalone.js → agent-engine.js（同一套）

6. ReAct Engine

buildAgentMessages() → requestChatCompletions(messages, { tools })
→ 如果没有 tool_calls → 返回最终文本
→ 执行每个 tool_call（串行）
→ tool result 追加到 scratchpad
→ 最多循环 3~5 次

边界：最多 5 轮、每轮最多 3 个工具、每个工具有 timeout。tool result 不进入普通聊天历史。

10. 分阶段实施路线

📦 Phase 0：准备和保护网

保证改动可测、可回滚。梳理 api.js、chat.js，新增测试夹具。

🛠️ Phase 1：只读 Agent MVP

新增 agent/engine.js，api.js 支持 tool_calls，QQ "莲莲工具 现在几点"。

🔐 Phase 2：安全与配置闭环

safety.js + pending.js + Dashboard 工具管理 API。

📁 Phase 3：文件和搜索工具

web_search、read_file、write_file（仅 Dashboard confirm）。

🖥️ Phase 4：Dashboard Agent 控制台

AgentPanel.vue：工具开关、审批队列、调用统计。

🌐 Phase 5：浏览器核心动作

browser.js 核心 8 action + idle watchdog。

📦 Phase 6：上下文压缩与 Skill 市场

context.js compact + skill-hub CLI。

13. 最小可交付版本

  1. QQ 显式命令能调用 get_current_time / calculator
  2. Dashboard 能看到工具列表、开关和调用日志
  3. api.js 正确支持 tool_calls 和 fallback
  4. 普通聊天完全不受影响
  5. 所有工具调用都有审计日志和结果截断

14. 与原计划的主要差异

原计划修订后
直接在 chat.js 嵌 ReAct新增独立 agent/engine.js
10 天做完全部6 个阶段，可验收可回滚
浏览器 34 action先 8 个核心 action
工具首轮发送、第二轮不发每轮保持 schema 稳定
Skill 市场较早进入后置到最后
危险工具列入早期默认关闭，安全系统完成后上线
