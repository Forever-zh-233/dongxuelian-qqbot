# 东雪莲 Bot 教训总结

## 一、代码设计原则

### 1. 避免 index.js 过度膨胀

- **非必要情况不要优先考虑在 index 加代码**。新增功能时先判断是否适合独立模块（forward.js、vision.js、sensitive.js 等），而不是一股脑塞进 index.js 的中间件。
- **设计代码时优先避免膨胀，以好维护好阅读为首要目的。** 一个函数超过 50 行且职责不单一，就应该考虑拆分。
- **行为变更优先加 scenario 测试，不要加源码字符串扫描。** 源码扫描只防结构契约破坏。

### 2. 模块依赖规则

- 新模块只从 `constants.js`、`utils.js`、`api.js`、`conversation.js`、`persona.js` 等稳定模块导入。
- **不反向 import `index.js`**，避免循环依赖。
- 聊天核心链路（`chat()`) 保留模型调用所有权，子模块通过回调与主流程交互，不直接调用 `callOpenAI()`。

### 3. 按边界拆，不按行数拆

文件膨胀不是 bug，以下才是真危险信号：
- 改一处要同时理解 5 个状态 Map 和 3 条异步路径
- 新功能和旧功能共享隐式变量，顺序一变就炸
- 测试只能靠源码字符串扫描，不知道真实链路有没有变
- 一个分支里同时塞了"新功能 + 命令 + 状态 + 定时器 + IO + 模型调用"

如果只是小改（加一条判断、补一个日志、调整一个已有分支的条件、改一点 prompt 组合），直接改原文件完全可以。别为了模块洁癖硬拆。硬拆有时候比膨胀更烦——像把抽屉整理成 18 个小盒子，最后找钥匙还得开半天。

**判断标准**：

1. 只是补现有功能的一小段逻辑 → 直接放原文件
2. 少于 50 行，但有独立 Map/cache/timer/env/file → 倾向新模块
3. 少于 50 行，但未来明显会扩张 → 可以先新模块，别等它变成 200 行
4. 少于 50 行、无状态、单点调用 → 放原文件当 helper
5. 超过 80 行或需要单独 scenario/纯函数测试 → 新模块

**一句话决策**：这段代码是在补旧功能，还是在长出一个新功能器官？补旧功能直接改，长新器官新模块。


## 二、代码拆分教训

### 拆文件 5 步法（必须遵守）

从 index.js 拆出新文件（如 repeat.js、forward.js）时必须遵守以下步骤，不可跳过：

```
第 1 步：只创建目标文件，完整复制函数 + 状态，跑 node -c 确认语法
第 2 步：源文件加 import，跑 node -c 确认语法
第 3 步：注释掉源文件里的旧定义（不删），跑 npm run test:quick 验证 import 生效
第 4 步：测试通过后删掉注释掉的旧代码，再跑一次 npm run test:quick
第 5 步：更新 cascade-test.js 的引用路径（modPaths/exports/syntaxFiles/duplicateScanFiles 等），再跑测试
```

每步之间 `node -c` + `npm run test:quick` 验证，不过不进入下一步。后三步中任何一步失败，退回上一步重来。

### 常见失误案例

| 操作 | 后果 |
|---|---|
| 删旧代码 + 加新 import 同时做 | 漏了大括号，语法检查通过前已污染文件 |
| cascade-test.js 改了源码字符串匹配但没同步更新引用 | 测试报 `xxx is not a function` |
| 函数搬到 utils.js 后没及时删 index.js 里的旧定义 | 函数去重检查报 FAIL |
| 拆模块时一次性改了 4 个文件然后跑测试 | 出问题时不知道是哪个操作引入的 |

### 要点

- 拆模块时如果原代码大量使用 `var`，第一版保持 `var` 不变，先保行为。等测试稳定后再局部改成 `let/const`，避免 TDZ 问题。
- 新增生产模块后同步维护 `package.json scripts.check` 和 cascade-test.js 的 `modPaths`、`expectedExports`、`syntaxFiles`、`duplicateScanFiles`。
- 不要在 `saveConversationTurn` 里耦合与对话保存无关的逻辑（如当初的 `saveSensitiveCache` 需求→应挂在中间件独立执行）。


## 三、服务器部署教训

### 1. pkill -f 误杀 SSH

```bash
# 错误 ❌ — 会杀 SSH 守护进程导致服务器失联
pkill -f "start"

# 正确 ✅
pkill -f 'koishi/lib/worker'
```

`pkill -f` 用 ERE 匹配进程名，"start" 会匹配所有包含 start 的进程包括 sshd。

### 2. 进程残留导致双实例

重启时若 pkill 没杀干净，旧进程孤儿化后继续监听端口，新进程启动失败。必须 `pkill -f 'koishi/lib/worker'` 后 `sleep 3` 确保端口释放。

### 3. SCP 编码损坏

PowerShell `Get-Content file | ssh ... Set-Content` 会输出 UTF-16 LE 带 BOM，损坏中文字符和 API key。必须直接用 SCP 二进制传输：

```bash
scp file user@host:/path/    # ✅ 正确
```

### 4. API Key 格式

| 供应商 | Key 前缀 | 文件 |
|---|---|---|
| DeepSeek | `sk-*` | `ai-deepseek-key.txt` |
| mimorium | `tp-*` | `ai-mimorium-key.txt` |
| opencode | `sk-*` | `ai-openai-key.txt` |
| dashscope | `sk-*` | `ai-dashscope-key.txt` |
| GLM | 无固定前缀 | `ai-glm-key.txt` |

服务端和本地的 key 相互不通用。

### 5. 部署确认

部署后等 15 秒，确认日志出现 `dongxuelian-ai.*loaded` 且只有一条，不要盲目重试。第一次重试前先确认上次是否已经成功。

### 6. koishi.yml 格式

`server` 插件必须以根级 `port`/`selfUrl` 配置。`adapter-onebot` 必须用 `ws` 协议正向连接 NapCat（NapCat 是 WS 服务端，Koishi 是客户端）。

## 时间依赖测试教训

fake timers 最初是为了稳定复读冷却、sticker 冷却、敏感检测缓存过期等测试，避免真实等待 30-120 秒。它有用，但不能变成默认依赖：全局 patch `Date.now` / `setTimeout` / `setInterval` 会扩大污染面，后续排查测试时容易看不清真实时间和 fake 时间的边界。

新的默认做法：生产代码优先支持局部 `now` 注入或 `expiresAt` 惰性过期；scenario 默认不用 fake timers。只有确实要测 timer 调度本身时，才显式启用 fake timers，并写清原因。

## 六、`@satorijs/core` 版本锁死（新接手第一坑）

### 为什么必须锁 3.7.0

```
koishi@4.18.11
  └─ @koishijs/core@4.18.11
       └─ @satorijs/core@^4.6.0    ← koishi 4.x 需要 4.x
adapter-onebot@6.9.3
       └─ @satorijs/core@^3.0.0    ← adapter 仍依赖 3.x
```

`^3.0.0` 和 `^4.6.0` 没有交集，npm 无法同时满足。
- `overrides` 强制全项目用 `@satorijs/core@3.7.0`
- 但 `3.7.0` 的 `Session.prototype` 缺少 `resolve` / `stripped` / `send` / `text` 四个方法
- 所以 `dongxuelian-ai/index.js` 在加载时自动补上这四个方法（无需额外文件）

### 为什么不能升级

- 升级 `@satorijs/core` 到 4.x → `adapter-onebot` 加载后 `http.ws()` 静默失败，不报错、不记重试日志，WebSocket 连不上 NapCat
- 升级 `adapter-onebot` → 目前 `6.9.3` 已是最新版，上游未发适配 4.x 的版本
- 升级 koishi 到 5.x → 破坏性 API 变更，所有插件需要重写
- **三者版本是锁死的铁三角**：koishi 4.x + @satorijs/core 3.7.0 + adapter-onebot 6.x，动任何一个都需要验证另外两个的兼容性

### 接手者必读

```bash
# 每次 npm install 后验证根 node_modules 有 @satorijs/core
node -e "console.log(require('@satorijs/core/package.json').version)"
# 必须输出 3.7.0

# 验证 adapter 连接
grep "adapter connect to server" koishi.log
```

补丁位置：缺少的 Session 方法（stripped / resolve / send / text）在 `dongxuelian-ai/index.js` 加载时自动补全，无需单独的 patch.js 文件，也不要 `node -r` 预加载。补丁是插件自身初始化代码的一部分。

## 七、本次修复记录（2026-05-04）

### 现象

移除 `overrides` 后 bot 启动无响应，群聊私聊消息不触发（0 日志）。

### 排查链路

1. 日志显示所有插件正常加载，`adapter-onebot` 的 `loader apply plugin` 行存在
2. 但缺少关键行 `adapter connect to server: ws://127.0.0.1:8081/onebot/v11/ws`
3. 直接 `new WebSocket('ws://127.0.0.1:8081/onebot/v11/ws')` 连接成功 → NapCat 正常
4. `@satorijs/core` 从 `3.7.0` 变更为 `4.6.0` → `adapter-onebot` 的 `http.ws()` 静默失败
5. `WsClientBase.start()` 的 `prepare()` 抛错被 catch，`reconnect()` 调 `logger.warn(error.toString())`，但 `event.message` 为空时不打印 → 日志无痕迹
6. 恢复 `overrides` 并将 `@satorijs/core` 加入 `dependencies` 后 adapter 立即连接成功

### 根因

`@satorijs/core` 3→4 是 breaking change。`adapter-onebot@6.9.3` 通过 `koishi` 间接使用 `@satorijs/core`，其 `http.ws()` 内部 API 在 4.x 发生了变化，连接异常被 `WsClientBase` 的 error 回调吞掉（空 `event.message` 时不记日志）。

### 预防

- 永远不要单独动 `@satorijs/core` 版本，必须三个组件（koishi / @satorijs/core / adapter-onebot）一起验证
- 改完重启后，日志搜索 `adapter connect to server` 确认连接，不要只看插件加载
- 任何依赖变更后跑完全部测试（`npm test`）+ 人工发一条消息验证
- 如果要去掉 `overrides`，必须先确认 `adapter-onebot` 发布了支持 `@satorijs/core@4.x` 的新版本

## 九、2026-05-04 服务器部署教训

### 1. GitHub 不可达时的 SCP 部署

GitHub 连不上时只能用 SCP。大重构后首次部署**不要只挑改动的文件传**——漏一个 `rulesets/jailbreak.js` 整站崩溃，且报错看起来像不相关的问题（`Cannot destructure 'hasAt'`）。

**做法：** 首次传整个 `lib/` 目录（`scp -r lib/*.js` + `scp -r lib/rulesets`），后续增量更新再挑文件。

### 2. patch.js 不再需要（2026-05-05 已删除）

补丁已合并到 `dongxuelian-ai/lib/index.js`，插件加载时自动补齐 `stripped` / `resolve` / `send` / `text`。不再需要独立的 `patch.js` 文件，也不要 `node -r` 预加载。

**服务器正确启动命令：**
```bash
DONGXUELIAN_AI_DATA_DIR=/root/koishi-app/data node /root/koishi-app/node_modules/.bin/koishi start
```

### 3. 重启前必须杀干净再等端口释放

多次尝试不同启动方式会产生孤儿进程抢占端口。**正确做法：**
```bash
pkill -9 -f 'koishi'    # 杀全部
sleep 4                  # 等端口释放
# 确认端口无监听后再启动
```

### 4. 环境变量耦合

`constants.js` 的 `DATA_DIR` 改成相对路径后，服务器的 key 文件路径变了（`/root/koishi-app/data` → `packages/.../data`），导致 key 读取为空。服务器启动时必须显式设置 `DONGXUELIAN_AI_DATA_DIR=/root/koishi-app/data`。

### 5. 覆盖前询问

部署时先确认服务器当前状态再覆盖配置或代码文件。

## 十一、2026-05-05 新插件开发教训

### 1. 禁止重复造轮子

新插件的 API 调用必须复用主插件的 `runtime-config.js` + `api.js`，禁止自己写 API 客户端。主插件的 fallback 链、超时控制、reasoning 处理是几十个小时打磨出来的，自己写的 30 行代码没有这些护栏。

### 2. 中间件 return 必须阻止后续

Koishi 中间件中匹配到命令后，必须 `await session.send()` + `return`，不能 `return session.send()`（有返回值会终止但语义不清），更不能漏掉 `return`（会导致后续插件继续处理，出现神秘数字）。

### 3. 部署前检查端口和NapCat状态

重启 bot 前先确认 NapCat 进程和端口监听状态。 koishi.yml 的 endpoint 端口必须和 NapCat 的 websocketServers 端口一致。部署后日志必须看到 `adapter connect to server` 才算连接成功。

### 4. Puppeteer 并发和内存管理

日报插件使用 Puppeteer 截图，每次调用会启动一个 Chromium 进程（~200MB内存）。必须加信号量限制并发（最多2个），并设置超时（30秒）。否则多个群同时请求会导致内存溢出，bot 崩溃。另外 `base64:` 协议已弃用，应改用 `data:image/png;base64,`。

### 5. 模板系统必须用占位符

动态模板必须用 `{{variable}}` 占位符，不能用 `${variable}`（会和 JS 模板字符串冲突）。静态 HTML 文件不能直接作为模板使用，必须改造为动态模板。

### 6. scp -r 不会删除远程已删除的文件

`scp -r` 只做增量复制，本地删除的文件不会从远程删除。导致服务器上残留旧文件（如paper.html、miku.html），被随机选中后崩溃。**应改用 `rsync -av --delete` 做同步。**

### 7. 频繁重启的根因排查

bot 反复崩溃时，优先检查：1) 端口是否被占用 2) NapCat 是否运行 3) Puppeteer 僵尸进程。用 `ps aux | grep koishi` 确认进程是否存在，用 `tail -20 koishi.log` 看最后的错误。

### 8. 重启流程必须用脚本

手动 SSH 拼命令重启容易漏步骤（忘了 pkill、忘了等端口释放、忘了设环境变量），导致：
- `No open ports available` — 旧进程残留占端口
- 双实例 — 先启动的没杀掉又启了一个
- bot 不响应 — 实际上没启动成功

**正确做法（2026-05-05 起）**：

```bash
# 服务器端（已在 /root/koishi-app/restart.sh）：
bash /root/koishi-app/restart.sh

# 本地部署+重启（scripts/deploy-and-restart.bat）：
scripts\deploy-and-restart.bat
```

`restart.sh` 做 5 件事：杀干净所有 koishi 进程 → 等 4 秒确认端口释放 → 设置环境变量启动 → 等 15 秒 → grep 日志确认 `daily-report loaded` + `adapter connect`。

### 9. Dashboard 硬编码教训

Dashboard 插件（`koishi-plugin-dashboard`）的端口、路径、Shell 命令不能硬编码，否则换服务器就崩。

**已修复的硬编码（2026-05-06）：**

| 硬编码 | 改为 |
|--------|------|
| `const PORT = 5150` | `process.env.DASHBOARD_PORT \|\| 5150` |
| `bash /root/koishi-app/restart.sh` | `process.env.KOISHI_DIR \|\| \_\_dirname` + path.join |
| `=== 'on'` 维护检测 | `!!readFileSync(...)` 非空检测 |

**新增但可能需配置的（有环境变量覆盖）：**

- `DASHBOARD_PASSWORD` — Dashboard 登录密码（默认 `Aa****`）
- `DASHBOARD_PORT` — Dashboard HTTP 端口，默认 5150
- `KOISHI_DIR` — Koishi 根目录，默认自动推算
- `NAPCAT_HOST` / `NAPCAT_PORT` / `NAPCAT_TOKEN` — NapCat WebUI 代理配置

**未来守则：**
- 端口、路径、Shell 命令必须有环境变量覆盖，不允许裸硬编码
- 跨插件路径用 `path.join(__dirname, '..', '插件名')`，避免绝对路径
- Shell 命令统一放到 `restart.sh` 等脚本文件，不在 JS 里拼命令
- Dashboard 的 NapCat 代理通过服务器本地 `127.0.0.1` 访问，避免硬编码公网 IP

### 10. NapCat WebUI Token 验证不可靠

NapCat WebUI 的 API token 是运行时动态生成的，配置文件中的 token 不一定匹配。直接代理 WebUI 的 iframe 方案不可行，改为通过 SSH 隧道 + 手动复制 token 的方式访问 NapCat 管理面板。

### 11. Dashboard 敏感操作二次验证

Dashboard 的敏感操作（启动/停止 Bot、改配置、改白名单）应通过服务器密码二次验证，不可仅凭登录密码放行。验证弹窗需在点击操作按钮后才弹出，不能提前阻塞界面。

**失败教训：** 首次实现时 `AdminDialog` 的 `visible` 默认设为 `true`，导致弹窗默认挡住整个界面，且 `z-index` 低于鼠标拖尾 Canvas（9999），按钮点不了。正确做法：`visible` 默认 `false`，在 API 返回 `403 ADMIN_REQUIRED` 时才弹窗；`z-index` 要高于所有覆盖层（设为 10000）。

- 敏感 API 请求头带 `X-Admin-Token` 字段
- 密码存储在 `data/` 目录下的独立文件（`dashboard-admin-pwd.txt`、`dashboard-access-pwd.txt`），不在代码中硬编码
- 环境变量 `DASHBOARD_PASSWORD` 和 `DASHBOARD_ADMIN_PASSWORD` 可覆盖默认值

**改密码流程：**
- 改访问密码 → 需要服务器密码验证
- 改服务器密码 → 需要旧服务器密码验证

## 十三、2026-05-09 session.stripped 修复教训

### 1. `require('koishi').Session` 是 undefined

`require('koishi')` 不导出 Session 类，必须从 `@satorijs/core` 直接获取：
```javascript
// 错误 ❌
const { Session } = require('koishi')   // → undefined

// 正确 ✅
const satoriCore = require('@satorijs/core')
const Session = satoriCore.Session
```

### 2. 全局 /usr/bin/koishi 与本地 binary 不通用

全局 `/usr/bin/koishi` 有自己的 `node_modules/@satorijs/core@4.6.0`，与 app 的 `@satorijs/core@3.7.0` 不兼容。用全局 binary 启动时：
- `adapter-onebot` 的 `http.ws()` 静默失败（不报错、不记日志）
- `session.stripped` 等补丁打到 3.7.0 的 Session 上但运行时 session 是 4.6.0 的

**必须用本地 binary：** `node node_modules/koishi/bin.js start`

### 3. PowerShell 管道损坏二进制文件

```powershell
# 错误 ❌ — 输出 UTF-16，损坏中文/编码
git show HEAD:file.js > file.js
git show HEAD:file.js | Set-Content file.js -Encoding UTF8

# 正确 ✅ — 字节级复制
cmd /c "git show HEAD:file.js > file.js"
# 或直接 git checkout
git checkout HEAD -- file.js
```

### 4. 补丁不要放独立文件，放插件模块加载代码

独立 `patch.js` + `NODE_OPTIONS=--require=patch.js` 有两个问题：
- `NODE_OPTIONS` 不一定能穿透 `fork()` 到 worker
- 环境变量没设对就回到原点

**正确做法：补丁放在 `dongxuelian-ai/index.js` 顶部、`exports.name` 之前**，随插件 require 自动执行，worker 加载插件时自然生效。

### 5. 修复前先确认原始代码是否有问题

这次花了大量时间绕路，根本原因是**没有先验证 `main` 分支代码在干净环境下是否正常**，而是直接假设代码有 bug 开始打补丁。正确的排查顺序应该是：
1. 确认 `node_modules` 状态正常（`@satorijs/core` 版本）
2. 用正确的启动命令（本地 binary + DATA_DIR）
3. 确认 adapter 连接日志
4. 只有以上都确认后才考虑代码问题

## 十四、2026-05-10 Dashboard 环境变量 + 按钮反馈教训

### 1. Dashboard standalone.js 启动必须带 DONGXUELIAN_AI_DATA_DIR

Dashboard 的 `standalone.js` 使用 `process.env.DONGXUELIAN_AI_DATA_DIR || path.join(__dirname, ...)` 解析数据目录。如果直接 `node standalone.js` 启动（没有环境变量），`DATA_DIR` 会回退到 `packages/koishi-plugin-dongxuelian-ai/data`，而 bot 读取的是 `/root/koishi-app/data/`（由 `restart.sh` 设置环境变量）。

后果：前端显示"更新成功"，但文件实际写到了 packages 路径。bot 读不到修改。

**正确做法：** 每次手动启动 dashboard 时必须带环境变量：
```bash
DONGXUELIAN_AI_DATA_DIR=/root/koishi-app/data nohup node standalone.js &
```
或直接用 `restart.sh`（它会自动设置环境变量并同时重启 bot + dashboard）。

### 2. 按钮点击后必须给出即时反馈

"点编辑没反应" 的原因是点击编辑按钮后：
- 没有立即改变按钮文字（显示"加载中..."）
- 没有滚动到编辑表单区域

**修复原则：** 所有触发异步操作的按钮，点击后必须：
1. 立即改变按钮文字（如"编辑"→"加载中..."，"保存"→"保存中..."）
2. 禁用按钮防止重复点击（`:disabled`）
3. 操作完成后自动滚动到结果/编辑区域（`scrollIntoView({ behavior: 'smooth' })`）
4. 取消操作时也要清除对应的 loading 状态

涉及组件：`PersonaPanel.vue`（人格编辑）、`DeployPanel.vue`（部署）、`ConfigPanel.vue`（配置保存）等。已检查确认 ConfigPanel 各保存按钮均有 loading 状态。PersonaPanel 已补修编辑按钮反馈+滚动。
