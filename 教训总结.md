# 东雪莲 Bot 教训总结

## 一、代码设计原则

### 1. 避免 index.js 过度膨胀

- **非必要情况不要优先考虑在 index 加代码**。新增功能时先判断是否适合独立模块（forward.js、vision.js、sensitive.js 等），而不是一股脑塞进 index.js 的中间件。
- **设计代码时优先避免膨胀，以好维护好阅读为首要目的。** 一个函数超过 50 行且职责不单一，就应该考虑拆分。
- **行为变更优先加 scenario 测试，不要加源码字符串扫描。** 源码扫描只防结构契约破坏。

### 2. 模块依赖规则

- 新模块只从 `constants.js`、`utils.js`、`api.js`、`conversation.js`、`persona.js` 等稳定模块导入。
- **不反向 import `index.js`**，避免循环依赖。
- 聊天核心链路（`chat()`) 保留模型调用所有权，子模块通过回调与主流程交互，不直接调用 `callOpenAI()`。

### 3. 按边界拆，不按行数拆

文件膨胀不是 bug，以下才是真危险信号：
- 改一处要同时理解 5 个状态 Map 和 3 条异步路径
- 新功能和旧功能共享隐式变量，顺序一变就炸
- 测试只能靠源码字符串扫描，不知道真实链路有没有变
- 一个分支里同时塞了"新功能 + 命令 + 状态 + 定时器 + IO + 模型调用"

如果只是小改（加一条判断、补一个日志、调整一个已有分支的条件、改一点 prompt 组合），直接改原文件完全可以。别为了模块洁癖硬拆。硬拆有时候比膨胀更烦——像把抽屉整理成 18 个小盒子，最后找钥匙还得开半天。

**判断标准**：

1. 只是补现有功能的一小段逻辑 → 直接放原文件
2. 少于 50 行，但有独立 Map/cache/timer/env/file → 倾向新模块
3. 少于 50 行，但未来明显会扩张 → 可以先新模块，别等它变成 200 行
4. 少于 50 行、无状态、单点调用 → 放原文件当 helper
5. 超过 80 行或需要单独 scenario/纯函数测试 → 新模块

**一句话决策**：这段代码是在补旧功能，还是在长出一个新功能器官？补旧功能直接改，长新器官新模块。


## 二、代码拆分教训

### 拆文件 5 步法（必须遵守）

从 index.js 拆出新文件（如 repeat.js、forward.js）时必须遵守以下步骤，不可跳过：

```
第 1 步：只创建目标文件，完整复制函数 + 状态，跑 node -c 确认语法
第 2 步：源文件加 import，跑 node -c 确认语法
第 3 步：注释掉源文件里的旧定义（不删），跑 npm run test:quick 验证 import 生效
第 4 步：测试通过后删掉注释掉的旧代码，再跑一次 npm run test:quick
第 5 步：更新 cascade-test.js 的引用路径（modPaths/exports/syntaxFiles/duplicateScanFiles 等），再跑测试
```

每步之间 `node -c` + `npm run test:quick` 验证，不过不进入下一步。后三步中任何一步失败，退回上一步重来。

### 常见失误案例

| 操作 | 后果 |
|---|---|
| 删旧代码 + 加新 import 同时做 | 漏了大括号，语法检查通过前已污染文件 |
| cascade-test.js 改了源码字符串匹配但没同步更新引用 | 测试报 `xxx is not a function` |
| 函数搬到 utils.js 后没及时删 index.js 里的旧定义 | 函数去重检查报 FAIL |
| 拆模块时一次性改了 4 个文件然后跑测试 | 出问题时不知道是哪个操作引入的 |

### 要点

- 拆模块时如果原代码大量使用 `var`，第一版保持 `var` 不变，先保行为。等测试稳定后再局部改成 `let/const`，避免 TDZ 问题。
- 新增生产模块后同步维护 `package.json scripts.check` 和 cascade-test.js 的 `modPaths`、`expectedExports`、`syntaxFiles`、`duplicateScanFiles`。
- 不要在 `saveConversationTurn` 里耦合与对话保存无关的逻辑（如当初的 `saveSensitiveCache` 需求→应挂在中间件独立执行）。


## 三、服务器部署教训

### 1. pkill -f 误杀 SSH

```bash
# 错误 ❌ — 会杀 SSH 守护进程导致服务器失联
pkill -f "start"

# 正确 ✅
pkill -f 'koishi/lib/worker'
```

`pkill -f` 用 ERE 匹配进程名，"start" 会匹配所有包含 start 的进程包括 sshd。

### 2. 进程残留导致双实例

重启时若 pkill 没杀干净，旧进程孤儿化后继续监听端口，新进程启动失败。必须 `pkill -f 'koishi/lib/worker'` 后 `sleep 3` 确保端口释放。

### 3. SCP 编码损坏

PowerShell `Get-Content file | ssh ... Set-Content` 会输出 UTF-16 LE 带 BOM，损坏中文字符和 API key。必须直接用 SCP 二进制传输：

```bash
scp file user@host:/path/    # ✅ 正确
```

### 4. API Key 格式

| 供应商 | Key 前缀 | 文件 |
|---|---|---|
| DeepSeek | `sk-*` | `ai-deepseek-key.txt` |
| mimorium | `tp-*` | `ai-mimorium-key.txt` |
| opencode | `sk-*` | `ai-openai-key.txt` |
| dashscope | `sk-*` | `ai-dashscope-key.txt` |
| GLM | 无固定前缀 | `ai-glm-key.txt` |

服务端和本地的 key 相互不通用。

### 5. 部署确认

部署后等 15 秒，确认日志出现 `dongxuelian-ai.*loaded` 且只有一条，不要盲目重试。第一次重试前先确认上次是否已经成功。

### 6. koishi.yml 格式

`server` 插件必须以根级 `port`/`selfUrl` 配置。`adapter-onebot` 必须用 `ws` 协议正向连接 NapCat（NapCat 是 WS 服务端，Koishi 是客户端）。


## 四、测试文件维护要点

### 测试入口职责

| 入口 | 命令 | 职责 |
|---|---|---|
| 语法门卫 | `npm run check` | 所有生产 JS 文件语法检查 |
| 结构契约 | `npm run test:quick` | 导出、常量、脚本、静态防线、覆盖映射 |
| 行为链路 | `npm run test:scenario` | fake Koishi 集成测试 |
| 辅助插件 | `npm run test:plugins` | 辅助插件独立测试 |
| 信任入口 | `npm test` | test:quick + test:scenario |

### 核心规则

- 行为变更优先加 scenario，不要加源码字符串扫描。
- cascade 的 COVERAGE_MAP 要当索引维护。文件名或 needle 变了要同步。
- 每拆一个生产模块，同步维护 package.json check 脚本 + cascade 的模块/导出/语法/重复扫描。
